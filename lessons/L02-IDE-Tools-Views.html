Lesson 2: The IDE — Tools, Views, and Navigation

Source: AVEVA Application Server 2023 Training Manual, Module 1, Section 4 (pages 1-21 to 1-30)

A complete tour of the Integrated Development Environment — the workspace where you design, configure, and deploy your Galaxy.

2.1 IDE Overview

Reference: Module 1, Section 4 (page 1-21)

Now that you have created a Galaxy (Lesson 1), it is time to explore the tool you will spend most of your time in: the Integrated Development Environment (IDE).

The IDE is the design environment for Application Server. It is a client application that connects to the Galaxy Repository over a network. The IDE does NOT run your application — it is a development tool. All actual operations (deploy, start, stop, import) are sent as commands to the Galaxy Repository, which executes them remotely.

Client-Server Architecture

Understanding the IDE's role is critical. The relationship works like this:

ComponentRoleWhat It Does
IDE (your workstation) Client Presents the user interface, lets you design objects, configure settings, and issue commands
Galaxy Repository (GR) Server Stores the Galaxy database, executes all commands, manages deployment to platforms

This means:

  • Multiple engineers can run IDEs connecting to the same Galaxy Repository simultaneously
  • The IDE can be closed without affecting the running Galaxy
  • You can work on multiple Galaxies by connecting to different Galaxy Repositories
  • If the GR goes down, no IDE can function until it recovers
The IDE interface — Galaxy hierarchy on the left, work area in the center, and ribbon toolbar at the top for commands and navigation (Module 1, page 1-21)
Think of it this way: The IDE is like a remote control for your Galaxy. You point it at the GR, and it lets you control everything — but the GR does the actual work.

2.2 Key Functions of the IDE

Reference: Module 1, Section 4 (page 1-22)

The IDE provides three categories of functionality: Galaxy-wide settings, object management, and import/export.

Galaxy-Wide Settings

These settings apply to the entire Galaxy and affect all objects within it. You configure them once and they apply everywhere:

SettingPurposeKey Detail
Security Controls who can access and modify the Galaxy Define users, roles, and permissions; integrates with Windows Active Directory
Language Sets the display language for the IDE Affects menus, dialogs, and object display names
I/O Configuration Global I/O communication settings Default timeouts, communication parameters, scan intervals
Graphics Default graphics settings for the Galaxy Screen profiles, default graphic sizes, display preferences
Time Master Defines the master clock source All platform clocks synchronize to this reference
Alarms Global alarm configuration Alarm classes, priorities, logging, and suppression settings
Preferences IDE user preferences Layout, window positions, display options per user
Multi-Galaxy Cross-Galaxy communication Configure how multiple Galaxies share data at runtime

Object Management

The IDE lets you create, modify, delete, and organize automation objects. You can:

  • Create new objects from templates
  • Modify object properties (attributes, scripts, graphics bindings)
  • Delete objects (with dependency checking)
  • Check-out/check-in objects for multi-user development (like source control for automation)
  • Copy/paste objects between areas or equipment groups

Import and Export

You can move objects between Galaxies or back up your work:

  • Export — Extract selected objects to a file; useful for backup or migration
  • Import — Bring objects from another Galaxy or a file into the current one
  • Galaxy Backup — Complete backup of the entire Galaxy for disaster recovery
  • Galaxy Restore — Recreate a Galaxy from a backup
Warning: Import/export operations require careful attention to version compatibility. Objects exported from one version of Application Server may not be fully compatible with a different version. Always check release notes before importing.

2.3 Tools in the IDE

Reference: Module 1, Section 4 (page 1-23)

The IDE organizes its workspace into two tools (tabs) and multiple application views. This structure keeps different aspects of your Galaxy organized.

Two tools, two purposes:
Templates tool — where you browse and use template objects (the building blocks of your application)
Graphics tool — where you browse and use graphic symbols (the visual elements for your HMI)

Below the tools, the IDE provides several application views, each showing a different perspective on your Galaxy:

ViewPurpose
Model ViewShows the plant model hierarchy — areas, equipment, and object containment
Deployment ViewShows platforms, engines, and which objects are deployed where
Derivation ViewShows parent-child relationships between templates and instances
IO Devices ViewShows device integration objects, scan groups, and I/O bindings
Mapping ViewShows I/O reference validation and autobinding information
Operations ViewShows deployment operations and their status
The IDE's two tools (Templates and Graphics) and the application views — each provides a different perspective on the Galaxy (Module 1, page 1-23)
Navigation tip: The active tool determines what appears in the top-left pane. The application views appear in the bottom-left pane. The central area shows whatever you select.

2.4 Templates Tab

Reference: Module 1, Section 4 (page 1-24)

The Templates tab is where you browse, create, and manage template objects — the reusable building blocks that define your automation application.

Template Toolsets

The Templates tab organizes templates into a hierarchy of toolsets (folders). The default toolsets include:

ToolsetContentsTypical Use
Application Application-level templates (e.g., ViewApp, AppEngine) Organizing application structure, hosting graphics
Device Integration I/O device templates, communication drivers Connecting to PLCs, RTUs, and other field devices
System System-level templates (platforms, engines) Defining computers and runtime infrastructure

Base Templates vs. Protected Templates

Base templates are the root templates for each object type. They cannot be modified — only derived templates can be changed. This protects the foundational definitions that all other objects inherit from.

Protected templates are marked with an orange lock icon. They are read-only. You cannot modify, delete, or rename protected templates. This ensures the integrity of the template hierarchy.

Containment Display

Each template shows what it contains — which other object types can be nested inside it. This containment hierarchy is how you organize your Galaxy. For example:

  • A $Area object can contain other $Area objects (sub-areas) and equipment objects
  • A $Equipment object can contain I/O and logic objects
  • A $Platform object can host engines and objects

Custom Toolsets

You can create your own toolsets to organize custom templates. This is useful for grouping equipment types, separating phases of a project, or organizing by function (e.g., "Pumps", "Valves", "Heat Exchangers").

The Templates tab — Application, Device Integration, and System toolsets organize template objects. Protected templates show an orange lock icon (Module 1, page 1-24)
Important: You should never modify base templates directly. Instead, always derive from them to create custom versions. Modifying a base template can break every object that inherits from it.

2.5 Graphics Tab

Reference: Module 1, Section 4 (page 1-25)

The Graphics tab is where you browse and use graphic symbols — the visual elements you drag onto your HMI screens.

Graphic Toolsets

Like templates, graphics are organized into toolsets (folders). The two main graphic libraries are:

LibraryStyleBest For
Industrial Graphics Traditional, detailed symbols Classic HMI applications with detailed equipment graphics
Situational Awareness (SA) Modern, flat design with dynamic coloring Alarm-focused displays, high-performance HMI

Protected Symbols

Like template objects, some graphic symbols are protected (read-only). These are the system-provided symbols that ship with Application Server. You can derive from them to create custom versions, but you should never modify the originals.

Using Graphics

To use a graphic symbol:

  1. Select the Graphics tab
  2. Browse the toolset hierarchy to find the symbol you need
  3. Drag the symbol into a graphic display (in InTouch or OMI)
  4. Bind the symbol's attributes to object properties
The Graphics tab — symbol libraries include Industrial Graphics (traditional) and Situational Awareness (modern, dynamic coloring) (Module 1, page 1-25)
Choosing between libraries: Use Industrial Graphics for traditional, detailed equipment symbols. Use Situational Awareness for modern, alarm-focused displays where color changes convey status. Many projects use both.

2.6 Model View

Reference: Module 1, Section 4 (page 1-26)

The Model View shows your Plant Model — the logical hierarchy of areas, equipment, and objects that represents your physical plant.

The Plant Model and $Area Hierarchy

The Plant Model is organized as a tree of $Area objects. Areas represent physical locations or functional groups in your plant. You can nest areas up to 10 levels deep.

Typical area hierarchy:

  • Level 1: Plant (e.g., "Chemical Plant")
  • Level 2: Unit (e.g., "Distillation Unit")
  • Level 3: Sub-unit (e.g., "Column 1")
  • Level 4+ Equipment groups (e.g., "Feed System", "Product System")

Tagname vs. Alias Toggle

The Model View offers two ways to identify objects:

ModeShowsExample
Tagname The full hierarchical path from the root $Area1\Distillation\Column1\FeedPump
Alias A user-friendly, human-readable name Feed Pump 101

Toggle between the two views using the Tagname/Alias button in the toolbar. Alias view is often easier for operators to read; Tagname view is more precise for engineering.

Containment Display

The Model View shows which objects are contained within each area. This containment hierarchy is the same one enforced by the templates — an object can only be placed where its parent template allows it.

The Model View — Tagname view (left) shows full paths; Alias view (right) shows human-readable names. Both represent the same Plant Model hierarchy (Module 1, page 1-26)

2.7 Deployment View

Reference: Module 1, Section 4 (page 1-27)

The Deployment View shows your Deployment Model — which computers, engines, and objects are part of the running application.

Deployment Model Structure

The Deployment Model represents the runtime infrastructure of your Galaxy. At its core is the $WinPlatform object, which represents a physical computer that hosts engines and objects.

Key elements of the Deployment View:

ElementPurpose
$WinPlatform Represents a physical computer; must be deployed before anything else can run on it
Engines Runtime engines that execute object logic (e.g., AppEngine, Alarm Engine)
Object Deployment Shows which objects are deployed to which platform/engine
Unassigned Host Temporary holding area for objects not yet assigned to a platform

Deployment Workflow

The deployment process follows this order:

  1. Deploy $WinPlatform — claims the computer for the Galaxy
  2. Deploy engines — starts the runtime services on the platform
  3. Deploy objects — sends configuration to the engines
  4. Start engines — begins actual execution
The Deployment View — $WinPlatform objects host engines and deployed objects. The Unassigned Host folder holds objects not yet placed on a platform (Module 1, page 1-27)
Deployment order matters: You must deploy the $WinPlatform first, then engines, then objects. Deploying an object before its platform is ready will fail. The IDE enforces this ordering automatically.

2.8 Derivation View

Reference: Module 1, Section 4 (page 1-28)

The Derivation View shows the parent-child relationships between templates and instances — the inheritance hierarchy of your objects.

Parent-Child Relationships

Every object in the Galaxy is either a base template or is derived from a base template. The Derivation View shows this hierarchy. Changes to a parent template propagate to all child instances, unless a child has overridden that property.

Example hierarchy:

  • $Pump (base template) → defines default attributes like speed, status, alarm limits
  • $Pump_Drive (derived template) → adds drive-specific attributes
  • Pump_001 (instance) → inherits from $Pump_Drive, has specific values

Change Propagation Impact

When you modify a base template, the change cascades to all derived objects. This is powerful but requires caution:

Change TypeImpact
Adding a new attribute New attribute appears in all derived objects (no existing values affected)
Modifying an existing attribute default Only affects instances that have NOT overridden that attribute
Removing an attribute Attribute is removed from all derived objects — potential data loss!
Modifying a script All derived objects inherit the new script (unless overridden)

Unused Base Templates Folder

The Derivation View includes an Unused Base Templates folder. This shows template objects that have been defined but have no instances derived from them. Review this folder periodically to clean up unused templates.

The Derivation View — parent-child hierarchy showing how base templates propagate to derived instances. Changes to parents cascade to all children (Module 1, page 1-28)
Before modifying a base template: Use the Derivation View to see how many objects will be affected. A change to a widely-used base template can impact hundreds of objects.

2.9 IO Devices View

Reference: Module 1, Section 4 (page 1-29)

The IO Devices View shows your device integration objects — the objects that connect your Galaxy to field devices (PLCs, RTUs, sensors).

Device Integration Objects

Each communication path to a field device is represented by a device integration object. These objects define:

  • Communication protocol (OPC, SuiteLink, DDE)
  • Device address (IP address, node name)
  • Scan parameters (poll rate, timeout)

Scan Groups

Scan groups organize I/O objects that share the same communication characteristics. Objects in the same scan group are polled together at the same interval, improving communication efficiency.

Autobinding

Autobinding is a feature that automatically links I/O object attributes to device integration objects. When you name an I/O attribute to match a tag in the field device, the system can automatically create the binding — saving you from manually configuring every connection.

Unassigned IO Device Folder

Device integration objects that have not yet been assigned to a scan group appear in the Unassigned IO Device folder. All device objects must be placed in a scan group before deployment.

The IO Devices View — device integration objects (like Simulator) organize communication with field devices. Objects are grouped into scan groups for efficient polling (Module 1, page 1-29)

2.10 Mapping and Operations Views

Reference: Module 1, Section 4 (page 1-30)

The Mapping View and Operations View provide validation and monitoring for your Galaxy configuration.

Mapping View — I/O Reference Validation

The Mapping View helps you verify that all I/O references in your objects are correctly bound. It shows:

  • I/O references — every attribute that reads from or writes to a field device
  • Autobinding status — whether bindings are auto-mapped or manually configured
  • Binding conflicts — any objects with duplicate or mismatched bindings

Autobinding Overrides

When autobinding assigns an I/O reference, you can override it to point to a different device tag. The Mapping View shows which bindings are auto-mapped and which have been manually overridden.

Operations View

The Operations View shows the status of deployment operations:

  • Deploy status — which objects are deployed, pending, or have errors
  • Start/Stop status — engine and object runtime states
  • Validation results — errors found during configuration checks
Use the Mapping View before deploying to catch I/O binding errors early. An incorrect binding means your object will not read or write data correctly at runtime.
The Mapping View (left) shows I/O reference validation and autobinding status. The Operations View (right) tracks deployment operations and validation results (Module 1, page 1-30)

2.11 Keyboard Shortcuts and Navigation Tips

Reference: Module 1, Section 4 (general IDE navigation)

Efficient navigation in the IDE saves significant time. Here are the essential shortcuts and tips:

Tree View Keyboard Shortcuts

KeyActionWhen to Use
* (asterisk / multiply) Expand all nodes in the current tree When you need to see the complete hierarchy at a glance
+ (plus) Expand one level of the current node When you want to drill down gradually through the hierarchy
- (minus) Collapse the current node When you want to hide details and clean up the view

View Menu

Not all views are visible by default. To show or hide views:

  1. Go to the View menu in the ribbon
  2. Select the view you want to toggle on/off
  3. The view appears or disappears from the bottom-left pane

Available views include: Model, Deployment, Derivation, IO Devices, Mapping, and Operations.

Sync Option

The Sync option synchronizes the selected view with the current object selection. When enabled, selecting an object in one view will automatically navigate to that same object in the other views. This is invaluable for understanding how an object appears across different perspectives.

Reset Layout

If your IDE windows get rearranged or lost, you can reset the layout:

  1. Go to View → Reset Layout
  2. The IDE returns to its default window arrangement
Power user tip: Use the Sync feature regularly. When you select an object in the Model View, enabling Sync will show you its deployment status, derivation hierarchy, and I/O bindings — all at once. It's the fastest way to get a complete picture of any object.

Quiz — Test Your Understanding

Q1: What is the relationship between the IDE and the Galaxy Repository?

Click to reveal answer

The IDE is a client that sends commands to the Galaxy Repository (the server). All operations — deploy, import, export — are executed by the GR, not by the IDE. The IDE is a remote control.

Q2: Which view should you use to see the plant area hierarchy (areas, equipment, objects)?

Click to reveal answer

The Model View. It shows the Plant Model hierarchy with $Area objects at the top level, and supports both Tagname (full path) and Alias (human-readable) display modes.

Q3: What happens to child objects when you modify a base template in the Derivation View?

Click to reveal answer

Changes propagate to all derived instances. If a child has overridden a specific attribute, the parent change does not affect that attribute on the child. Otherwise, the change cascades down.

Q4: In which view do you verify that I/O bindings are correctly configured before deployment?

Click to reveal answer

The Mapping View. It shows all I/O references, their autobinding status, and any binding conflicts. Check this view before deploying to catch errors early.

Q5: What are the keyboard shortcuts to expand all nodes (*) and collapse nodes (-) in a tree view?

Click to reveal answer

Press * (asterisk) to expand all nodes, + (plus) to expand one level, and - (minus) to collapse nodes. These work in any tree view in the IDE.

Key Takeaways

  1. The IDE is a client — all commands are sent to the Galaxy Repository for execution.
  2. The IDE has two tools (Templates and Graphics) and six application views (Model, Deployment, Derivation, IO Devices, Mapping, Operations).
  3. Templates Tab — browse and create object templates; protected templates (orange lock) cannot be modified.
  4. Graphics Tab — browse and use graphic symbols from Industrial Graphics or Situational Awareness libraries.
  5. Model View — plant hierarchy using $Area objects (up to 10 levels deep); toggle Tagname/Alias display.
  6. Deployment View — platforms, engines, and object deployment; deploy order is Platform → Engine → Objects.
  7. Derivation View — template inheritance hierarchy; changes to parents propagate to children.
  8. IO Devices View — device integration objects, scan groups, and autobinding.
  9. Mapping View — validate I/O bindings before deployment.
  10. Use keyboard shortcuts (*, +, -) and the Sync feature for efficient navigation.
