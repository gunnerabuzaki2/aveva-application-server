Lesson 10: Data Simulation + Module 3 Summary

Source: AVEVA Application Server 2023 Training Manual, Module 3, Section 5 + Lab 5 (pages 3-55 to 3-63)

Simulating real-world process data — using the OI Simulation Server and $OPCClient to build a realistic test environment without physical PLC hardware.

10.1 Data Simulation Overview

Reference: Module 3, Section 5 (pages 3-55 to 3-56)

In a production environment, Application Server objects communicate with real hardware — PLCs, RTUs, sensors, and actuators via communication drivers. During development and testing, however, you typically do not have physical I/O available at your workstation. Data simulation solves this problem.

OI Simulation Server (OI.SIM) is a software-based OPC DA server that generates simulated process data — analog values, digital states, and more. It produces realistic data patterns (ramp, sine, step, random, constant) so you can fully test your Galaxy objects, scripts, and alarms without connecting to any physical devices.

The simulation architecture connects three components:

Data simulation overview — OI.SIM generates simulated data; $OPCClient reads it into the Galaxy via OPC DA (Module 3, page 3-55)
  1. OI Simulation Server (OI.SIM) — runs locally as a Windows service, exposing simulated tags as an OPC DA server. Each tag is configured with an algorithm (ramp, sine, step, random, or constant) and parameters like update rate and value range.
  2. $OPCClient — the built-in Galaxy OPC client object that connects to OI.SIM by specifying the OPC server name. It translates OPC tags into Galaxy-accessible attributes.
  3. I/O References on object attributes — you link individual object attributes to specific OI.SIM tag paths, so the attribute receives the simulated value at runtime.

Why Simulate?

BenefitDescription
No hardware requiredDevelop and test without physical PLCs or field I/O at your desk
Reproducible scenariosGenerate consistent data patterns for repeatable testing
Safe testingTest alarm conditions, scripting logic, and edge cases without risk to real equipment
Rapid iterationChange simulation parameters instantly — no waiting for field-side modifications
Full stack validationVerify the entire data path: OI.SIM → OPC → $OPCClient → I/O Reference → Object Attribute → Visualization

10.2 Lab 5 — Data Simulation: Connecting OI.SIM to Your Galaxy

Reference: Module 3, Lab 5 (pages 3-57 to 3-63)

Lab Objectives:
  • Configure OI.SIM PLC simulation parameters (algorithm, rate, range)
  • Configure $OPCClient to connect to OI.SIM as its OPC server
  • Set up I/O references linking object attributes to OI.SIM tag paths
  • Deploy the Galaxy and verify simulated data using Object Viewer
Lab 5 introduction — objectives for connecting OI.SIM simulator to the Galaxy (Module 3, page 3-57)

10.3 Step 1 — Configure OI.SIM Simulation Parameters

Reference: Module 3, Lab 5 (page 3-58)

Prerequisite: Ensure the OI Simulation Server is installed and running on your machine. It typically runs as a Windows service. Open the OCMC (Operations Control Management Console) to view and configure its nodes.

Before connecting $OPCClient to OI.SIM, you should review and configure the simulation parameters for the PLC node. Open the OCMC and expand the OI.SIM server tree to locate the PLC configuration.

OCMC showing PLC_TEST configuration — Algorithm set to Random, Rate = 200 ms, Range = 20, Force Quality = Good (Module 3, page 3-58)

Key simulation parameters for each PLC tag:

ParameterExample ValueDescription
AlgorithmRandomThe pattern the simulated value follows: Random, Ramp, Sine, Step, or Constant
Rate200 msHow often the simulated value updates (update interval)
Range20The spread of values for random/ramp simulations (e.g., 0 to Range)
Force QualityGoodSets the quality field on the OPC data — "Good" simulates healthy field communication
Watch out: If "Force Quality" is set to Bad or Uncertain, your objects will see bad-quality data even though the values themselves look correct. This is useful for testing bad-quality handling in scripts, but can cause confusion if unintentional.

In this lab, the PLC is configured with Algorithm = Random, Rate = 200 ms, Range = 20, and Force Quality = Good. This produces a randomly varying value that updates five times per second with good quality — a realistic simulation of a noisy sensor reading.

10.4 Step 2 — Configure $OPCClient to Connect to OI.SIM

Reference: Module 3, Lab 5 (page 3-59)

The next step is to tell the Galaxy's $OPCClient object to use OI.SIM as its data source. The $OPCClient is a built-in system object that handles OPC DA communication.

Step 2a: In the IDE, open your Galaxy and navigate to the $OPCClient object in the Galaxy hierarchy (it resides under the platform's system area).

Step 2b: Double-click $OPCClient to open its configuration properties. Locate the OPC server connection settings.

Step 2c: Set the OPC Server Name field to OI.SIM. This is the registered OPC server name that OI.SIM exposes to the system.

$Templates pane — right-click SgOPCClient to create a new Instance for connecting to OI.SIMated data via OPC DA (Module 3, page 3-59)

Step 2d: Apply and save the configuration. The $OPCClient will now attempt to connect to OI.SIM when the Galaxy runs.

Common error: If OI.SIM is not running or the OPC server name is misspelled, the $OPCClient connection will fail. All attributes with I/O references through $OPCClient will show Unknown quality and stale values. Always verify OI.SIM is running in OCMC before troubleshooting further.

10.5 Step 3 — Set Up I/O References for Simulated Data

Reference: Module 3, Lab 5 (page 3-60)

With $OPCClient connected to OI.SIM, the next step is to link your object attributes to specific simulated data points. This is done through I/O References — each reference specifies the OPC tag path that an attribute should read from.

Step 3a: Open the object that will receive simulated data (for example, an object under the Simulator area of your Galaxy model). Navigate to the attributes you want to link to simulation data.

Step 3b: For each attribute, set the I/O Reference field to the appropriate OI.SIM tag path. The tag path format follows OPC tag naming conventions — it typically includes the PLC name and tag name as configured in OI.SIM.

Configuring I/O references — linking object attributes to OI.SIM tag paths so they receive simulated values at runtime (Module 3, page 3-60)

Step 3c: Verify the I/O references are correct. Each reference should point to a valid tag path in the OI.SIM server. A mismatch will result in Unknown quality at runtime.

Tip: You can use the OCMC to browse the OI.SIM tag tree and confirm exact tag paths before entering them as I/O references. This avoids typos and ensures the paths match what OI.SIM actually exposes.

10.6 Step 4 — Organize the Galaxy Model for Simulation

Reference: Module 3, Lab 5 (page 3-61)

The Galaxy model view provides a visual hierarchy of all objects. In this lab, the simulated objects are organized under a dedicated area in the model.

Galaxy model view — Plant > ControlSystem > Simulator area highlighted, alongside AppEngine1, GRPlatform, ProdPlatform, Packaging, Production areas (Module 3, page 3-61)

The model tree shows the full organizational hierarchy:

  • Plant — the top-level container
    • ControlSystem — communication and control objects
      • Simulator — objects receiving data from OI.SIM (highlighted)
    • AppEngine1 — the application engine platform
    • GRPlatform, ProdPlatform — platform objects
    • Packaging — packaging area objects
    • Production — contains Line1, Line2, Receiving, Shipping
Why a dedicated Simulator area? Keeping simulation objects in their own area makes it easy to deploy and undeploy them independently. When you connect to a real PLC later, you can remove the Simulator area and update the I/O references to point to actual hardware — without disturbing the rest of the model.

10.7 Step 5 — Deploy and Verify Simulated Data

Reference: Module 3, Lab 5 (pages 3-62 to 3-63)

With all configuration complete, deploy the objects to runtime and verify the data flow.

Step 5a: Right-click the modified objects (or the Simulator area) and select Deploy. Wait for deployment to complete successfully.

Deploying simulated objects to runtime — the IDE pushes the updated configuration to the Application Engine (Module 3, page 3-62)

Step 5b: After deployment completes, open the OCMC to verify that OI.SIM is communicating correctly. Expand the OI.SIM server node to see the PLC structure.

OCMC showing OI.SIM.1 expanded — Configuration > PORT > PLC hierarchy visible; Platform Manager showing TrainingGalaxy(DefaultUser) (Module 3, page 3-63)

The OCMC confirms:

  • OI.SIM.1 is the running OPC server instance
  • Under it: Configuration > PORT > PLC — the hierarchical structure of simulated endpoints
  • The Platform Manager shows TrainingGalaxy(DefaultUser) — confirming the Galaxy is connected and active

Step 5c: Open Object Viewer on one of the deployed simulator objects. You should see live values updating at the configured rate (e.g., every 200 ms for the Random algorithm).

Verification checklist:
  • Are attribute values changing? — If yes, the OPC data path is working end to end
  • Is the quality field "Good"? — Confirms Force Quality = Good in OI.SIM
  • Are values within the expected range? — Confirms the Range parameter
  • Is the update frequency correct? — Confirms the Rate parameter
Troubleshooting — Object Viewer shows "Unknown" or stale values:
  • Verify OI.SIM is running (check OCMC or Windows Services)
  • Confirm the OPC server name in $OPCClient matches OI.SIM exactly
  • Verify the I/O reference tag paths match the OI.SIM tag structure
  • Ensure the object was deployed after making I/O reference changes
  • Check the Alarm Summary for OPC communication errors
