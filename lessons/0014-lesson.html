14.1 Containment — Organizing Objects Within Objects

Reference: Module 4, Section 4 (pages 4-45 to 4-47)

In the previous lessons, we modeled individual equipment components — pumps, valves, sensors, and instruments — as separate Galaxy objects. In a real plant, however, these components don't exist in isolation. A mixer contains an agitator, inlets, an outlet, level and temperature sensors, and pumps. Containment is the mechanism that lets you model these physical relationships directly in the Galaxy.

Containment is the ability of one Galaxy object to contain other objects within it, creating a parent-child hierarchy. Containment models the physical structure of real equipment — a mixer physically contains pumps, sensors, and inlets, so in the Galaxy, the Mixer object contains those component objects.

Containment is not just organizational grouping. When an object is contained within another object:

  • The contained object's fully qualified name reflects the hierarchy (e.g., Plant.Production.Line1.Mixer100.Pump1_001)
  • The contained object lives inside the parent object in the Galaxy Model tree
  • The parent object can reference and interact with its contained objects via scripts and attribute bindings
  • When the parent is deployed, all contained objects are deployed together as a unit
  • The containment relationship is part of the object's design — it is defined in templates, not at runtime

Containment vs. Other Relationships

RelationshipWhat It DoesWhen to Use
Containment Parent physically contains child — child is part of the parent's design Components that are integral to an equipment piece (sensors built into a mixer, pumps attached to a vessel)
Reference One object references another by attribute path — objects remain separate Objects that need to read/write data from other objects but are not physically part of them
Inheritance Child template derives from parent template — shares structure and behavior Creating reusable equipment templates (all pumps share common attributes)
Think physically: If you would need a wrench to remove component X from equipment Y, then X should be contained by Y in the Galaxy. If X and Y just need to share data, use references instead.

Naming convention: The fully qualified name always starts from the $Plant object and flows down through each level of containment. The dot-separated path mirrors the hierarchy exactly. This makes it easy to locate any object and understand its position in the plant structure.

Controlling Containment

Containment is defined by the Container attribute on any object. When you drag an object onto another in the Model pane, or when you set the Container attribute in the Object Viewer, you establish the containment relationship.

Important: Containment is a design-time decision. Once you set the Container attribute on an object, that object becomes part of the parent. Moving or removing a contained object requires updating the Container attribute — you cannot simply delete the parent without addressing the children.

14.3 Containment Benefits

Reference: Module 4, Section 4 (pages 4-51 to 4-53)

Containment provides several practical benefits for building and managing a Galaxy:

  • Logical organization: The Galaxy Model tree mirrors the physical plant, making it easy to navigate large applications with thousands of objects
  • Deployment efficiency: Deploying a parent object automatically deploys all contained children — no need to deploy each sensor and pump individually
  • Cascade deletion: Deleting a parent object removes all its contained children, preventing orphaned objects
  • Attribute scoping: Scripts within the parent can easily reference contained objects using relative paths (e.g., Me.Pump1_001.Running)
  • Template reuse: Once a Mixer template is built with its contained components, every Mixer instance in the plant inherits the same structure automatically

Q1: What is containment in Application Server, and how does it differ from a reference relationship?

Click to reveal answer

Containment means one object physically contains another — the child is part of the parent's design and is deployed/deleted with it. A reference is a data link between two independent objects that remain separate in the Galaxy hierarchy.

Q3: What happens when you deploy a parent object that contains children?

Q4: What attribute on an object defines its containment relationship?

The Container attribute — it specifies the fully qualified name of the parent object that contains this object.

Key Takeaways

  1. Containment models physical relationships — parent objects contain child objects as integral components.
  2. The Container attribute defines the containment relationship for each object.
  3. Fully qualified names reflect the containment hierarchy (e.g., Plant.Production.Line1.Mixer100.Pump1_001).
  4. Deploying a parent automatically deploys all contained children.
  5. The Mixer model demonstrates all Module 4 concepts together: templates, inheritance, containment, and attribute configuration.
  6. Use containment for integral components; use references for data links between independent objects.
