Lesson 23: Alarms and Events Overview

Source: AVEVA Application Server 2023 Training Manual, Module 7 Section 1

Comprehensive overview of alarm types, alarm modes, alarm attributes, severities, aggregation, shelving, and the role of the Platform in alarm/event infrastructure. (pages 7-3 through 7-19)

23.1 Events and Alarms — What's the Difference?

Reference: Module 7, pages 7-3 through 7-5

Before diving into the alarm system, it is essential to understand the distinction between an event and an alarm. These two terms are often used interchangeably in casual conversation, but in the context of Application Server, they carry very different meanings — and understanding that difference is key to using the platform effectively.

An event is a general occurrence of a condition at a specific point in time. Events represent state changes in your system and are not inherently abnormal. They simply record that something happened. For example, a plant process started, an operator changed a setpoint, a new object was deployed to a platform, a platform recovered from a failure, or a user logged in. All of these are events. They are recorded and can be viewed in history, but they do not require any action from an operator.

An alarm, on the other hand, is a special, prioritized type of event that signals an abnormal condition requiring immediate attention. When a process measurement exceeds a configured limit, a device fails to reach its desired state, or hardware moves outside acceptable limits, the system generates an alarm. Alarms are distinguished from ordinary events by the fact that they indicate something is wrong — and that someone needs to respond.

Key Concept: Events vs. Alarms
An event records that something happened (state change, operator action, system recovery). An alarm is a prioritized event that signals an abnormal condition requiring operator attention. All alarms are events, but not all events are alarms.

It is also important to note what falls outside the scope of events and alarms entirely. Galaxy Repository configuration actions — such as modifying a template or changing object properties in the IDE — are not events or alarms. Similarly, Bootstrap installation activities and Logger messages are not categorized as events or alarms within the Application Server framework.

23.2 The Platform as Alarm Provider

Reference: Module 7, pages 7-5 through 7-6

The Application Server Platform plays a central role in the alarm and event infrastructure. Every alarm and event that originates within your system is routed through the Platform, which acts as the bridge between the Alarm/Event Notification Distributors (such as AVEVA Alarm Notification or AVEVA Historian) and the distributed alarm infrastructure.

When you subscribe an Area to alarm and event notifications, the Platform collects all alarms and events from that Area and routes them to the appropriate subscribers. It also handles acknowledgments, enabling, and disabling operations — and it tracks which user performed each action for security and auditing purposes.

This architecture means that the Platform is the single point through which all alarm traffic flows. If you need to understand why an alarm was or was not delivered to a particular client, the Platform's routing behavior is the first place to look.

23.3 Types of Alarms

Reference: Module 7, pages 7-6 through 7-10

Application Server supports five distinct types of alarms, each designed to detect a specific kind of abnormal condition. Understanding when to use each type is fundamental to building an effective alarm strategy.

23.3.1 Limit Alarms

Limit alarms are the most common type and the most straightforward to configure. They trigger when a process value crosses a defined threshold. Application Server provides four configurable limits for any numeric attribute:

LimitDescription
LoLoLow-Low limit — critical low threshold
LoLow limit — warning low threshold
HiHigh limit — warning high threshold
HiHiHigh-High limit — critical high threshold

Each limit is individually configurable: you can enable or disable it, set its threshold value, assign a priority, and define a custom alarm message. The diagram below shows how the four limit levels relate to a typical process value trending over time.

Limit alarm levels — LoLo, Lo, Hi, and HiHi thresholds shown relative to a process value trending over time

Two important concepts govern how limit alarms behave once triggered: the alarm deadband and the time deadband.

The alarm deadband defines how far the process value must return toward the normal range before the alarm clears. For example, if the HiHi limit is set to 80 EU with a deadband of 5 EU, the process value must drop below 75 EU (80 − 5 = 75) before the alarm is cleared. This prevents rapid alarm toggling when a value hovers near the threshold.

The time deadband defines the minimum duration the value must remain outside the threshold before the alarm condition is actually reported. This filters out momentary spikes that do not represent a genuine alarm condition.

Limit alarm configuration — each limit can be individually enabled with its own threshold, priority, and message
Key Concept: Alarm Deadband Prevents Alarm Chatter
The alarm deadband ensures a process value must return a specified distance past the threshold before the alarm clears. Without it, values fluctuating near the threshold would cause the alarm to rapidly toggle on and off — a condition known as "alarm chatter."

23.3.2 Deviation Alarms

Deviation alarms compare a measured value against a target setpoint rather than against a fixed threshold. The system calculates the absolute difference between the current value and the target engineering units, then compares that difference against configured minor and major deviation limits.

Two additional parameters govern deviation alarm behavior: the settling period, which is the time after a device starts during which no alarms are generated (allowing the process to stabilize), and the deviation deadband, which defines how far the deviation must decrease before the alarm clears.

Alarm deadband and deviation alarm concept — the measured value is compared against a target, and absolute deviation is checked against minor and major limits
Deviation alarm configuration — minor and major deviation limits, settling period, and deadband are all configurable per alarm

23.3.3 Rate of Change (ROC) Alarms

Rate of Change alarms detect when a process value is changing too quickly — regardless of whether it is within acceptable absolute limits. The system calculates the slope of change as the absolute difference between the current and previous values divided by the time interval: Slope = |Current − Previous| / Interval.

ROC alarm limits can be configured in engineering units per second, per minute, per hour, or per day, giving you fine-grained control over the sensitivity of the alarm. Both upward and downward rate limits are configurable independently.

ROC alarm example — a tank level changes too quickly, exceeding the configured rate-of-change threshold
ROC alarm configuration — upward and downward rate limits are set independently in engineering units per second, minute, hour, or day

23.3.4 Bad Value Alarms

Bad Value alarms trigger when an attribute returns a bad quality status. This typically occurs when a data source becomes unavailable, a communication link fails, or a value cannot be reliably determined. Rather than reporting a stale or incorrect value, the system flags the condition as an alarm so operators know the data they are relying on may be compromised.

23.3.5 Diagnostic Alarms

Diagnostic alarms are system-level alarms generated by the platform itself to indicate infrastructure problems — such as a platform failing, a network interruption, or a device going offline. These alarms are not tied to a specific process measurement but rather to the health and availability of the system components.

Bad value alarms and plant states — the five default plant states and their associated alarm modes

23.4 Plant State-Based Alarms

Reference: Module 7, pages 7-10 through 7-11

Application Server provides a powerful mechanism for tying alarm behavior to the operational state of your plant. Rather than manually enabling and disabling alarms when the plant transitions between operational modes, you can map alarm modes to plant states and have the system automatically apply the correct alarm configuration.

Every Galaxy ships with five default plant states:

Plant StateDefault Alarm ModeEditable?
RunningEnableRead-only (can rename, but alarm mode cannot change from Enable)
MaintenanceDisableYes
StartupSilenceYes
ShutdownDisableYes
TestingSilenceYes

You can create new plant states, rename existing ones, and delete states you no longer need — with one exception. The Running state can be renamed, but its alarm mode is locked to Enable. This safeguard ensures that when the plant is in its normal operating condition, alarms are always active.

To assign a plant state to an Area, you set the PlantState attribute on the Area object. When you do this, the PlantAlarmMode attribute automatically updates to reflect the alarm mode associated with that state.

Alarms and Events configuration — the Plant States tab where default states and their associated alarm modes are defined and managed

23.5 Enabling, Silencing, and Disabling Alarms

Reference: Module 7, pages 7-11 through 7-13

Alarms in Application Server can exist in three distinct modes: Enabled, Silenced, or Disabled. These modes can be applied at any level of the containment hierarchy — on an Area, a container, an individual object, or even a single alarm.

Enabled is the most permissive state. When alarms are enabled, all alarm conditions are reported to subscribed clients and recorded in the historian. This is the normal operating mode.

Silenced is more restrictive. In this mode, alarms are still detected, still saved to the historian, and still sent to clients — but they are not displayed. This is useful during planned maintenance or startup phases when you want alarms to be recorded for later analysis but do not want operators to be distracted by known conditions.

Disabled is the most restrictive state. When alarms are disabled, no alarm conditions are detected at all. There is nothing to acknowledge because no alarms are generated. Use this mode sparingly, as it means the system is completely blind to abnormal conditions.

Warning: Most Restrictive Wins
When alarm modes are configured at different levels of the containment hierarchy, the most restrictive setting always wins. If an Area has alarms enabled but a child object has them silenced, the child's alarms will be silenced. If a parent has alarms disabled, all children inherit that disabled state regardless of their own settings.

23.6 Alarm Mode Attributes

Reference: Module 7, pages 7-12 through 7-14

Understanding how alarm modes are determined requires familiarity with three key attributes that work together in a parent-child hierarchy:

AlarmModeCmd is a writeable attribute that allows you to set the commanded alarm state for an object. You can write Enable, Silence, or Disable to this attribute to control the alarm mode directly.

AlarmInhibit is a writeable Boolean attribute that acts as a kill switch. When set to TRUE, all alarms for that object and every object contained within it are immediately disabled. When set back to FALSE, the inhibition is removed and the system recalculates alarm mode based on other factors.

AlarmMode is a calculated, read-only attribute that represents the effective alarm mode after the system has evaluated AlarmModeCmd, AlarmInhibit, and the parent's AlarmMode. The most restrictive setting among all of these always determines the final value.

Alarm mode attribute hierarchy — AlarmModeCmd and AlarmInhibit combine with the parent's AlarmMode to determine the effective AlarmMode for each object

The parent-child hierarchy is important to understand. If a parent object has AlarmInhibit set to TRUE, all children are disabled regardless of their own AlarmModeCmd settings. However, individual alarms within an object are autonomous — changing the alarm mode on one alarm does not affect the others.

Key Concept: Alarm Inhibition Propagates Downward
Setting AlarmInhibit to TRUE on a parent object disables alarms for the parent and all contained children. This is a powerful tool for temporarily suppressing large groups of alarms, but use it with care — it affects everything beneath the inhibited object.

23.7 Alarm Attributes Reference

Reference: Module 7, pages 7-14 through 7-15

Every alarm in Application Server exposes a set of attributes that control its behavior and provide runtime information. The following table summarizes the key alarm attributes:

Notice the difference between .Inhibit (singular, per-alarm) and AlarmInhibit (on the object). The singular .Inhibit disables only the specific alarm it belongs to, while the object-level AlarmInhibit affects all alarms on the object and its children.

23.8 Alarm Severities

Reference: Module 7, pages 7-15 through 7-16

Alarm severities provide a way to classify alarms by their urgency and importance. Application Server defines four severity levels, each mapped to a default range of priority values:

SeverityDefault Priority RangeTypical Use
Critical (1)1 – 250Safety-critical conditions requiring immediate response
High (2)251 – 500Significant process upsets requiring prompt attention
Medium (3)501 – 750Operational deviations that should be investigated soon
Low (4)751 – 999Informational conditions or minor deviations

At runtime, the system ranks alarms using this hierarchy: Alarm Mode > InAlarm > Acked > Severity Level. This means the alarm mode (enabled/silenced/disabled) takes precedence over whether the alarm is active, which in turn takes precedence over whether it has been acknowledged, and severity is considered last.

Alarm severity levels and their default priority ranges — priorities are configured per alarm using the .Priority attribute

23.9 Alarm Aggregation

Reference: Module 7, pages 7-16 through 7-17

When your system contains hundreds or thousands of alarms spread across many Areas and objects, it becomes impractical to monitor each one individually. Alarm aggregation provides an efficient way to identify whether alarms exist anywhere within a containment hierarchy by rolling up alarm states from child objects into summary attributes on the parent.

The aggregation system tracks four states: UNACK_ALM (unacknowledged active alarm), ACK_ALM (acknowledged active alarm), UNACK_RTN (unacknowledged return-to-normal), and ACK_RTN (acknowledged return-to-normal). However, only the first three are aggregated — ACK_RTN is not aggregated. Additionally, silenced alarms are still included in the aggregation.

Five read-only attributes on Area objects provide the aggregated information:

AttributeDescription
AlarmCntsBySeverityCount of active alarms broken down by severity level
AlarmMostUrgentAckedPriority of the most urgent acknowledged active alarm
AlarmMostUrgentInAlarmPriority of the most urgent active alarm (regardless of acknowledgment)
AlarmMostUrgentModeMode of the most urgent active alarm
AlarmMostUrgentSeveritySeverity of the most urgent active alarm

To enable aggregation on an Area, you check the "Aggregate Alarms" checkbox in the SgArea Editor. Once enabled, the Area and all objects beneath it will contribute to the aggregation attributes.

SgArea Editor — the Aggregate Alarms checkbox enables alarm aggregation for the Area and all objects it contains
Warning: ACK_RTN Is Not Aggregated
When an alarm returns to normal and the operator acknowledges that return, the ACK_RTN state is not rolled up into aggregation attributes. This means that areas with only acknowledged return-to-normal conditions will not show active aggregation indicators. Keep this in mind when designing your alarm display screens.

23.10 Monitoring Alarms at Runtime

Reference: Module 7, pages 7-17 through 7-18

At runtime, you can monitor alarm activity directly through the Object Viewer. The Object Viewer provides a real-time view of alarm attributes including the current alarm state, acknowledgment status, timestamps, and priority. This is the primary tool for operators who need to interact with alarms during normal plant operations.

Object Viewer alarm monitoring — real-time display of alarm attributes including state, acknowledgment status, priority, and timestamps

23.11 Shelving Alarms

Reference: Module 7, pages 7-18 through 7-19

Shelving is a mechanism for temporarily hiding alarms from operator view without disabling them entirely. When an alarm is shelved, it continues to be detected and historized, but it does not appear in alarm displays. Shelved alarms are automatically unshelved after a configurable timeout period.

Shelving differs from silencing in several important ways:

FeatureSilencingShelving
ScopeApplies to Areas, containers, objects, or individual alarmsApplies only to individual alarms
TimeoutNo automatic timeoutHas a built-in timeout — alarm returns after expiration
Access ControlStandard alarm mode controlsRole-based access controls who can shelve alarms
Audit TrailTracks user who changed modeTracks user, workstation, reason, start time, and expiration
HistorizationAlarms are historizedAlarms are still historized when shelved

Seven attributes support the shelving mechanism:

AttributeDescription
AlarmShelveCmdWriteable command to shelve or unshelve an alarm
AlarmShelvedRead-only indicator of whether the alarm is currently shelved
AlarmShelveStartTimeTimestamp when the alarm was shelved
AlarmShelveStopTimeTimestamp when the alarm will be automatically unshelved
AlarmShelveReasonFree-text reason for shelving the alarm
AlarmShelveUserUser who performed the shelving operation
AlarmShelveNodeWorkstation from which the shelving was performed

To use shelving, you must first enable it at the Area level through the Galaxy configuration. Navigate to Galaxy > Configure > Galaxy > Alarms and Events, and enable shelving for the relevant Areas. Individual alarms can then be shelved at runtime through the Object Viewer.

Shelving configuration — the Alarms tab provides shelving controls with timeout settings and reason documentation
Object Viewer shelving attributes — runtime display of shelving status, timestamps, user, workstation, and reason
Tip: Shelving vs. Silencing — When to Use Which?
Use silencing when you want to suppress alarm displays for an entire Area or object during a known condition (like startup). Use shelving when you need to temporarily hide a specific alarm with a documented reason and automatic timeout — for example, during temporary maintenance on a single instrument.

23.12 Summary

In this lesson, we covered the complete alarm and event infrastructure in AVEVA Application Server. We distinguished between events (general state changes) and alarms (prioritized abnormal conditions), explored the five alarm types (Limit, Deviation, ROC, Bad Value, and Diagnostic), and understood how the Platform routes alarms through the distributed infrastructure.

We examined the three alarm modes — Enabled, Silenced, and Disabled — and how the parent-child hierarchy enforces the most restrictive setting. We reviewed the key alarm attributes, severity levels, and the aggregation mechanism that allows you to monitor alarm status across entire containment hierarchies. Finally, we explored shelving as a targeted, time-limited mechanism for hiding individual alarms with full audit tracking.

Quiz: Test Your Understanding

Q1: An operator says, "The tank level is at 78 EU and I'm seeing the HiHi alarm active." The HiHi limit is set to 80 EU with a deadband of 5 EU. The level then drops to 76 EU. Is the alarm cleared? Why or why not?

Click to reveal answer

No, the alarm is not cleared. With a HiHi limit of 80 EU and a deadband of 5 EU, the process value must drop below 75 EU (80 − 5 = 75) before the alarm clears. At 76 EU, the value has not returned far enough past the threshold.

Q2: You set AlarmInhibit to TRUE on the Line1 Area. What is the effect on alarms on objects inside Line1/Mixer100?

Click to reveal answer

All alarms for every object contained within Line1 — including Mixer100 and everything inside it — are immediately disabled. Setting AlarmInhibit to TRUE acts as a kill switch that propagates downward through the entire containment hierarchy.

Q3: What is the difference between .Inhibit (singular, on an individual alarm) and AlarmInhibit (on an object)?

Click to reveal answer

.Inhibit disables only the single alarm it belongs to and has no effect on any other alarms on the same object. AlarmInhibit (the object-level attribute) disables ALL alarms for that object and every object contained beneath it in the hierarchy.

Q4: Name the three alarm states that are aggregated, and the one that is NOT.

Click to reveal answer

The three aggregated states are UNACK_ALM, ACK_ALM, and UNACK_RTN. ACK_RTN (acknowledged return-to-normal) is NOT aggregated.

Q5: You need to temporarily hide a single alarm on Mixer100 for 30 minutes while you perform maintenance on the sensor. The alarm should still be recorded in the historian, and it should come back automatically. Should you use silencing or shelving? Why?

Click to reveal answer

You should use shelving. Shelving applies to individual alarms (not entire Areas), has a built-in timeout that will automatically unshelve the alarm after 30 minutes, and the alarm is still historized while shelved. Silencing would hide all alarms in an Area with no automatic timeout, which is not what you need here.
