1.1 How Application Server Communicates with Field Devices

Reference: Module 5, Section 1 (pages 5-3 to 5-4)

Your Application Server Galaxy objects need to talk to real-world field devices — PLCs, RTUs, DCS controllers — to read and write process data. But Application Server doesn't talk directly to field hardware. Instead, it uses a layered communication architecture with dedicated servers in between.

Core idea: Application Objects never communicate with field devices directly. Instead, they use DDE, SuiteLink, or OPC to talk to an OI Server (Operations Integration Server), which in turn uses a PLC-specific proprietary protocol to talk to the actual field device.

The Communication Chain

The diagram below shows the complete communication path from an Application Object all the way down to the physical PLC:

Communication architecture: Application Object → DDESuiteLinkClient → OI Server → PLC. The OI Server bridges the gap between standard protocols (DDE, SuiteLink, OPC) and the PLC-specific proprietary protocol (page 5-4)

Each layer in the chain has a specific responsibility:

LayerComponentProtocol(s)
Application Object Your automation object in the Galaxy (e.g., a tag, a symbol instance) —
Client Interface DDESuiteLinkClient (built into the object) DDE, SuiteLink, OPC
OI Server Operations Integration Server — a standalone service that manages communication DDE, SuiteLink, OPC (upstream); PLC-specific protocol (downstream)
Field Device PLC, RTU, DCS controller Proprietary protocol (Modbus, Allen-Bradley, Siemens, etc.)
Why a separate OI Server? Application Server focuses on application logic, alarming, and visualization. It doesn't need to know the details of every PLC manufacturer's proprietary protocol. The OI Server handles all the device-specific communication details, so Application Server stays clean, portable, and protocol-independent.

Upstream vs. Downstream Protocols

It's important to understand that the OI Server speaks two different languages on two different sides:

Key point: The protocol between Application Server and the OI Server (DDE, SuiteLink, or OPC) is completely separate from the protocol between the OI Server and the PLC. The OI Server is the bridge — it translates on both sides.
ProtocolTypeNotes
DDEDynamic Data ExchangeLegacy Windows protocol. Simple but limited — no built-in diagnostics or quality indicators.
SuiteLinkAVEVA proprietaryBetter than DDE — provides quality indicators, timestamps, and diagnostics. Recommended for Wonderware/AVEVA ecosystems.
OPCOpen Platform CommunicationsIndustry-standard protocol. Most widely supported across different vendor systems.

1.2 Operations Integration (OI) Servers

Reference: Module 5, Section 1 (pages 5-4 to 5-5)

An Operations Integration (OI) Server is a standalone software service that manages communication between Application Server and field devices. Each OI Server is designed for a specific family of devices (e.g., Modicon/Modbus, Allen-Bradley, Siemens).

OI Server = Communication Manager. It maintains connections to PLCs, polls data at configured intervals, handles protocol translation, and makes data available to Application Server via DDE, SuiteLink, or OPC.

OI Server Hierarchy in OCMC

The Operations Control Management Console (OCMC) shows the hierarchical relationship between your Galaxy, the deployment platform, and the OI Servers configured on it. The hierarchy is read top-down, from the Galaxy down to the specific PLC device:

OCMC showing the server hierarchy: Operations Integration Server Manager → TrainingGalaxy → ProdPlatform → Modicon MBTCP → OI.MBTCP.1 → PORT → PLC (selected) (page 5-5)
LevelExampleMeaning
Galaxy TrainingGalaxy The application being served
Platform ProdPlatform (S03PROD) The physical computer hosting the OI Server
OI Server Type Modicon MBTCP The driver family — Modbus TCP for Modicon PLCs
OI Server Instance OI.MBTCP.1 A specific configured instance of the driver
Configuration PORT → PLC The port and PLC device being managed
Think of it like an address: The OCMC hierarchy is like a postal address — Galaxy → Computer → Driver Type → Driver Instance → Device. Each level narrows down to exactly which device you're communicating with.

1.3 OI Server Configuration — PLC Parameters

Reference: Module 5, Section 1 (pages 5-5 to 5-6)

Each OI Server instance requires configuration to match the physical PLC it communicates with. The most important settings are on the PLC Parameters tab:

OCMC PLC Parameters tab — Node Type=ModbusPLC, IP=1.1.1.1, Port=502, Reply timeout=3s, Swap bytes=checked, Register size=6, Register type=Binary (page 5-6)
ParameterExample ValuePurpose
Node Type ModbusPLC The type of device — determines which protocol dialect to use
IP Address 1.1.1.1 Network address of the PLC
Port 502 Network port (502 is standard for Modbus TCP)
Reply Timeout 3 seconds How long to wait for a PLC response before declaring a timeout
Swap Bytes Checked Whether to swap byte order — depends on PLC endianness
Register Size 6 Size of data registers to read
Register Type Binary How to interpret the register data
Common configuration error: If your data comes back garbled or byte-swapped, check the Swap Bytes setting first. Different PLC manufacturers use different byte orders (big-endian vs. little-endian). Getting this wrong means every numeric value you read will be incorrect.

1.4 Device Groups — Managing Poll Rates

Reference: Module 5, Section 1 (page 5-6)

An OI Server doesn't just blindly poll every register on the PLC. It organizes data requests into Device Groups, each with its own update interval. This lets you control how fast different types of data are refreshed.

OCMC Device Groups tab — configured groups with different update intervals (page 5-6)
Device Groups organize your data items by refresh rate. For example, you might group critical process variables (temperature, pressure) at a fast update rate (500 ms), while grouping status flags at a slower rate (5 seconds). This optimizes network bandwidth and reduces PLC processor load.

Device Items — The Actual Data Points

Within each device group are individual Device Items — the specific PLC addresses (registers, coils) that map to your Application Server tags.

OCMC Device Items — address mapping for PLC data points, with right-click context menu showing options like "Activate Auto start after reboot" (page 5-6)

You can right-click a device item to access additional options such as Activate Auto start after reboot, which ensures the device communication starts automatically when the OI Server machine restarts.

Device Items can be imported from a CSV file instead of being entered one by one. The file dialog below shows importing a pre-built list of PLC addresses — a massive time-saver for large plants with hundreds of tags.
Open file dialog for importing Device Items from a CSV file — PLCItemList_for_SystemPlatform2023.csv (page 5-6)

1.5 OI Server Configuration Files

Reference: Module 5, Section 1 (page 5-6)

All OI Server configuration is stored on disk in the Wonderware/OI-Server directory. The key files are:

Windows Explorer showing OI-Server configuration files: MBTCP.AAcfg (configuration) and MBTCP.AArul (rules), with Copy selected for backup (page 5-6)
File ExtensionPurpose
.AAcfgOI Server configuration — all PLC parameters, device groups, and device items
.AArulOI Server rules — communication rules and filtering logic
Backup your OI Server configuration! These files are the entire configuration of your OI Server. If your OI Server fails or needs to be reinstalled, you can restore these files to get back up and running quickly. Copy them to a safe location as part of your standard deployment procedure.

1.6 Summary — The Complete Communication Picture

Here's how all the pieces fit together when you connect Application Server to a real PLC:

  1. Create an OI Server instance in OCMC for your PLC type (e.g., Modicon MBTCP)
  2. Configure PLC Parameters — IP address, port, timeouts, byte order
  3. Create Device Groups with appropriate update intervals for different data types
  4. Add Device Items — map specific PLC addresses to the groups (or import from CSV)
  5. Configure Application Server tags to point to the OI Server via DDE, SuiteLink, or OPC
  6. Deploy — Application Server communicates with the OI Server, which polls the PLC

Key Takeaways

  1. Application Server does not talk to PLCs directly — it communicates through OI Servers.
  2. The communication chain is: Application Object → DDE/SuiteLink/OPC → OI Server → PLC-specific protocol → PLC.
  3. OI Servers are standalone services, configured in OCMC, organized by Galaxy → Platform → Driver → Instance → Device.
  4. PLC Parameters (IP, port, timeout, byte order) must match the physical PLC configuration.
  5. Device Groups control polling rates — group fast-changing data separately from slow-changing data.
  6. Device Items map specific PLC addresses — they can be imported from CSV for efficiency.
  7. Backup your .AAcfg and .AArul files for disaster recovery.
