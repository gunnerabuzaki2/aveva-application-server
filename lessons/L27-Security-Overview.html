27.1 Security Overview

Pages 9-3 through 9-4

Security in AVEVA Application Server is designed to prevent users from performing unauthorized activities. This includes users of:

  • IDE — for configuring and managing objects
  • Operations Control Management Console (OCMC) — for performing maintenance and system administration functions
  • Runtime operations — any live process control activities
Important Limitation: The security system is not designed to stop malicious access. Passwords are encrypted but stored in an accessible database. For higher-level security, IT teams can integrate with Microsoft tools using OS authentication, password timeout, and electronic signature authentication.

The Security Model

Figure 27-1: The Security Model diagram shows the hierarchical relationships between Development Security (IDE configuration), Runtime Security (operational permissions), Security Classifications, Security Groups, Roles, and Users. Security Classifications are assigned to AutomationObject attributes; Security Groups grant Operational Permissions to Roles; Users belong to one or more Roles. (page 9-3)

The security model has two major domains:

Development Security (General Permissions): Controls access to the IDE for configuring and managing objects in the Development/Testing/Troubleshooting environment.

Runtime Security (Operational Permissions): Controls access to live operational activities. This involves:
  • Security Classifications — assigned to each AutomationObject attribute to define who can write to it
  • Security Groups — group objects that need identical security behavior
  • Roles — generalize user functions (e.g., Intake Operator, Dispatcher)
  • Users — the final component; users can be granted multiple Roles

For example, a DiscreteDevice may have attributes for operation (set by an operator) and tuning (set by a technician). Rather than configuring access for every user on every object individually, the system uses Security Groups, Roles, and Classifications to simplify management.

Security Groups

  • Group objects that need to behave identically with respect to security
  • Every AutomationObject belongs to exactly one Security Group
  • Default: all new objects belong to the Default Security Group (cannot be removed)

Roles

  • Generalize user functions (e.g., Software Engineer, Intake Operator, Dispatcher)
  • Granted permissions onto Security Groups
  • If a Role has Tuning access to a Security Group, that Role has write permissions to all attributes classified as Tuning in that group
  • Also granted utility function permissions (e.g., Deploy, Can Edit)

Users

  • The final component defining system access
  • Users can be granted as many Roles as needed
  • Default users: Administrator and DefaultUser (cannot be deleted)

27.2 Authentication Modes

Pages 9-5 through 9-7

Figure 27-2: The Security Settings window shows the Authentication mode tab with radio buttons for None, Galaxy, OS User based, OS Group based, and Authentication providers. OS Group based is selected, with Login time set to 3000 milliseconds. (page 9-5)

The Authentication Mode page lets you choose the Galaxy security mode:

ModeDescription
NoneDefault for new Galaxies. Open Security — all functions open to all users. No authentication, no login dialogs.
GalaxyUses local Galaxy configuration for authentication. Creates a user security system controlled by the Galaxy database.
OS User basedEnables authorization of individual OS users. Leverages the operating system (NT) user authentication on an individual user basis.
OS Group basedEnables authorization based on OS Group membership. Maps OS groups to Galaxy security roles. Includes Login time and Role update intervals.
Authentication providersUses AVEVA Identity Manager (AIM) with OS security and Azure Active Directory (AD).

OS Group Based — Configurable Intervals

  • Login time (ms): Timeout for validating user membership against OS groups. After timeout, defaults to local cache. Use for intermittent/slow networks. Min: 0 (off), Max: 9,999,999, Default: 0 (disabled).
  • Role update (ms): Time between each validation attempt per OS group. Minimizes network usage. Operates independently of Login time.
Authentication Mode Behaviors:
  • Open security gives all users DefaultUser credentials — no login dialogs
  • Switching modes disables users from the old mode (not deleted, just disabled)
  • When switching to any mode other than None, you must log in after closing the dialog
  • When Galaxy authentication is selected, the IDE customizes the UI to the user's preferences
  • OS User Based: user provides domain, username, and password
  • OS Group Based: user provides domain, username, and password; OS groups mapped to Galaxy roles

User Authentication

Client utilities (IDE, OCMC, InTouch for System Platform) require users to be authenticated. An authenticated user is granted the sum of all Permissions within their assumed Roles.

Operational Permissions

Operational permissions relate to the Security Groups listed on the Security Groups page. By default, the Administrator has all permissions.

PermissionDescription
Can Modify "Operate" AttributesDaily operational tasks: changing setpoints, outputs, control modes
Can Modify "Tune" AttributesTune attributes in live runtime: alarm setpoint sensitivity, PID tuning
Can Modify "Configure" AttributesMajor configuration changes; requires object set to Off scan
Can Acknowledge AlarmsManually acknowledge active alarms in runtime
Can Shelve AlarmsTemporarily suppress or restore alarm notifications
Can Modify Alarm ModesChange operational mode of an alarm
Can Modify Plant StatesModify plant states controlling state-based alarm rules
Can Verify WritesAuthentication signature for Verified Writes security classification
Security Levels Summary:
  • Full Access: All General + All Operational permissions (Administrator role)
  • Limited Access: Selective General + Selective Operational permissions per role
  • No Access: Permissions unchecked — user is denied that function

Quiz: Security Overview + Lab 16

Test Your Knowledge

Q1: What are the four authentication modes available in the Security configuration?

Click to reveal answer

None (Open Security), Galaxy, OS User based, and OS Group based. There is also an Authentication providers option for AIM/Azure AD integration.

Q2: In the Security Model, what is the relationship between Security Groups, Roles, and Users?

Click to reveal answer

Security Groups contain AutomationObjects. Roles are granted Operational Permissions onto Security Groups. Users belong to one or more Roles and receive the sum of all permissions from their assigned Roles.

Q3: What is the difference between "Can Modify Operate Attributes" and "Can Modify Configure Attributes"?

Click to reveal answer

"Can Modify Operate Attributes" allows daily operational tasks like changing setpoints, outputs, and control modes. "Can Modify Configure Attributes" allows major configuration changes and requires the object to be set to Off scan first.

Q4: Why did the Plant Maintenance user receive an "Access Denied" message when trying to open the IDE?

Click to reveal answer

The Plant Maintenance 1 role was not granted "Can Start the IDE" permission (under IDE Permissions in General permissions). This role only had SMC Permissions and Operational permissions (Tune and Configure) for the production lines.

Q5: What happens to all new objects created in a Galaxy with respect to Security Groups?

Click to reveal answer

All new objects belong to the Default Security Group by default. This group cannot be removed. Objects must be manually moved to other security groups as needed.

Q6: What is the Login time setting used for in OS Group based authentication, and why did the lab set it to 3000?

Click to reveal answer

Login time (in milliseconds) is the timeout period for validating user membership against OS groups. After the timeout, the system defaults to the local cache. The lab set it to 3000 ms to allow for a more seamless experience when testing different users, especially in intermittent or slow network scenarios.

Q7: What is the Plant Supervisors 1 role allowed to do that Plant Operators 1 is not?

Click to reveal answer

Plant Supervisors 1 has access to all three operational areas (Line1, Line2, ControlSystem) and has Can Verify Writes permission. Plant Operators 1 only has access to Line1 (Can Acknowledge Alarms + Can Modify Operate Attributes) and no Verify Writes permission.

Q8: After enabling security and assigning new Security Groups, what must happen to deployed objects before they function correctly?

Click to reveal answer

Objects must be redeployed because their SecurityGroup attribute was modified when they were moved between security groups. The IDE will show objects needing deployment, and this will be addressed in Lab 17.

Q9: Can you modify the General permissions for the Administrator role? Why or why not?

Click to reveal answer

No, you cannot modify the General permissions for the Administrator role. The system enforces that the Administrator always has full General permissions to prevent accidental lockout from critical system functions.

Q10: What security limitation should be aware of regarding password storage in Galaxy authentication mode?

Click to reveal answer

Passwords are encrypted but stored in a database that is accessible. The system is not designed to stop determined programmers from accessing it. For higher-level security, use OS-based authentication with Microsoft tools, password timeout, and electronic signature authentication.
