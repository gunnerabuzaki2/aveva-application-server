Lesson 14: Containment + Lab 8 — Modeling the Mixer

Source: AVEVA Application Server 2023 Training Manual, Module 4, Section 4 + Lab 8 (pages 4-45 to 4-66)

How objects contain other objects to form composite equipment — and building a complete Mixer model using containment in Lab 8.

14.1 Containment — Organizing Objects Within Objects

Reference: Module 4, Section 4 (pages 4-45 to 4-47)

In the previous lessons, we modeled individual equipment components — pumps, valves, sensors, and instruments — as separate Galaxy objects. In a real plant, however, these components don't exist in isolation. A mixer physically contains an agitator, inlets, an outlet, level and temperature sensors, and pumps. Containment is the mechanism that lets you model these physical relationships directly in the Galaxy.

Containment is the ability of one Galaxy object to contain other objects within it, creating a parent-child hierarchy. Containment models the physical structure of real equipment — a mixer physically contains pumps, sensors, and inlets, so in the Galaxy, the Mixer object contains those component objects.

Containment is not just organizational grouping. When an object is contained within another object:

  • The contained object's fully qualified name reflects the hierarchy (e.g., Plant.Production.Line1.Mixer100.Pump1_001)
  • The contained object lives inside the parent object in the Galaxy Model tree
  • The parent object can reference and interact with its contained objects via scripts and attribute bindings
  • When the parent is deployed, all contained objects are deployed together as a unit
  • The containment relationship is part of the object's design — it is defined in templates, not at runtime

Containment vs. Other Relationships

RelationshipWhat It DoesWhen to Use
Containment Parent physically contains child — child is part of the parent's design Components that are integral to an equipment piece (sensors built into a mixer, pumps attached to a vessel)
Reference One object references another by attribute path — objects remain separate Objects that need to read/write data from other objects but are not physically part of them
Inheritance Child template derives from parent template — shares structure and behavior Creating reusable equipment templates (all pumps share common attributes)
Think physically: If you would need a wrench to remove component X from equipment Y, then X should be contained by Y in the Galaxy. If X and Y just need to share data, use references instead.

14.2 Containment Hierarchy and Naming

Reference: Module 4, Section 4 (pages 4-48 to 4-50)

When you place an object inside another object, the contained object's fully qualified name (FQN) includes the entire containment path. This naming convention is automatic — you don't build it manually.

The deployment view below shows the full hierarchy for Mixer100. Every component — Agitator, Inlets, Outlet, Pumps, Level, and Temperature — is nested under the Mixer, which itself sits under Line1 within the Production area:

Naming convention: The fully qualified name always starts from the $Plant object and flows down through each level of containment. The dot-separated path mirrors the hierarchy exactly. This makes it easy to locate any object and understand its position in the plant structure.

Controlling Containment

Containment is defined by the Container attribute on any object. When you drag an object onto another in the Model pane, or when you set the Container attribute in the Object Viewer, you establish the containment relationship.

Important: Containment is a design-time decision. Once you set the Container attribute on an object, that object becomes part of the parent. Moving or removing a contained object requires updating the Container attribute — you cannot simply delete the parent without addressing the children.

14.3 Containment Benefits

Reference: Module 4, Section 4 (pages 4-51 to 4-53)

Containment provides several practical benefits for building and managing a Galaxy:

  • Logical organization: The Galaxy Model tree mirrors the physical plant, making it easy to navigate large applications with thousands of objects
  • Deployment efficiency: Deploying a parent object automatically deploys all contained children — no need to deploy each sensor and pump individually
  • Cascade deletion: Deleting a parent object removes all its contained children, preventing orphaned objects
  • Attribute scoping: Scripts within the parent can easily reference contained objects using relative paths (e.g., Me.Pump1_001.Running)
  • Template reuse: Once a Mixer template is built with its contained components, every Mixer instance in the plant inherits the same structure automatically

14.4 Lab 8 — Modeling the Mixer

Reference: Module 4, Lab 8 (pages 4-54 to 4-66)

Lab Objectives:
  • Create a Mixer template with all contained equipment components
  • Configure contained object attributes (I/O references, alarm settings, engineering units)
  • Deploy the complete Mixer model and verify it in the Object Viewer

This lab brings together everything learned so far — template creation, inheritance, attribute configuration, and containment — into a single comprehensive equipment model. The Mixer is the most complex piece of equipment in the training Galaxy, and modeling it properly demonstrates how all these concepts work together.

Step 1: Create the Mixer Template

Reference: Module 4, Lab 8 (pages 4-55 to 4-57)

The Mixer template serves as the parent object that will contain all sub-components. It defines the mixer's own attributes (name, status, mixing speed) as well as the structural framework for containment. Start by examining the available templates in your Galaxy.

Templates pane: Project folder with SMixer (selected), SMeter, SMotor, SValve, SLevel, and STemperature. System templates above.

Step 1a: Expand the SMixer template to see its child objects. The SMixer template already has Level and Temperature as contained child templates — these are the sensor components built into the mixer design.

Templates pane: SMixer expanded with Level and Temperature children. SValve selected.

Step 1b: Examine the Project folder hierarchy. You will see: $Inlet1, $Inlet2, $Mixer (expanded to show Level and Temperature children), $Outlet, and $Valve — all at the top level of the Project folder.

Project hierarchy: $Inlet1, $Inlet2, $Mixer (expanded: Level, Temperature), $Outlet, $Valve
Tip: The Mixer template inherits attributes from its parent template. Make sure you understand which attributes come from inheritance versus which are unique to the Mixer before adding contained objects.

Step 2: Configure Contained Object Attributes

Reference: Module 4, Lab 8 (pages 4-57 to 4-61)

With the Mixer hierarchy in place, the next step is configuring each contained object's unique attributes — I/O connections, Boolean labels, and command/state definitions. Start with the valve and motor attributes.

Step 2a — Configure the Valve's CMD attribute: The Command attribute on $Valve controls whether the valve is open or closed. Set the data type to Boolean, the False label to "Close", the True label to "Open", and the Initial value to "Close". The I/O link writes to <IODevice>.$Valve.CMD.

CMD attribute config: Name=CMD, Description=Valve Command, Boolean, False=Close, True=Open, Initial=Close, I/O Write to <IODevice>.$Valve.CMD

Step 2b — Configure the Motor's PV attribute: The motor process variable (PV) reports the actual running state. Set the data type to Boolean, False label to "Stopped", and True label to "Running". The I/O link reads from <IODevice>.$Motor.PV.

PV attribute config: Name=PV, Description=State, Boolean, False=Stopped, True=Running, I/O Read from <IODevice>.$Motor.PV
Watch out: The CMD attribute is both Read and Write (operator can send commands), while the PV attribute is Read only (it reports actual state). This distinction is critical — you don't want the system accidentally overwriting the motor's real status.

Step 2c — Configure the Motor's CMD attribute: The motor command attribute controls starting and stopping the agitator motor. Set the data type to Boolean, False label to "Stop", True label to "Start". The I/O link reads from and writes to <IODevice>.$Motor.CMD.

CMD config for $Motor: Name=CMD, Description=Command, Boolean, False=Stop, True=Start, I/O Read/Write to <IODevice>.$Motor.CMD

Step 3: Build the Containment Structure

Reference: Module 4, Lab 8 (pages 4-60 to 4-62)

Now create the actual containment relationships. Expand $Mixer in the Project folder to verify all sub-components are properly contained: Inlet1, Inlet2, Level, Outlet, and Temperature.

Project hierarchy with $Mixer expanded: Inlet1, Inlet2, Level, Outlet, Temperature
Verify the hierarchy: After building containment, always check the Galaxy Model tree to confirm the nesting is correct. Every contained object should appear indented under its parent, and the fully qualified names should reflect the expected path.

After checking in the templates, the instantiated objects in the Project folder reflect the updated structure. Verify that each template now appears with its correct child objects.

Templates pane: Project folder with $Agitator, $Meter, $Mixer (expanded), $Motor, $Valve

Step 6: Deploy and Verify

Reference: Module 4, Lab 8 (pages 4-64 to 4-66)

With all attributes configured and containment relationships established, deploy the Mixer and verify that all contained objects come up correctly.

Step 6a: Right-click on Mixer100 in the Galaxy Model and select Deploy. Because Mixer100 contains all the child objects, deploying the parent automatically deploys every contained child — Agitator, Inlets, Outlet, Pumps, Level, and Temperature sensors.

Step 6b: Verify the deployment status shows successful for all objects. If any contained object fails to deploy, check its I/O link configuration and Container attribute.

Step 7: Lab Completion — Module 4 Summary

Reference: Module 4, Lab 8 (page 4-66)

Congratulations — you have completed Lab 8 and Module 4. The Mixer model demonstrates how all the concepts from this module come together:

  • Template creation — the Mixer and all its components are built from reusable templates (SMixer, SValve, SMotor, SLevel, STemperature)
  • Inheritance — each component inherits common attributes and behavior from its base template
  • Containment — all components are contained within the Mixer, reflecting the physical equipment structure
  • Attribute configuration — I/O links, Boolean labels, and command/state definitions are set per component
  • Deployment — deploying the parent deploys the entire composite unit
Module 4 Complete: You now understand how to create equipment templates, use inheritance to specialize them, configure attributes with I/O links and alarms, and use containment to model composite equipment. These are the fundamental building blocks of any Application Server project.

Q1: What is containment in Application Server, and how does it differ from a reference relationship?

Click to reveal answer

Containment means one object physically contains another — the child is part of the parent's design and is deployed/deleted with it. A reference is a data link between two independent objects that remain separate in the Galaxy hierarchy.

Q2: If Mixer100 contains Pump1_001, what is Pump1_001's fully qualified name?

Plant.Production.Line1.Mixer100.Pump1_001

Q3: What happens when you deploy a parent object that contains children?

Click to reveal answer

All contained child objects are deployed automatically as part of the deployment — you do not need to deploy each child individually.

Q4: What attribute on an object defines its containment relationship?

Click to reveal answer

The Container attribute — it specifies the fully qualified name of the parent object that contains this object.

Key Takeaways

  1. Containment models physical relationships — parent objects contain child objects as integral components.
  2. The Container attribute defines the containment relationship for each object.
  3. Fully qualified names reflect the containment hierarchy (e.g., Plant.Production.Line1.Mixer100.Pump1_001).
  4. Deploying a parent automatically deploys all contained children.
  5. The Mixer model demonstrates all Module 4 concepts together: templates, inheritance, containment, and attribute configuration.
  6. Use containment for integral components; use references for data links between independent objects.
