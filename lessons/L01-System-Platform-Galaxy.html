Lesson 1: What is Application Server? The Galaxy

Source: AVEVA Application Server 2023 Training Manual, Module 1, Sections 2-3 (pages 1-9 to 1-20)

The foundation — what Application Server is, what a Galaxy is, and how they fit into System Platform.

1.1 AVEVA System Platform — The Big Picture

Reference: Module 1, Section 2 (pages 1-9 to 1-12)

Before diving into Application Server, you need to understand what System Platform is, because Application Server is just one part of it.

AVEVA System Platform is the complete industrial automation software suite — formerly called Wonderware System Platform. It is built on ArchestrA technology and provides everything you need to run a plant: data acquisition, alarming, historization, visualization, and scripting.

Think of System Platform as a layered system with three groups:

System Platform layered architecture — Supervisory Clients on top, System Platform Components in the middle, Field Devices at the bottom (Module 1, page 1-9)
LayerComponentsWhat They Do
Supervisory Clients OMI (Operations Management Interface), InTouch for System Platform, Historian Client The screens the operator sees — HMI visualization, trending, data analysis
System Platform Components Application Server, Historian, Communication Drivers The engine room — handles data, logic, alarms, historization
External Data Sources Controllers (PLCs, RTUs), Software (ERP, MES), Data Sources Real field devices providing process data

Core Services Provided by System Platform

  • Real-time data acquisition from field devices (PLCs, RTUs)
  • Scaling, statistics, and data manipulation
  • Process control
  • Alarm management — triggering, logging, acknowledgment
  • Historization, trending, and analysis of process data
  • Visualization of real-time process data
  • Reporting
Key distinction: System Platform is NOT a single product — it's a collection of products working together. Application Server is the heart; OMI/InTouch are the faces; Historian is the memory; Communication Drivers are the nerves.

ArchestrA Technology — The Foundation

ArchestrA is the distributed, object-oriented architecture that all System Platform products are built on. It provides:

  • Object management for creating object-oriented applications
  • A component object framework for modeling plants, factories, and equipment
  • A global namespace — every object has a unique path, no matter which computer it runs on
  • Inter-process communication with diagnostics
  • Centralized security for multi-user environments
  • Version management — every object tracks its own configuration history

When you work with Application Server, you are using ArchestrA — even if you never hear the term again. Every object, every deployment, every script runs on ArchestrA.

1.2 Application Server — The Heart of System Platform

Reference: Module 1, Section 3 (pages 1-13 to 1-16)

Application Server is the engine that creates, manages, and deploys your automation application. An application created with Application Server is called a Galaxy. The two terms are interchangeable.

Based on an object-oriented framework, Application Server lets you model your entire plant as objects. From the area level, to equipment (valves, tanks, pumps), to the computers running your application — everything is an object in a Galaxy.

Key Features of Application Server

1.3 The Three Components of Application Server

Reference: Module 1, Section 3 (page 1-14)

Application Server is NOT one piece of software. It is three components working together:

ComponentTypeWhat It Does
Bootstrap Windows Service The software foundation on every computer that will be part of the Galaxy. Without Bootstrap, a computer cannot receive a platform.
IDE (Integrated Development Environment) Application The tool where you design, configure, and deploy your Galaxy. You build everything here.
Galaxy Repository (GR) Windows Service The central server that holds the Galaxy's configuration database and manages development + deployment. All IDE commands actually execute here.
Common misunderstanding: When you click "deploy" in the IDE, the IDE does NOT deploy from your computer. The IDE sends a command to the Galaxy Repository, and the GR does the actual deployment from its computer to the target platform. The IDE is just a remote control.

1.4 What is a Galaxy?

Reference: Module 1, Section 3 (pages 1-14 to 1-15)

A Galaxy is the entire application ecosystem. It is NOT just a database — it is:

  • A collection of objects — from computers and engines, to every valve and tank in the field
  • A project database — holds all configuration information
  • One or more networked computers — running the application at runtime
  • A common set of policies — security, alarm, and communication settings shared across all objects
Critical rule: A Galaxy Repository can host multiple Galaxies, but only ONE Galaxy can be deployed and running at any given time on a GR node.

The WinPlatform Object

The $WinPlatform object represents a physical computer in the Galaxy. When you deploy a $WinPlatform, the Galaxy "owns" that computer. A computer can belong to one and only one Galaxy at a time.

Multi-Galaxy Communication

Multiple Galaxies on the same network are isolated by default. They cannot see each other's data. However, they can be configured for multi-Galaxy communication if you need two Galaxies to share data at runtime.

1.5 Galaxy Templates — Starting Points

Reference: Module 1, Section 3 (page 1-15)

When you create a new Galaxy, you choose a template — a starting point that pre-populates the Galaxy with certain objects and settings.

Galaxy templates — the starting points for new Galaxies (Module 1, page 1-15)
TemplateContentsBest For
Default Galaxy Default screen profiles, templates, basic equipment model, OMI ViewApp example Recommended starting point for most projects
Blank Galaxy Only base template objects — nothing else When you want full control from scratch
InTouch Base Objects and graphics needed for tag-based Managed InTouch applications Legacy InTouch-focused projects
Reactor Demo Pre-built demo Galaxy Learning and evaluation
You can create your own Galaxy templates by backing up an existing Galaxy. This is how organizations distribute their standards — pre-configured objects, graphics, and system settings packaged as a Galaxy backup.

1.6 Lab 1 — Creating Your First Galaxy

Reference: Module 1, Lab 1 (pages 1-17 to 1-20)

Lab Objectives:
  • Create a Galaxy
  • Connect to it using the AVEVA System Platform IDE

Step-by-Step

Step 1: Open the AVEVA System Platform IDE. The Backstage area appears — this is the starting screen where you choose which Galaxy to work with.

Step 2: Click Blank Galaxy to create a new Galaxy from scratch. The Create Galaxy dialog appears.

Steps 3-5: In the dialog, enter the following:

FieldValue
Galaxy nameTrainingGalaxy
Template nameDefault_EMPTY.cab (should already be selected)
Connect to this Galaxy☑ Checked

Step 6: Click Create.

Steps 1-6: The IDE Backstage area with Blank Galaxy selected (top), and the Create Galaxy dialog with TrainingGalaxy entered (bottom) (Module 1, page 1-18)
The Create Galaxy progress dialog — shows the Galaxy being built (top), then "Successfully created TrainingGalaxy" when done (bottom) (Module 1, page 1-19)

Step 7: Wait for the progress to complete. Verify there are no error messages.

Step 8: Click Close. The IDE opens with your new Galaxy loaded.

The IDE opens with TrainingGalaxy loaded — Templates and Model panes show the Galaxy hierarchy (Module 1, page 1-20)
What just happened? The IDE told the Galaxy Repository to create a new Galaxy using the Default_EMPTY template. The GR created the database, restored the default objects, and deployed the baseline services. Your IDE then connected to this new Galaxy.

1.7 Encrypted Communication & System Requirements

Reference: Module 1, Sections 6-7 (pages 1-53 to 1-59)

AVEVA Application Server supports end-to-end encrypted communication between server and client software. Key requirements:

  • Operating System: Windows Server or Windows 10/11 (64-bit)
  • Database: SQL Server (Express or full) for the Galaxy Repository
  • .NET Framework: 4.7.2 or later
  • Network: TCP/IP connectivity between all nodes

Licensing is per-node — each computer running Application Server components needs a valid license.

Key Takeaways

  1. System Platform is the complete suite; Application Server is its core engine.
  2. Application Server has three components: Bootstrap, IDE, and Galaxy Repository.
  3. A Galaxy is the entire application — objects, database, computers, and policies.
  4. The Galaxy Repository does all the real work; the IDE is just a remote control.
  5. $WinPlatform claims a computer for one Galaxy exclusively.
  6. Only one Galaxy can be deployed at a time per GR node.
