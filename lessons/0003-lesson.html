Lesson 3: Automation Objects — Types, Templates, and the Object Editor

Based on Module 1, Section 5: Automation Objects (pages 1-31 to 1-42)

In the AVEVA Application Server, everything in the system is an object. From the simplest I/O tag to the most complex alarm engine, the automation object is the fundamental building block of the entire platform. Understanding how these objects are structured, how they relate to one another, and how you configure them is essential before building any application.

1. Three Types of Automation Objects

Source: page 1-31

All automation objects in AVEVA Application Server belong to one of three categories. These categories are clearly visible in the Templates pane of the IDE:

CategoryPurposeExamples
System Platform, engine, and model infrastructure objects — the backbone of the runtime $Area, $WinPlatform, $AppEngine, $ViewEngine
Device Integration (DI) Objects that connect to external data sources and PLCs $DDESuiteLinkClient, $OPCClient, $RedundantDIObject
Application Objects that model process equipment and logic — the objects you build your system with $Valve, $AnalogDevice, $DiscreteDevice, $UserDefined
The Templates pane in the IDE showing the three categories: System, Device Integration, and Application objects.
Key Concept: The category determines where you can place an object in the hierarchy and what role it plays. System objects build the platform structure, DI objects provide data connectivity, and Application objects model the process itself.

2. Object Encapsulation — The 7 Pillars

Source: page 1-32

Every automation object — regardless of type — is an encapsulated entity. This means that all the functionality needed to represent a real-world device or piece of logic lives inside the object itself. No external scripting, no separate databases, no disconnected alarm configurations.

A single automation object encapsulates seven functional areas:

#FunctionDescription
1I/OInput and output connections — how the object reads from and writes to the process or data sources
2ScriptingInternal logic and calculations — scripts that execute on scan cycles or on events
3HistoryData logging and historical recording configuration
4SecurityAccess control — who can view, modify, or operate this object
5AlarmsAlarm generation, classification, and notification rules
6Version / DocumentationBuilt-in versioning, check-out/check-in tracking, and documentation fields
7GraphicsLinkage to graphical symbols for display in HMI screens
Encapsulation diagram: every automation object — like this valve — contains I/O, Scripting, History, Security, Alarms, Version/Documentation, and Graphics all within it.
Key Concept: Encapsulation is what makes the Application Server "object-oriented" in practice. When you move an object from development to deployment, everything moves with it — no orphaned scripts, no broken alarm links, no missing graphics.

3. Configuring Objects — Attributes, the Object Editor, and the Template-Instance Model

Source: page 1-33

Every automation object is configured through its attributes. Attributes are the named properties that define an object's behavior — things like scan rate, alarm limits, engineering units, and I/O addresses.

Attribute Types

  • Built-in attributes: Defined by the base template. Every $Valve has a PV, a Mode, and a Command attribute, because those are part of the template definition.
  • Custom attributes: Added by the developer to store application-specific data — for example, adding a Location attribute to track where a valve is installed.

The Template-Instance Model

The Application Server uses a template-instance architecture. You do not create objects from scratch. Instead:

  1. You start with a base template (e.g., $Valve)
  2. You may create derived templates to customize behavior for your plant
  3. You create instances from the template (e.g., CV101, CV102, CV103)
  4. Each instance inherits all attributes from its template, but you can override values per-instance
The template-instance model: $Valve is the template, and CV101, CV102, CV103 are instances. Each instance inherits the template's attributes but can have its own values.
Tip: Think of the template as a cookie cutter. The template defines the shape (attributes, scripts, alarms), and each instance is a cookie cut from that shape. Change the cutter, and you can propagate changes to all cookies — but only if you change the template, not the instances.

The Object Editor is the primary interface for configuring an object's attributes. You access it by double-clicking any object in the Object Viewer or by right-clicking and choosing Edit. We will explore it in detail in Section 8 below.

4. Application Objects — The Building Blocks

Source: page 1-34

Application objects are the templates you use to model real-world process equipment and logic. The Application Server ships with several pre-built application templates:

TemplateDescription
$AnalogDevice Represents an analog (continuous) point. Has 2 modes of operation: I/O connected (live data from a field device) and Calculated (computed from other attributes via script).
$DiscreteDevice Represents a discrete (on/off) point. Supports 5 states: e.g., Open, Closed, Opening, Closing, and Fault — or any custom state labels.
$Sequencer Implements sequential logic — step-by-step operations such as batch sequences or equipment startup procedures.
$SQLData Connects to SQL databases to read and write data — useful for integrating with MES, LIMS, or ERP systems.
$UserDefined A blank canvas — a template with no pre-built behavior. You define everything from scratch. Ideal for custom logic or unique equipment types.
The pre-built application object templates available in the IDE's Templates pane.
Key Concept: These pre-built templates are starting points. In practice, you almost always create a derived template from one of these base templates, customize it for your specific needs, and then create instances from that derived template. This is covered in detail in Lesson 4.
Beyond the built-ins: AVEVA's Object Toolkit is a separate development tool that lets you create entirely new base templates with custom attributes, methods, and behaviors. Objects created with Object Toolkit appear alongside the standard templates and follow the same template-instance rules.

5. Device Integration Objects — Connecting to the Outside World

Source: page 1-35

Device Integration (DI) objects provide the bridge between the Application Server and external data sources. They live in the Device Integration category of the Templates pane:

TemplatePurpose
$DDESuiteLinkClient Connects to devices using DDE (Dynamic Data Exchange) or SuiteLink protocols — commonly used with InTouch HMI tags and Wonderware-compatible PLCs.
$OPCClient Connects to OPC servers (OPC DA or OPC UA) to read/write data from PLCs, RTUs, and other industrial controllers.
$RedundantDIObject A wrapper that provides automatic failover between two DI connections — ensuring data continuity if the primary path fails.
$InTouchProxy Acts as a proxy for InTouch tag data — allowing the Application Server to access InTouch window variables and tags.
Device Integration objects ($DDESuiteLinkClient, $OPCClient, etc.) and System objects ($Area, $WinPlatform, etc.) in the Templates pane.
Key Concept: DI objects are not placed directly in the Plant Model hierarchy. Instead, they are children of a $WinPlatform object (part of the Deployment Model) and are referenced by Application objects that need their data connectivity.

6. System Objects — The Infrastructure Layer

Source: page 1-35 (continued)

System objects form the infrastructure that the Application Server needs to run. They define the Plant Model, the Deployment Model, and the runtime engines.

TemplateRoleKey Details
$Area Plant Model element + alarm distribution Represents a physical or logical area of the plant. Also serves as the node through which alarms are distributed to downstream consumers (e.g., displays, alarm printers).
$WinPlatform Deployment Model element Represents a deployment target (a Windows machine or server). Application objects are deployed to a platform. DI objects live under the platform.
$AppEngine Runtime scan engine Executes object scripts and handles I/O scanning. Configurable scan period from 10 ms to 1 hour. Multiple engines can run on a single platform for workload distribution.
$ViewApp Runtime HMI application Represents a running HMI application instance — the container for graphics, displays, and operator interaction.
$ViewEngine Graphics rendering engine The engine that renders graphics and manages display performance.
$InTouchViewApp InTouch compatibility layer Provides backward compatibility with legacy InTouch applications and window scripts.
Tip: $Area is unique in that it serves a dual purpose — it organizes your Plant Model hierarchy and it manages alarm distribution. If you need alarms to reach an operator display, that display's area must be a child (or descendant) of the area where the alarm is generated.
Pitfall: Forgetting that $AppEngine scan period affects both I/O polling and script execution. Setting it too slow (e.g., 1 hour) means your scripts only run once an hour. Setting it too fast (e.g., 10 ms) on a large engine can overload the CPU. Choose wisely based on your actual control requirements.

7. Templates vs. Instances — The Inheritance Hierarchy

Source: pages 1-36 to 1-38

Understanding the distinction between templates and instances — and between the different kinds of templates — is one of the most important concepts in the Application Server.

Three Levels of Objects

LevelDescriptionEditabilityCreated By
Base Template The original template shipped with the system (e.g., $Valve, $AnalogDevice) Read-only — cannot be modified in the IDE AVEVA / Object Toolkit
Derived Template A customized version of a base template, created in the IDE for your specific plant Writeable — you add/modify attributes, scripts, alarms You, in the IDE
Instance A runtime object created from a derived template — represents actual equipment Attribute values can be overridden; structure comes from template You, in the IDE (deployed to runtime)

The Derivation Hierarchy

Templates can be derived from other templates, forming a hierarchy. Each level inherits all attributes from its parent and can add new ones or override defaults. The hierarchy might look like this:

$Valve              ← Base Template (read-only, built into the system)
  └─ $GateValve     ← Derived Template (your customized version)
       └─ V101      ← Instance (actual valve in the field)
       └─ V102      ← Instance (another valve)
       └─ V103      ← Instance (another valve)
The derivation hierarchy: base templates at the top, derived templates in the middle, and runtime instances at the bottom.

Inheritance Rules

  • Instances inherit everything from their parent template — all attributes, scripts, alarms, graphics links, and security settings.
  • Instances can override attribute values — but not the attribute structure itself. You can change the scan rate of CV101 but you cannot delete the PV attribute from CV101.
  • Derived templates can add new attributes — these then propagate down to all instances of that derived template.
  • Changes to a derived template propagate to its instances — but only if the attribute value has not been overridden at the instance level. This is the concept of attribute inheritance.
  • Base templates cannot be modified — they are locked by the system. Always derive before customizing.
Pitfall: Do not confuse "overriding an attribute value at the instance" with "changing the attribute at the template level." If you override a value at the instance, the template change will not affect that instance for that specific attribute. This is by design — it protects your per-instance customizations.

8. The Object Editor — Your Configuration Workbench

Source: page 1-39

The Object Editor is the primary interface for configuring any automation object. You open it by double-clicking an object in the Object Viewer, or by right-clicking and selecting Edit. It has three main tabs:

Attributes Tab

  • Lists all attributes of the object — both built-in (inherited) and custom (added locally)
  • Each attribute shows its name, type, value, and source (inherited vs. overridden)
  • Right-click to add new custom attributes, edit existing values, or set I/O connections
  • Inherited attributes appear in a different color or with a special indicator so you can tell them apart from locally-defined attributes

Scripts Tab

  • Contains embedded scripts that execute based on the object's scan cycle or on events
  • Scripts can read and modify attribute values, communicate with other objects, and perform calculations
  • The scripting language is proprietary to AVEVA but similar in structure to VBScript

Object Information Tab

  • Shows metadata: object name, type, template path, creation date, description
  • Contains the check-out/check-in controls for version management
  • Displays documentation fields where you can record design notes and purpose
The Object Editor with its three tabs: Attributes (configuration), Scripts (logic), and Object Information (metadata and check-out).

Check-Out / Check-In

The Application Server includes a built-in version control system. Before you can modify an object's attributes or scripts, you must check it out. This locks the object so that no other developer can edit it simultaneously. When you are done, you check it in, which saves your changes and releases the lock.

Key Concept: Check-out/check-in is the Application Server's answer to multi-developer environments. If two engineers try to edit the same object simultaneously, the second one is blocked until the first checks it back in. This prevents conflicting changes from corrupting your configuration.

9. Object States — Reading the Status Indicators

Source: pages 1-41 to 1-42

The Object Viewer uses a system of icons and indicators next to each object to communicate its current state. Learning to read these indicators is essential for troubleshooting and for understanding what has been deployed vs. what is still in development.

Deployment Indicators

IndicatorMeaning
🟢 Green circleObject is deployed and running normally
⬛ Yellow squareObject is deployed but not currently running (deployed to platform but engine not started)
⬆️⬇️ Arrows (up/down)Object is being deployed or undeployed (transition state)

Configuration Indicators

IndicatorMeaning
❗ Blue exclamation markObject has been modified but the changes have not yet been deployed — the running version differs from the configured version
❌ Red XObject has a configuration error — something is preventing it from being deployed (e.g., missing required attribute, broken I/O connection)

Check-Out Indicators

IndicatorMeaning
🔴 Red check markObject is checked out by you (current developer) — you have exclusive edit access
⬛ Black check markObject is checked out by another developer — you cannot modify it until they check it back in
The Object Viewer showing various deployment, configuration, and check-out status indicators next to each object.
Tip: When you see the blue exclamation mark, it means your local changes have not been pushed to the runtime. Always check for this indicator before a plant startup — you may have configured everything but forgotten to deploy.
Pitfall: A red X means the object cannot be deployed. Common causes include: referencing a non-existent template, having circular dependencies, or missing required attributes in a derived template. The Object Editor's Object Information tab often provides more details on the error.

10. Object Wizards — Simplified Instance Configuration

Source: page 1-42

Creating instances from complex template hierarchies can involve many steps. Object Wizards provide a simplified, guided interface for this process.

  • A wizard presents a focused, step-by-step dialog that asks only the questions relevant to creating a specific type of instance
  • Instead of navigating through a deep template hierarchy and configuring every attribute, the wizard reduces the apparent hierarchy depth — making it feel like you are creating an object directly
  • Under the hood, the wizard still creates a derived template or instance from the existing hierarchy, but it handles the intermediate steps automatically
  • Wizards are particularly useful for common object types that follow a well-defined pattern (e.g., creating a standard valve with predefined alarm limits and graphics)
Key Concept: Object Wizards are an accessibility feature — they do not add new functionality. They simply make it faster and less error-prone to create instances from templates that might otherwise require many clicks and configuration decisions.
Lab Preview: In the next lesson (Lesson 4), you will create derived templates and instances by hand, without wizards. This gives you a deeper understanding of how the template hierarchy actually works. Once you understand the fundamentals, you can use wizards with confidence — knowing exactly what they are doing behind the scenes.
