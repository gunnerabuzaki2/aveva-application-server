Lesson 18: Lab 10 — Configuring the DI Object

Source: AVEVA Application Server 2023 Training Manual, Module 5, Lab 10 (pages 5-19 to 5-26)

Understanding and configuring the Device Integration Object — the bridge between your Galaxy and field devices.

18.1 What is a DI Object?

Reference: Module 5, Lab 10 (pages 5-19 to 5-20)

When you connected to an OI Server in Lab 9, Application Server automatically created a special object in your Galaxy to represent that connection. This is the Device Integration (DI) Object.

DI Object (Device Integration Object) is a special system object within the Galaxy that represents a connection to an OI Server. It serves as the bridge between your automation application and the real-world field devices (PLCs, RTUs, sensors). The DI Object is not something you create from scratch — it is automatically generated when you connect the Galaxy to an OI Server.

In your TrainingGalaxy, the DI Object is called ProdPLC. It represents the connection to your Production PLC through the OI Server. Every attribute in your Galaxy that is bound to a PLC tag flows through this DI Object.

Key Characteristics of DI Objects

CharacteristicDescription
Auto-created Generated automatically when the Galaxy connects to an OI Server — you do not create it manually
System-managed Contains system-level attributes that monitor and control the OI Server connection
Hierarchical Organized under the Galaxy in the IO Devices view, with topics and tags nested below it
Deployable Must be deployed to the runtime engine before any I/O binding will work
Where to find it: The DI Object appears in the IO Devices view under your Galaxy name. In the Model view, it may also appear as a deployment node. You can also find it in the deployment tree by right-clicking objects and looking at their deployment targets.

18.2 The Deployment View and Context Menu

Reference: Module 5, Lab 10 (pages 5-19 to 5-20)

Understanding the deployment context menu is essential for managing your DI Object and the objects that depend on it. When you right-click an object in the deployment tree, you get a context menu with key operations:

Right-clicking ProdPLC in the deployment view reveals the context menu: Deploy, Undeploy, Assign To, Export, and Properties (Module 5, page 5-19)

Context Menu Options Explained

OptionWhat It Does
Deploy Pushes the object to the runtime engine. The object begins executing, communicating with the OI Server, and processing data.
Undeploy Removes the object from the runtime engine. It returns to development-only status and stops all data communication.
Assign To Changes which deployment node (e.g., which $WinPlatform or AppEngine) the object runs on. Useful for restructuring your deployment topology.
Export Exports the object and its configuration for use in another Galaxy or for backup purposes.
Properties Opens the object's properties dialog where you can configure deployment-specific settings such as which nodes to deploy to and deployment priorities.
Deployment Chain: When you deploy Mixer100, it must be deployed to a deployment node (such as an AppEngine running on a $WinPlatform). That deployment node must itself be deployed. And if Mixer100 has I/O bindings, the DI Object (ProdPLC) must also be deployed. Application Server tracks these dependencies and will warn you if required objects are not deployed.
⚠ Pitfall — Missing Dependencies: If you deploy an automation object that has I/O bindings but the DI Object is not deployed, the attribute will show Quality = Bad or Quality = Uncertain. Always ensure the DI Object and its OI Server are deployed and connected before binding attributes to I/O.

18.3 DI Object Attributes — ConnectionStatus

Reference: Module 5, Lab 10 (pages 5-20 to 5-22)

The DI Object contains several critical system attributes that let you monitor and control the connection to the OI Server. Let's examine the ProdPLC DI Object's key attributes:

The ProdPLC DI Object with its key attributes visible: ConnectionStatus = Connected, Reconnect = false, ScanState = true, ScanStateCmd = true (Module 5, page 5-20)

Key Attributes of the DI Object

AttributeValue in ScreenshotWhat It Means
ConnectionStatus Connected Indicates the current state of the connection to the OI Server. Common values: Connected, Disconnected, Connecting, Faulted. When Connected, the OI Server is actively communicating with the field devices.
Reconnect false Controls automatic reconnection behavior. When false, Application Server will NOT automatically attempt to reconnect if the OI Server connection drops. When true, it will automatically retry the connection. In production, you typically want this set to true.
ScanState true Controls whether the OI Server actively polls the field devices. When true, data is being collected at the configured scan rate. When false, polling stops — useful for maintenance or troubleshooting without disconnecting entirely.
Topic1.$Sys$Status true A system-level attribute specific to Topic1. It indicates whether the topic is healthy and communicating. true means the topic is operational and data is flowing.
Understanding $Sys$ Attributes: Attributes prefixed with $Sys$ are system-level attributes that Application Server uses internally for monitoring. They are read-only and provide real-time diagnostic information about the communication health. You cannot write to them — they are populated automatically by the runtime engine.

ConnectionStatus Values in Detail

StatusMeaningAction Required
Connected Normal operation — OI Server is communicating with field devices None — system is healthy
Disconnected Connection lost — OI Server is not reachable Check network, OI Server service, and deployment status
Connecting Attempting to establish connection Wait — connection attempt in progress
Faulted Connection failed due to configuration error Review OI Server configuration and topic settings

18.4 Monitoring the DI Object in Real Time

Reference: Module 5, Lab 10 (pages 5-21 to 5-23)

Once your DI Object is deployed and connected, you can monitor its attributes in real time. This is essential for verifying that communication is healthy and for diagnosing issues.

Monitoring the ProdPLC DI Object: the same attributes now show timestamps and Quality values. Quality = C0:Good indicates that all data is fresh and reliable (Module 5, page 5-21)

Understanding Timestamps and Quality

When you monitor DI Object attributes in the Object Viewer or the Galaxy IDE, each attribute displays three pieces of information:

FieldDescription
Value The current attribute value (e.g., Connected, true)
Timestamp When the value was last updated — critical for determining if data is fresh or stale
Quality A code indicating data reliability. C0:Good means the value is fresh and trustworthy.

Quality Codes Explained

Quality CodeMeaningTypical Cause
C0:Good Data is fresh and reliable Normal operation — OI Server is polling and returning valid data
00:Bad Data is unreliable or unavailable OI Server disconnected, PLC offline, or communication timeout
40:Uncertain Data may or may not be valid Intermittent communication issues or the last known value may be stale
80:Good, Local Override Value has been manually overridden locally Someone used the Modify Value dialog to force a value
Tip — Continuous Monitoring: Leave the Object Viewer open while you work. If you see the Quality change from C0:Good to 00:Bad, you'll know immediately that a communication issue has occurred. This is far better than discovering the problem hours later when an operator reports that the HMI is showing stale data.

18.5 Configuring DI Object Properties

Reference: Module 5, Lab 10 (pages 5-22 to 5-24)

While most DI Object attributes are system-managed, there are several properties you can and should configure for your specific application needs:

Scan State Configuration

The ScanState attribute controls whether the OI Server polls the field devices. This is one of the most useful operational tools:

  • ScanState = true — Normal operation. The OI Server polls all configured tags at their defined rates.
  • ScanState = false — Pauses all polling. Useful during maintenance windows, equipment testing, or when you need to freeze data for troubleshooting.
ScanState vs. Disconnect: Setting ScanState to false pauses polling but keeps the OI Server connection alive. Disconnecting the OI Server tears down the entire communication link. Use ScanState for temporary pauses; use disconnect for major maintenance.

Reconnect Configuration

The Reconnect attribute determines automatic recovery behavior:

Reconnect ValueBehaviorWhen to Use
true Application Server automatically retries the connection if it drops Production environments — you want automatic recovery without operator intervention
false No automatic retry — you must manually re-enable the connection Development/testing environments, or when you want controlled reconnection

Topic-Level Configuration

Each topic under the DI Object also has system attributes:

AttributePurpose
Topic1.$Sys$Status Indicates if the topic is healthy (true) or faulted (false)
Topic1.$Sys$LastError Contains the last error message if the topic encountered a problem
Topic1.$Sys$ScanRate The configured polling rate for this topic (in milliseconds)
⚠ Common Issue — Topic Not Ready: After deploying, if Topic1.$Sys$Status shows false, check: (1) Is the OI Server running? (2) Is the topic configured in the OI Server? (3) Is the network path to the PLC valid? (4) Is the DI Object deployed? The $Sys$ attributes are your first line of diagnostic defense.

18.6 Lab 10: Configuring the DI Object — Step by Step

Reference: Module 5, Lab 10 (pages 5-19 to 5-26)

Lab 10 — Configuring the DI Object

Objective: Configure and verify the DI Object (ProdPLC) to ensure reliable communication between your Galaxy and the field devices through the OI Server.

Prerequisites: Lab 9 completed — OI Server is running, Galaxy is connected, and the ProdPLC DI Object exists in your Galaxy.

Step 1: Locate the DI Object

  1. Open the Galaxy IDE with TrainingGalaxy loaded.
  2. In the IO Devices view, expand your Galaxy to see the DI Object hierarchy.
  3. Locate ProdPLC — this is your DI Object representing the OI Server connection.
  4. Verify that Topic1 is listed under ProdPLC and that tags (Agitator_001, Inlet1_001, Level1, etc.) are visible.

Step 2: Deploy the DI Object

  1. In the Deployment or Model view, locate ProdPLC.
  2. Right-click ProdPLC and select Deploy.
  3. Review the deployment wizard — confirm the target deployment node(s).
  4. Complete the deployment. Wait for the deployment to finish successfully.
Tip: If you deployed the entire Galaxy in a previous lab, the DI Object may already be deployed. Check its status — deployed objects typically show a specific icon or status indicator in the deployment tree.

Step 3: Verify Connection Status

  1. After deployment, locate the ConnectionStatus attribute on the ProdPLC DI Object.
  2. Verify it shows Connected. If it shows Disconnected or Faulted, troubleshoot the OI Server connection.
  3. Check ScanState — it should be true, indicating active polling.
  4. Check Reconnect — note its current value. For production, you would typically set this to true.

Step 4: Check Topic Health

  1. Expand the ProdPLC DI Object to see Topic1.
  2. Check Topic1.$Sys$Status — it should be true, indicating the topic is healthy.
  3. If the status is false, check the OI Server configuration and verify the PLC is accessible.

Step 5: Monitor with Timestamps and Quality

  1. Open the Object Viewer (press F12 or use the toolbar).
  2. Navigate to the ProdPLC DI Object.
  3. Observe the attribute values along with their timestamps and Quality indicators.
  4. Verify that Quality shows C0:Good for all attributes — this confirms data is fresh and reliable.
  5. Note the timestamps — they should be recent (within the last few seconds), confirming active communication.

Step 6: Test ScanState Control

  1. In the Object Viewer, locate the ScanState attribute.
  2. Double-click the value and change it from true to false.
  3. Wait a few seconds and observe: the tag values should stop updating (timestamps freeze).
  4. Set ScanState back to true. Verify that tag values resume updating.
⚠ Caution: Only modify ScanState in a training/development environment. In a production system, setting ScanState to false will stop all data collection from that OI Server, potentially causing operators to see stale data on their screens.

Step 7: Test Reconnect Behavior

  1. Note the current value of the Reconnect attribute.
  2. Set Reconnect = false.
  3. If possible, temporarily disconnect the OI Server (stop the OI Server service or disconnect the network).
  4. Observe that ConnectionStatus changes to Disconnected.
  5. Restart the OI Server / restore the network connection.
  6. With Reconnect = false, observe that the connection does NOT automatically recover — ConnectionStatus remains Disconnected.
  7. Now set Reconnect = true. The connection should automatically re-establish, and ConnectionStatus should return to Connected.
  8. For the remainder of the labs, leave Reconnect = true (the recommended production setting).

18.7 Key Terms

TermDefinition
DI Object Device Integration Object — a system object in the Galaxy that represents a connection to an OI Server
ConnectionStatus A system attribute indicating the current state of the OI Server connection (Connected, Disconnected, Connecting, Faulted)
ScanState A controllable attribute that starts (true) or pauses (false) OI Server polling of field devices
Reconnect A controllable attribute that enables (true) or disables (false) automatic reconnection after a connection loss
$Sys$ Attributes System-level read-only attributes prefixed with $Sys$ that provide real-time diagnostic information
Quality Code A code indicating data reliability: C0:Good (reliable), 00:Bad (unreliable), 40:Uncertain (may be stale)
Topic A communication channel within an OI Server configured to talk to a specific PLC or device
Deployment Chain The dependency chain: automation objects depend on deployment nodes, which depend on $WinPlatforms, which depend on Bootstrap services

18.8 Quiz

Question 1

What is a DI Object and how is it created?

Click to reveal answer

A DI Object (Device Integration Object) is a special system object in the Galaxy that represents a connection to an OI Server. It is automatically created when the Galaxy connects to an OI Server — you do not create it manually. In your TrainingGalaxy, the DI Object is called ProdPLC.

Question 2

What does the ConnectionStatus attribute indicate, and what value do you want to see during normal operation?

Click to reveal answer

The ConnectionStatus attribute indicates the current state of the connection between Application Server and the OI Server. During normal operation, you want to see Connected. Other values include Disconnected, Connecting, and Faulted — all of which indicate a problem that needs attention.

Question 3

What is the difference between setting ScanState = false and disconnecting the OI Server?

Click to reveal answer

ScanState = false pauses all polling of field devices but keeps the OI Server connection alive. Tag values freeze at their last known values. Disconnecting the OI Server tears down the entire communication link. Use ScanState for temporary pauses (maintenance, testing); use disconnect for major maintenance or shutdown.

Question 4

What does a Quality value of C0:Good mean, and what should you do if you see 00:Bad instead?

Click to reveal answer

C0:Good means the data is fresh and reliable — the OI Server is actively polling and returning valid data. If you see 00:Bad, it means the data is unreliable or unavailable, typically because the OI Server is disconnected, the PLC is offline, or there is a communication timeout. Troubleshoot by checking: OI Server status, network connectivity, PLC availability, and deployment status.

Question 5

In a production environment, what should the Reconnect attribute be set to, and why?

Click to reveal answer

In production, Reconnect should be set to true. This ensures that if the OI Server connection drops (due to a network glitch, OI Server restart, etc.), Application Server will automatically attempt to re-establish the connection without requiring manual operator intervention. Without automatic reconnection, a temporary disruption could leave the system in a disconnected state indefinitely until someone notices and manually intervenes.

  • What Quality codes mean: C0:Good, 00:Bad, 40:Uncertain
  • How to configure ScanState to pause/resume polling and Reconnect to control automatic recovery
  • Best practices: always ensure the DI Object is deployed before binding attributes, set Reconnect = true in production, and monitor Quality values continuously
