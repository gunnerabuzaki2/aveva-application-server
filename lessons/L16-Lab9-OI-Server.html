Lesson 16: Device Integration Objects + Lab 9 — Configuring the OI Server

Source: AVEVA Application Server 2023 Training Manual, Module 5, Section 2 + Lab 9 (pages 5-7 to 5-12)

How Application Server connects to field devices through Device Integration objects — and configuring a DDE/SuiteLink client to communicate with a PLC via an OI Server.

16.1 The Data Path — From Galaxy Object to PLC

Reference: Module 5, Section 2 (pages 5-7 to 5-8)

In Lesson 15, you learned that Application Server does not talk directly to PLCs. Instead, it uses a chain of intermediaries. Here is the complete data flow path that every piece of field data must travel:

Complete data flow: Application Object → DDE/SuiteLink Client → OI Server → PLC (Module 5, page 5-7)
The four-step data path:
  1. Application Object — your Galaxy object (e.g., a Mixer, Pump, or Meter) contains attributes that reference live field data.
  2. DDE/SuiteLink Client — the Device Integration (DI) object inside the Galaxy that maps Galaxy attribute references to communication protocol requests.
  3. OI Server — a standalone Windows service that translates SuiteLink/DDE requests into the native protocol spoken by the PLC (e.g., Modbus TCP, Allen-Bradley).
  4. PLC / Field Device — the actual hardware controller on the plant floor with physical I/O points.
Recap from Lesson 15: The OI Server acts as a protocol translator. Application Server speaks SuiteLink/DDE (both Windows-era protocols), and the OI Server converts those into whatever proprietary protocol the PLC uses. Without the OI Server, Application Server has no way to reach devices that speak Modbus, Allen-Bradley, Siemens, or other vendor protocols.

16.2 The DDE/SuiteLink Client — Your Galaxy's Communication Bridge

Reference: Module 5, Section 2 (pages 5-8 to 5-9)

Every Galaxy ships with a built-in system object called $gDDESuiteLinkClient. This is the Device Integration (DI) template that you instantiate and configure to connect to each OI Server. Think of it as the Galaxy's built-in telephone — you create an instance of it, point it at the right OI Server, and it handles the conversation.

The Topic Tab

When you open the $gDDESuiteLinkClient object editor, the Topic tab is where you define the connections to your OI Server. It contains two key tables:

The $gDDESuiteLinkClient Topic tab — where topic names and associated attributes are configured (Module 5, page 5-8)
TablePurpose
Available Topics Lists topic names and their scan mode. Each topic represents a named connection to a specific data set on the OI Server. Initially empty until you configure connections.
Associated Attributes Maps Galaxy attribute references (e.g., ProdPLC.Topic1.$Sys$Status) to item addresses on the OI Server side. This is where you wire up individual data points.
Topic = a named data connection. You configure one topic per PLC or device group. Within that topic, you map individual Galaxy attributes to specific item addresses on the PLC. The scan mode determines how often data is polled (e.g., Polled at a fixed interval, or Unsolicited when the device pushes changes).

16.3 Lab 9 — Configuring the OI Server Connection

Reference: Module 5, Lab 9 (pages 5-9 to 5-12)

Lab Objectives:
  • Create a Device Integration object from the $gDDESuiteLinkClient template
  • Configure the Server node and Server name to connect to an OI Server
  • Deploy the DI object and verify communication
  • Monitor live attribute values from the PLC

Step 1: Create the DI Object Instance

Reference: Module 5, pages 5-9 to 5-10

To connect to a PLC, you create a new instance of the $gDDESuiteLinkClient template. This instance becomes your dedicated communication object for a specific device.

Step 1a: In the Templates pane, expand the system templates and locate $gDDESuiteLinkClient.

Step 1b: Right-click it and select New → Instance (for a direct instance) or New → Derived Template (for a template you can customize further).

Step 1c: Name the new object ProdPLC — this object will represent your production PLC connection.

Instance vs. Derived Template: Use an Instance when you need a single, direct connection object. Use a Derived Template when you want to create a reusable template that other DI objects can inherit from — useful if you have many PLCs of the same type.

Step 2: Configure the General Tab

Reference: Module 5, pages 5-10 to 5-11

Open the ProdPLC object editor. The General tab has two critical settings that tell the DI object where to find the OI Server:

The Simulator DI object configured with Server node=SOOPROD, Server name=OI.SIM.1 — use the same pattern for ProdPLC (Module 5, page 5-10)
PropertySimulator ValueProdPLC ValueWhat It Means
Server node SOOPROD S00PROD The hostname of the computer where the OI Server software is installed and running.
Server name OI.SIM.1 MBTCP The OI Server instance name — assigned when you install/configure the OI Server on that machine.
Two settings, both critical:
  • Server node — the network hostname (or IP address) of the machine running the OI Server. Must match the actual machine name exactly.
  • Server name — the specific OI Server instance name on that machine. Must match exactly what the OI Server was configured with (e.g., MBTCP for Modbus TCP, OI.SIM.1 for a simulator). This is case-sensitive.
Common connection failures: If the DI object shows "Disconnected" or "Faulted" after deployment, check these first:
  • Is the server node name spelled correctly? (Check the Windows hostname on the OI Server machine.)
  • Is the server name exactly right? (Check the OI Server configuration on that machine — even one wrong character will fail.)
  • Is the OI Server service actually running on the target node?
  • Is there network connectivity between the Application Server node and the OI Server node?

Step 3: Deploy the DI Object

Reference: Module 5, pages 5-11 to 5-12

With the ProdPLC object configured, you need to deploy it so the runtime can begin communicating with the OI Server.

Step 3a: In the Model-Tagname view, locate the ProdPLC object.

Step 3b: Right-click it and select Deploy from the context menu.

Step 3c: Wait for deployment to complete. The object's icon should show a green indicator when successfully deployed.

Step 3d: Repeat for the Simulator object if not already deployed.

Deploy = push to runtime. Until you deploy, the DI object exists only in the Galaxy database and does nothing. When you deploy, the Platform Engine on the target node starts the communication session with the OI Server. No data flows until deployment.

Step 4: Verify Connection Status

Reference: Module 5, page 5-12

After deployment, check whether the DI object has successfully connected to the OI Server by examining its diagnostic attributes:

AttributeExpected ValueWhat It Means
ConnectionStatus Connected Live connection to the OI Server is established
Reconnect false Connection is stable — no reconnection attempts in progress
ScanState true DI object is actively polling the OI Server for data
Topic1.$Sys$Status true The configured topic is active and communicating

Step 5: Monitor Live Values

Reference: Module 5, page 5-12

Once connected, the DI object starts pulling live values from the PLC. Open the object and view its attributes to see real-time data:

  • Live values — the PLC data is flowing through successfully
  • Timestamps — each value shows when it was last updated from the device
  • Quality = C0:Good — the data quality indicator confirms the value is valid and current
Quality codes are your trust indicator. The Quality field uses standard OPC/DDE quality codes:
  • C0:Good — value is fresh, reliable, and current. Data can be trusted.
  • 80:Uncertain — value may be stale or the communication path has intermittent issues.
  • 0:Bad — value cannot be trusted. Check the communication path.
Always verify quality alongside the actual value — a number that looks reasonable but has Bad quality is not trustworthy.

16.4 Where DI Objects Fit in Your Galaxy

Reference: Module 5, page 5-12

Your ProdPLC DI object, like all other objects, lives within the Galaxy hierarchy. DI objects typically sit in the ControlSystem area alongside platform objects, because they represent communication infrastructure rather than physical equipment:

Galaxy AreaObjectsPurpose
Plant → ControlSystem AppEngine1, GRPlatform, ProdPlatform, ProdPLC, Simulator Infrastructure: engines, platforms, and DI objects
Plant → Production Line1, Line2 (with contained Meters, Pumps, Mixers) Equipment: the actual process objects
Placement convention: DI objects represent communication paths to devices — they are infrastructure, not equipment. Place them under ControlSystem alongside your platform objects. Your equipment objects (Meters, Pumps, Mixers) live under Production areas.

16.5 Summary — The Complete DI Picture

ComponentRoleWhere It Lives
$gDDESuiteLinkClient Galaxy DI object — maps attributes to PLC items via topics Galaxy database; deployed to Platform Engine
OI Server Protocol translator — bridges SuiteLink/DDE to PLC protocol Dedicated Windows machine (the OI Server node)
PLC Field device providing process data Plant floor
The complete picture: Your application objects (Mixers, Pumps, Meters) contain attributes that reference items on the PLC. The DDE/SuiteLink Client DI object handles the actual communication, and the OI Server translates that communication into whatever protocol the PLC speaks. Configure the DI object, deploy it, and the data flows.

Key Takeaways

  1. Data flows through a four-step path: Application Object → DI Client → OI Server → PLC.
  2. The $gDDESuiteLinkClient is the built-in Galaxy DI object that manages communication with field devices.
  3. DI objects are configured with the Server node and Server name — both must exactly match the OI Server's actual configuration.
  4. After deployment, check ConnectionStatus and ScanState to verify communication is working.
  5. Quality codes (like C0:Good) tell you whether the data is trustworthy — always verify quality alongside values.
  6. The OI Server acts as a protocol translator — without it, Application Server cannot communicate with PLCs that speak non-SuiteLink protocols.
