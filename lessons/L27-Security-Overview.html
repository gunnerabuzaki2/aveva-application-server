Lesson 27: Security Overview + Lab 16 — Configuring Security

Source: AVEVA Application Server 2023 Training Manual, Module 9 Section 1 + Lab 16

Security Overview

In any industrial automation environment, controlling who can do what is not just a best practice — it is essential. Application Server provides a built-in security framework that governs access to the IDE, the Operator Control and Monitoring Client (OCMC), and runtime operations. Before you can configure security effectively, you need to understand what the system can and cannot do, and how its four-layer security model fits together.

Key Concept: Application Server security is about authorization, not protection against malicious attack. Passwords are encrypted, but they are stored in a database that is physically accessible on the host machine. The system is designed to prevent accidental or unauthorized misuse by employees, not to withstand a determined external attacker. If your environment demands stronger authentication, the recommended approach is to use OS-based authentication with Microsoft tools such as password timeout policies and electronic signature authentication.

How Security Prevents Unauthorized Activities

When you enable security in a Galaxy, every action — from opening an object in the IDE to acknowledging an alarm at the operator console — requires the user to authenticate and possess the appropriate permissions. A user who is not logged in, or who lacks a specific permission, will simply be denied that action. This layered approach means that even within the same Galaxy, different users can have very different levels of access depending on their role.

Security touches three main areas of the system. First, the Integrated Development Environment (IDE) controls who can create, modify, and deploy objects. Second, the Operator Control and Monitoring Client (OCMC) controls who can view and interact with the runtime environment. Third, the Runtime itself controls who can write to object attributes, acknowledge alarms, and perform other operational tasks.

The Security Model: Four Core Components

The Application Server security model is built on four components that work together in a chain. Understanding this chain is essential before you configure anything.

The security model hierarchy — from Security Classifications on individual attributes, up through Security Groups and Roles, to Users at the top.

1. Security Classifications are labels assigned to individual attributes of an AutomationObject. Each classification — such as Operate, Tune, or Configure — defines what kind of operation is required to write a value to that attribute. For example, an attribute labeled Operate can be changed during normal daily operations, while an attribute labeled Configure requires a higher-level permission and often requires the object to be taken Off scan first.

2. Security Groups bundle together all the objects that should share the same security behavior. Think of a security group as a bucket: every AutomationObject in the Galaxy belongs to exactly one bucket. When you grant a role access to a security group, that role gains access to every object in the group. A Default Security Group is created automatically and cannot be removed — any object you have not explicitly placed into another group remains here.

3. Roles generalize the functions that different types of users perform. For instance, you might create a role called Plant Operator for staff who need to acknowledge alarms and adjust setpoints, and another role called Software Engineer for developers who need full IDE access. Each role is granted Operational Permissions onto one or more Security Groups, and also receives General Permissions for IDE and SMC tasks.

4. Users are the individuals who log in. A single user can be assigned multiple roles, and their effective permissions are the sum of all permissions granted by those roles. This means you can build up complex access patterns by combining simple roles rather than creating a unique role for every person.

Key Concept: Every AutomationObject belongs to exactly one Security Group. This 1:1 relationship is what makes the model predictable. When you move an object from one security group to another, its effective security changes immediately. New objects are always placed in the Default Security Group until you move them.

Authentication Modes

Before users can be authorized, they must first be authenticated — that is, the system must verify who they are. Application Server supports four authentication modes, and you choose one for the entire Galaxy:

ModeDescriptionWhen to Use
None (Open Security)The default. All functions are open with no authentication required. No login dialogs appear.Development or training environments where access control is not a concern.
GalaxyUses the local Galaxy database for authentication. Users log in with a name and password stored in the Galaxy.Simple deployments where you want basic authentication without relying on a Windows domain.
OS User basedUses Windows NT individual user authentication. Users provide their domain, username, and password.Environments where each person should be individually identifiable through their Windows account.
OS Group basedUses OS group membership mapped to Galaxy Roles. Users log in through Windows and the system looks up which OS groups they belong to, then maps those to Galaxy Roles.Enterprise environments with existing Windows domain groups — the most common production choice.
The Security Settings window shows the four authentication modes as radio buttons, along with the Login time and Role update fields used by OS Group based authentication.

When you choose OS Group based authentication, two additional settings appear:

  • Login time (ms) — The timeout period (in milliseconds) for validating a user's membership against their OS groups. The default is 0, meaning the system waits indefinitely. Setting a value like 3000 ms is useful in training environments to avoid long pauses.
  • Role update (ms) — The interval between re-validation attempts for each OS group. This controls how often the system checks whether a user's group memberships have changed.

Security Permissions: General and Operational

Once a user is authenticated, permissions determine what they can actually do. There are two categories of permissions:

General Permissions control access to IDE and SMC configuration tasks — things like starting the IDE, configuring objects in OCMC, or using the Object Viewer. The Administrator role is special: its General Permissions can never be modified, which guarantees that at least one role always has full administrative access.

Operational Permissions control what users can do at runtime. There are eight operational permissions, each corresponding to a specific type of action:

#PermissionWhat It Controls
1Can Modify "Operate" AttributesDaily operational tasks — changing setpoints, outputs, and control modes.
2Can Modify "Tune" AttributesRuntime tuning — adjusting alarm setpoints, PID sensitivity, and similar parameters.
3Can Modify "Configure" AttributesMajor configuration changes; requires the object to be taken Off scan first.
4Can Acknowledge AlarmsManually acknowledging alarms in the OCMC.
5Can Shelve AlarmsTemporarily suppressing or restoring individual alarms.
6Can Modify Alarm ModesChanging the operational mode of an alarm (e.g., enabling or disabling it).
7Can Modify Plant StatesModifying plant states used for state-based alarming.
8Can Verify WritesProviding an authentication signature for Verified Writes.
Important Distinction: "Can Modify Operate Attributes" is for day-to-day changes like adjusting a temperature setpoint. "Can Modify Configure Attributes" is for fundamental changes like altering an object's configuration structure — and it requires the object to be taken Offline first. Confusing these two is a common source of confusion when designing roles.

Key Behaviors with OS Group Based Authentication

When you are using OS Group based authentication, a few behaviors are worth keeping in mind:

  • Remote node access: A user must log on at least once before writing to attributes on a remote node. The system caches credentials locally after the first successful authentication.
  • Domain controller failure: If the domain controller becomes unavailable, the system falls back to locally cached data. This means existing users can still log in, but their group memberships will not be updated until the domain controller is reachable again.
  • User full name: The user's full name is only available if the domain controller was connected during the user's first login. If it was disconnected, the full name field may be blank.
  • Group type restriction: Only OS groups of type Security (not Distribution) are usable for mapping to Galaxy Roles.

Lab 16: Configuring Security

Lab Objectives:
  • Configure security in a Galaxy by enabling OS Group based authentication
  • Create security groups organized by plant area
  • Add security roles and assign appropriate permissions
  • Test the security configuration by logging in as different users

In this hands-on lab, you will walk through the complete process of enabling and configuring security for your Galaxy. By the end, different users will have different levels of access depending on their role — some will be able to open the IDE, while others will be restricted to alarm acknowledgment and setpoint changes.

Step 1: Enable OS Group Based Authentication

Start by opening the Security configuration from the IDE. Navigate to the Galaxy menu and select Configure → Security. This opens the Security Settings window where you will choose how users authenticate.

Accessing the Security configuration from the Galaxy menu in the IDE.

On the Authentication Mode tab, select the OS Group based radio button. This tells the Galaxy to use Windows domain groups for authentication. Set the Login time to 3000 milliseconds. This value provides a reasonable timeout for validation against OS groups during testing without causing long delays.

Do NOT click Save yet! You still need to configure security groups and roles before saving. If you save now with OS Group based authentication enabled, you will immediately be prompted to log in — and you may not have any roles configured yet.
Selecting OS Group based authentication and setting the Login time to 3000 ms.

Step 5: Configure Role Permissions

Now comes the most detailed part of the lab: configuring the permissions for each role. You will work through each role one at a time, checking or unchecking permissions based on what that role needs to do.

Default Role: This role applies to anyone who logs in but does not match any other role. Restrict it significantly: uncheck all IDE Permissions and the SMC "Can Start" permission. Leave only "Can Write to GObject Attributes using ObjectViewer" checked. Also uncheck the Default operational group so this role has minimal runtime access.

The Default role with minimal permissions — only ObjectViewer write access enabled.

Administrator Role: Leave this role with all permissions checked. The system prevents you from modifying the Administrator's General Permissions anyway, so this role always has full access to everything.

The Administrator role with full permissions — this role cannot be locked out.

Application Administrators: Grant all permissions. This role is for team leads who need complete access to both IDE configuration and runtime operations.

Application Administrators role with all IDE, SMC, and operational permissions checked.

Application Developers 1: Grant IDE Permissions (but not User Configuration) and SMC Permissions (but not Start/Stop). For operational permissions, grant Configure and Operate on all four groups: Default, Line1, Line2, and ControlSystem. This allows developers to modify object configurations and daily settings across the entire plant.

Application Developers 1 — IDE and SMC access plus Configure and Operate on all security groups.

Plant Maintenance 1: Grant SMC Permissions only (no Start/Stop). For operational permissions, grant Configure and Tune on Line1, Line2, and ControlSystem. This gives maintenance staff the ability to perform configuration changes and runtime tuning but no IDE access.

Plant Maintenance 1 — SMC access with Configure and Tune permissions on all production groups.

Plant Operators 1: This role is for operators working on Line 1 only. Grant Acknowledge Alarms and Can Modify "Operate" Attributes on Line1 only. No other operational groups are checked, and no IDE or SMC configuration permissions are granted.

Plant Operators 1 — limited to Line1 with Acknowledge Alarms and Modify Operate permissions.

Plant Operators 2: Identical to Plant Operators 1, but for Line 2. Grant Acknowledge Alarms and Can Modify "Operate" Attributes on Line2 only.

Plant Operators 2 — limited to Line2 with Acknowledge Alarms and Modify Operate permissions.

Plant Supervisors 1: Supervisors need broader access than operators. Grant Acknowledge Alarms, Can Modify "Operate" Attributes, and Can Verify Writes on all three operational groups: Line1, Line2, and ControlSystem. The Verify Writes permission is what distinguishes supervisors from operators — it allows them to provide an authentication signature when critical write operations require verification.

Plant Supervisors 1 — full operational access to all groups with Verify Writes permission.

Step 6: Save and Test

Click Save to apply your security configuration. Because you changed the authentication mode from None to OS Group based, the system immediately presents a login dialog. Every subsequent action in the IDE will require authentication.

The login dialog that appears after enabling security — users must authenticate before proceeding.

Test 1 — Plant Maintenance user: Attempt to log in as a Plant Maintenance user and try to open the IDE. You should receive an Access Denied message. This is expected because the Plant Maintenance 1 role was not granted any IDE permissions — it only has SMC and operational permissions.

Access Denied when a Plant Maintenance user tries to open the IDE — this role lacks IDE permissions.

Test 2 — Application Developer: Log out and log back in as an Application Developer. This time, the IDE should open successfully. Check the user information in the IDE to verify you are logged in with the correct identity.

Logging in as an Application Developer — the IDE opens and displays the authenticated user.
After enabling security, you may notice that some objects need redeployment. This happens because moving objects between security groups modified their SecurityGroup attribute. The IDE will flag these objects as needing deployment. The redeployment process will be covered in the next lesson (Lab 17).

📚 Knowledge Check

Q1: In the Application Server security model, what is the relationship between Security Groups, Roles, and Users?

Click to reveal answer

Security Groups contain AutomationObjects that share the same security behavior. Roles are granted Operational Permissions onto Security Groups. Users belong to one or more Roles and receive the sum of all permissions from their assigned Roles.

Q2: What is the difference between "Can Modify Operate Attributes" and "Can Modify Configure Attributes"?

Click to reveal answer

"Can Modify Operate Attributes" allows daily operational tasks like changing setpoints, outputs, and control modes. "Can Modify Configure Attributes" allows major configuration changes and requires the object to be taken Off scan first.

Q3: Why did the Plant Maintenance user receive an "Access Denied" message when trying to open the IDE?

Click to reveal answer

The Plant Maintenance 1 role was not granted "Can Start the IDE" permission under IDE Permissions in General Permissions. This role only has SMC Permissions and Operational permissions (Tune and Configure) for the production lines — no IDE access at all.

Q4: What is the Login time setting used for in OS Group based authentication, and why did the lab set it to 3000?

Click to reveal answer

Login time (in milliseconds) is the timeout period for validating user membership against OS groups. After the timeout, the system defaults to locally cached data. The lab set it to 3000 ms to allow seamless testing when switching between users, especially when the domain controller response is slow.

Q5: What distinguishes the Plant Supervisors 1 role from Plant Operators 1 in terms of permissions?

Click to reveal answer

Plant Supervisors 1 has access to all three operational areas (Line1, Line2, ControlSystem) and includes the Can Verify Writes permission. Plant Operators 1 is limited to Line1 only and has only Can Acknowledge Alarms and Can Modify Operate Attributes — no Verify Writes.
