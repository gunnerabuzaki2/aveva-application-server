Lesson 6: Module 3 — Plant Model + Deployment Model

Source: AVEVA Application Server 2023 Training Manual, Module 3, Sections 1-2 (pages 3-3 to 3-12)

How Application Server organizes your plant into a logical hierarchy, and how objects move from design to live execution.

1. The Plant Model

Reference: Module 3, Section 1 (pages 3-3 to 3-5)

The Plant Model is Application Server's logical representation of your physical plant. It organizes objects into an area hierarchy that mirrors how your plant is actually structured. The Plant Model is built using the Model view in the IDE.

Area Hierarchy

The Plant Model organizes objects into a hierarchical tree. The training Galaxy demonstrates a typical five-level structure:

Plant
  ├── Sections
  │     ├── Areas
  │     │     ├── Production Lines
  │     │     │     └── Manufacturing Cells
  │     │     └── ...
  │     └── ...
  └── ...
Hierarchy LevelTraining Galaxy ExampleDescription
PlantPlantTop-level container representing the entire facility
SectionProduction, PackagingMajor functional divisions within the plant
AreaControlSystemSpecific process areas within a section
Production LineLine1, Line2Individual production lines
Manufacturing CellMixer100Specific equipment groups or cells on a line

The exact hierarchy levels you use depend on your plant layout. The key principle: structure your Plant Model to mirror your physical plant so operators and engineers can navigate it intuitively.

The Plant Model — showing the area hierarchy from Plant down to Manufacturing Cells (Module 3, page 3-3)

Alarm Distribution

One of the primary functions of the area hierarchy is alarm scoping. The hierarchy determines which alarms each user can see:

User RoleAlarm Visibility
Operator Sees only alarms for their own area — focused on the equipment they are responsible for
Supervisor Sees all alarms across the entire plant — overview of all areas
Why this matters: An operator on Line1 doesn't need to see alarms from Line2 or Packaging — that would be distracting noise. A supervisor, on the other hand, needs the full picture. The area hierarchy makes this scoping automatic.

2. Plant Model in the IDE

Reference: Module 3, Section 1 (pages 3-4 to 3-5)

The Model-Tagname View

In the IDE, the Plant Model is visible in the Model pane. The Model-Tagname view shows the full hierarchy of all objects in your Galaxy, organized by area. This is where you build, configure, and navigate your plant structure.

The Model-Tagname view in the IDE — showing the TrainingGalaxy hierarchy with sub-areas and equipment (Module 3, page 3-4)

TrainingGalaxy Hierarchy

The training Galaxy organizes its objects as follows:

AreaContents
Unassigned Area New objects that haven't been placed into the hierarchy yet — a temporary holding zone
Plant Top-level area
  ↳ ControlSystem Control system objects
  ↳ Production Production-related areas
      ↳ Line1 Equipment on Production Line 1 (e.g., Mixer100, Mixer200)
      ↳ Line2 Equipment on Production Line 2
  ↳ Packaging Packaging area equipment
  ↳ Receiving Receiving area — inbound materials
  ↳ Shipping Shipping area — outbound products

Equipment objects (like Mixer100, Mixer200) are nested under their respective areas. You can see them in the Model-Tagname view when you expand each area node.

Workflow tip: Set up your area hierarchy first, then create objects directly into their target areas. Avoid dumping everything into the Unassigned Area — it becomes cluttered quickly and objects there cannot be deployed.

3. The Deployment Model

Reference: Module 3, Section 2 (pages 3-7 to 3-10)

Deployment is the process of moving an object's configuration from the Galaxy Repository to a live runtime environment. When you deploy an object, it becomes live — executing logic, reading data, and generating alarms.

Deployment vs. Undeployment

Application Server uses a deliberate deployment model — changes don't go live automatically:

ConceptDescription
Deploy Object configuration is sent to the target platform. The object starts executing at runtime.
Undeploy Object is removed from the runtime environment. It stops executing. The object still exists in the Galaxy (IDE) but is not running.
Manual Update Required When you change an object in the IDE, the changes are not automatically pushed to the running environment. You must manually redeploy for changes to take effect.
The Deployment Model — deployment vs. undeployment, instances only, manual updates, and GR offline behavior (Module 3, page 3-7)

Instances Only — Not Templates

Critical rule: You can only deploy instances, not templates. Templates (like $Area, $AppEngine, $WinPlatform) are base classes — they define how objects work. You must create an instance of the template and deploy the instance. Templates themselves are never deployed.

This is fundamental to how Application Server works:

  1. You inherit from a template to create a new object (an instance)
  2. You configure the instance with your specific settings
  3. You deploy the instance — never the template

Manual Update Requirement

Important: Changes made in the IDE are not automatically deployed. You must manually redeploy the object for the changes to take effect at runtime. This exists for safety — it prevents accidental changes from going live in a production environment.

Galaxy Repository Offline Behavior

What happens when the GR goes offline?

When the Galaxy Repository (GR) is unavailable, already-deployed objects continue running normally on the AppEngines. The GR is only needed for development operations (deploy, undeploy, configuration changes). The runtime executes independently — this means a GR failure does not stop your plant from running.

Key takeaway: The GR is needed for development operations (deploy, undeploy, modify). It is NOT needed for runtime — deployed objects keep running independently on the AppEngines. This separation of development and runtime is one of Application Server's strengths.

4. Deployment View — The Full Deployment Tree

Reference: Module 3, Section 2 (pages 3-8 to 3-10)

The Deployment View shows the hierarchy of platforms and engines that host your deployed objects. The training Galaxy demonstrates a typical multi-platform deployment:

The Deployment View — showing the full deployment tree with platforms, engines, and hosted objects (Module 3, page 3-9)

Deployment Tree Structure

The training Galaxy's deployment tree looks like this:

TrainingGalaxy
  ├── Unassigned Host
  │
  ├── GRPlatform
  │     ├── AppEngine1 Backup
  │     └── DIO_Engine2
  │           └── R_ProdPLC2
  │
  └── ProdPlatform
        ├── AppEngine1
        │     ├── ControlSystem
        │     │     └── ...
        │     ├── Line1
        │     │     └── Mixer100 / Mixer200 / ...
        │     └── ...
        └── DIO_Engine1
              └── R_ProdPLC1

Key Objects in the Deployment Tree

ObjectPurpose
GRPlatform The platform hosting the Galaxy Repository. Also hosts AppEngine1 Backup (redundancy) and DIO_Engine2 for I/O communication with R_ProdPLC2.
ProdPlatform The main production platform. Hosts AppEngine1 (which executes the plant objects: ControlSystem, Line1, Mixer100, etc.) and DIO_Engine1 for I/O communication with R_ProdPLC1.
AppEngine1 The runtime engine that executes deployed objects — equipment, areas, and their logic.
DIO_Engine1 / DIO_Engine2 Data I/O engines that handle communication with physical controllers (PLCs) via drivers.
R_ProdPLC1 / R_ProdPLC2 Communication paths to the production PLCs — configured through the DIO engines.
AppEngine1 Backup Redundant backup for AppEngine1 — provides failover capability.
Unassigned Host Objects that haven't been assigned to a specific platform yet.

Hosting Rules — Parent Must Deploy First

Critical rule: A child object cannot be deployed unless its parent is deployed first. For example, you cannot deploy an equipment object to AppEngine1 unless ProdPlatform (which hosts AppEngine1) is already deployed. Always deploy top-down: platform first, then engines, then objects.
Deployment order:
  1. Deploy the platform (e.g., ProdPlatform) — this claims the physical computer
  2. Deploy the engines (e.g., AppEngine1, DIO_Engine1) — these start running on the platform
  3. Deploy the objects (e.g., Mixer100, ControlSystem) — these execute within the engines

Key Takeaways

  1. The Plant Model organizes objects into a logical area hierarchy: Plant → Sections → Areas → Production Lines → Manufacturing Cells.
  2. The hierarchy determines alarm scoping: operators see only their area's alarms; supervisors see all.
  3. The Model-Tagname view in the IDE shows the full Galaxy hierarchy, with Unassigned Area for unplaced objects.
  4. Deployment moves objects from the Galaxy Repository to live runtime. Only instances can be deployed — never templates.
  5. Changes in the IDE require manual deployment — they do not auto-deploy to runtime.
  6. The Deployment View shows the hosting tree: platforms host engines, engines execute objects.
  7. Parent must deploy before child — you cannot deploy an object to an engine unless the engine's platform is already deployed.
  8. The GR can go offline without stopping running objects — runtime is independent of development.
