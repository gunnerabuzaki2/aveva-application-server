Lesson 17: Device Integration Objects

Source: AVEVA Application Server 2023 Training Manual, Module 5, Section 2 (pages 5-13 to 5-18)

The Galaxy objects that bridge your application to the real world — how Application Server communicates with field devices through Device Integration objects.

17.1 Data Flow — How Application Server Talks to Devices

Reference: Module 5, Section 2 (page 5-13)

To understand Device Integration (DI) objects, you first need to see the complete data path from an application object in the Galaxy all the way to a physical PLC on the plant floor.

Data flow path: Application Object → DDE/SuiteLink Client → OI Server → PLC (Module 5, page 5-13)
The four-step data path:
  1. Application Object — your Galaxy object (e.g., a Mixer or Pump) needs to read/write data from a field device.
  2. DDE/SuiteLink Client — the Device Integration object inside the Galaxy that translates Galaxy attribute references into communication protocol requests.
  3. OI Server (Operating Interface Server) — a standalone Windows application that speaks the native protocol of the PLC (e.g., Modbus TCP, Allen-Bradley, Siemens). It bridges the gap between SuiteLink and the device.
  4. PLC / Field Device — the actual hardware controller with I/O points.
Why not talk directly? Application Server does not natively understand every PLC protocol. The OI Server acts as a protocol translator — Application Server speaks SuiteLink/DDE (both Microsoft-era protocols), and the OI Server converts those into whatever protocol the PLC speaks.

17.2 The DDE/SuiteLink Client Object

Reference: Module 5, Section 2 (pages 5-13 to 5-14)

The $gDDESuiteLinkClient is the built-in Galaxy object that acts as the communication bridge. Every Galaxy ships with one — it is a system global object that you configure for your specific devices.

The $gDDESuiteLinkClient Topic tab — where topics and item references are defined (Module 5, page 5-13)

Key Tabs and Configuration

Tab / PropertyPurpose
Topic tab Defines topic names that map to specific data sets on the OI Server. Each topic has a Scan Mode and associated Attribute/Item Reference pairs.
Available Topics table Shows topic names and their scan mode (e.g., Polled, Unsolicited, etc.). Initially empty until you configure connections.
Associated Attributes table Maps Galaxy attribute references (e.g., ProdPLC.Topic1.$Sys$Status) to item references on the OI Server side.
Topic = a named connection to a specific set of data on the OI Server. You configure a topic per PLC (or per device group), and within that topic, you map individual attributes to item addresses on the PLC.

17.3 Creating a Device Integration Object

Reference: Module 5, Section 2 (pages 5-14 to 5-15)

To connect to a PLC, you create a new instance of the $gDDESuiteLinkClient template in the Templates pane. This becomes your dedicated communication object for that device.

Right-click $gDDESuiteLinkClient in Templates → New → Instance/Derived Template to create a new DI object (Module 5, page 5-14)

Step-by-Step: Creating the ProdPLC DI Object

Step 1: In the Templates pane, expand the system templates and locate $gDDESuiteLinkClient.

Step 2: Right-click it and select New → Instance (for a direct instance) or New → Derived Template (for a template you can customize further).

Instance vs. Derived Template: An Instance is a direct copy of the template — quick and simple for one-off use. A Derived Template lets you customize the template first and then create multiple instances from your customized version. For most projects, a Derived Template gives you more flexibility.

Step 3: Name the new object ProdPLC — this object will represent your production PLC connection.

Step 4: Open the ProdPLC object editor and configure its General tab:

Configuring the General Tab

The General tab of the DI object editor contains two critical settings that determine which OI Server the object will communicate with:

For comparison: the Simulator DI object configured with Server node=SOOPROD, Server name=OI.SIM.1 (Module 5, page 5-14)
PropertyValue for SimulatorValue for ProdPLC
Server node SOOPROD S00PROD
Server name OI.SIM.1 MBTCP
ProdPLC object: Server node=S00PROD, Server name=MBTCP — connects to a Modbus TCP OI Server (Module 5, page 5-15)
Two critical settings:
  • Server node — the computer name where the OI Server software is running. This must match the actual hostname of that machine.
  • Server name — the OI Server instance name, which you assign when you install/configure the OI Server. It must exactly match what the OI Server was configured with (e.g., MBTCP for Modbus TCP, OI.SIM.1 for a simulator).
⚠ Pitfall — Mismatched Server Settings: If the server node or server name is wrong, the DI object will fail to connect. Common symptoms include a ConnectionStatus of "Disconnected" or "Faulted" after deployment. Always verify both values against the actual OI Server configuration on the target machine (check OCMC for the correct server name).

17.4 Where DI Objects Fit in Your Galaxy

Reference: Module 5, Section 2 (page 5-15)

Your ProdPLC DI object, like all other objects, lives within the Galaxy hierarchy. Here is how the Galaxy structure looks with all the objects you have created so far:

Model-Tagname view: Plant → ControlSystem (ProdPLC selected) alongside Production (Line1, Line2 with contained equipment L1, T1) (Module 5, page 5-15)

The Galaxy hierarchy at this point:

AreaObjects
Plant → ControlSystem AppEngine1 (Application Engine), GRPlatform (GR node), ProdPlatform (production node), ProdPLC (DI object for production PLC), Simulator (DI object for simulator)
Plant → Production Line1, Line2 (production lines with contained equipment)
Placement matters: DI objects typically live in the ControlSystem area alongside the platform objects, because they represent infrastructure (communication paths) rather than equipment. Your equipment objects (Meters, Pumps, etc.) live under Production areas.
Two DI objects in our Galaxy:
  • ProdPLC — connects to a real Modbus TCP PLC via the MBTCP OI Server on node S00PROD
  • Simulator — connects to a software simulator via the OI.SIM.1 OI Server on node SOOPROD. The simulator is invaluable during development — it provides fake process data so you can build and test your application without any physical hardware.

17.5 Topics and Attribute Mapping

Reference: Module 5, Section 2 (pages 5-15 to 5-16)

Once you have created a DI object (like ProdPLC), you need to configure which PLC data points your Galaxy attributes will read from and write to. This is done through topics and attribute mapping.

How Topics Work

A topic is a named communication channel within the DI object. Each topic corresponds to a specific group of data points on the PLC. Within a topic, you define individual item references that map to specific PLC addresses.

ConceptExampleWhat It Means
Topic Name Topic1 A named channel pointing to a specific data set on the OI Server
Item Reference Agitator_001 A specific PLC register or tag address within the topic
Full Attribute Path ProdPLC.Topic1.Agitator_001 The complete path an Application Server attribute uses to reach a PLC value
The binding string format: When you configure an attribute's I/O source, you use the syntax TopicName.ItemName. For example, if your OI Server has a topic called Topic1 and the PLC tag is Agitator_001, the I/O source would be Topic1.Agitator_001. You type this into the I/O source field for whichever attribute needs the live data.

Attribute Types and I/O Binding

Different object types use I/O binding in different ways:

Object TypeCommonly Bound AttributesTypical Use
Discrete PV (Process Value), SP (Setpoint) On/off states: pump running, valve open/closed
Analog PV, SP, SPH (High), SPL (Low) Continuous values: temperature, pressure, flow, level
Motor PV (Status), SP (Command) Motors with start/stop and status feedback
Pump PV, SP, RunCmd Pumps with run command and status
Simulation vs. I/O: Before you bind an attribute to a real I/O source, it runs in simulation mode (using an internal expression or hardcoded value). This is extremely useful during development — you can build and test your entire application without any hardware. When you are ready to go live, you simply change the I/O source from the simulation expression to the real PLC tag address.

17.6 Connection Status and Monitoring

Reference: Module 5, Section 2 (pages 5-16 to 5-18)

After deploying a DI object, you need to verify that it has successfully connected to the OI Server and is receiving live data. Application Server provides several diagnostic attributes for this purpose.

Key Diagnostic Attributes

AttributeExpected Value (Connected)What It Means
ConnectionStatus Connected The DI object has established a live connection to the OI Server
Reconnect false The connection is stable — no reconnection attempts in progress
ScanState true The DI object is actively scanning/polling the OI Server for data
Topic1.$Sys$Status true The configured topic is active and communicating properly
ProdPLC attributes: ConnectionStatus=Connected, Reconnect=false, ScanState=true, Topic1.$Sys$Status=true — confirms successful communication (Module 5, page 5-17)

Monitoring Live Values

Once connected, the DI object starts pulling live values from the PLC. You can see these values in real time by opening the object and viewing its attributes.

ProdPLC live monitoring: attributes with timestamps and Quality=C0:Good — values are updating in real time from the PLC (Module 5, page 5-18)

When you see attributes with:

  • Live values — the PLC data is flowing through successfully
  • Timestamps — each value shows when it was last updated
  • Quality = C0:Good — the data quality indicator confirms the value is valid and current
⚠ Pitfall — If ConnectionStatus shows "Disconnected" or "Faulted":
  • Is the OI Server running on the target node?
  • Is the server node name correct?
  • Is the server name (OI Server instance) spelled exactly right?
  • Is there network connectivity between the Application Server node and the OI Server node?
  • Check the OI Server's own logs for error messages.
Quality codes matter: The Quality field uses standard OPC/DDE quality codes. C0:Good means the value is fresh and reliable. If you see 0:Bad or 80:Uncertain, it means the communication path has an issue, even if the value looks reasonable. Always check quality alongside the value.

17.7 Summary — DI Objects in Context

ComponentRoleLives On
$gDDESuiteLinkClient Galaxy DI object — maps attributes to PLC items via topics Galaxy database (deployed to Platform Engine)
OI Server Protocol translator — bridges SuiteLink to PLC protocol Dedicated Windows machine (the OI Server node)
PLC Field device providing process data Plant floor
The complete picture: Your application objects (Mixers, Pumps, Meters) contain attributes that reference items on the PLC. The DDE/SuiteLink Client DI object handles the actual communication, and the OI Server translates that communication into whatever protocol the PLC speaks. Deploy the DI object, and the data flows.

17.8 Key Terms

TermDefinition
Device Integration (DI) Object A Galaxy object (instance of $gDDESuiteLinkClient) that manages communication between Application Server and field devices through an OI Server
$gDDESuiteLinkClient The built-in system template for creating Device Integration objects in the Galaxy
Server Node The computer name where the OI Server software is running
Server Name The OI Server instance name (e.g., MBTCP, OI.SIM.1)
Topic A named communication channel within the DI object, mapping to a specific set of PLC data points
Item Reference A specific PLC address or tag within a topic (e.g., Agitator_001)
I/O Binding The process of linking an object's attribute to a real-world data point through a DI object and OI Server
ConnectionStatus A diagnostic attribute showing whether the DI object is connected to the OI Server
Quality Code A standard indicator (e.g., C0:Good, 0:Bad, 80:Uncertain) reflecting the reliability of a data value

17.9 Quiz

Question 1

What are the four components in the data path from an Application Object to a PLC?

Click to reveal answer

Application Object → DDE/SuiteLink Client → OI Server → PLC

The Application Object communicates through the DDE/SuiteLink Client DI object, which sends requests to the OI Server. The OI Server translates those requests into the PLC's native protocol and communicates with the field device.

Question 2

What is the purpose of the OI Server in the communication path?

Click to reveal answer

The OI Server is a protocol translator. It bridges SuiteLink/DDE communication from Application Server to the native protocol spoken by the PLC (e.g., Modbus TCP, Allen-Bradley). Application Server does not natively understand every PLC protocol — the OI Server handles the translation.

Question 3

What two properties must be correctly configured on a DI object for it to connect to an OI Server?

Click to reveal answer

The Server node (computer name where the OI Server runs) and the Server name (the OI Server instance name, e.g., MBTCP). Both must exactly match the actual OI Server configuration — a mismatch will cause the DI object to fail to connect.

Question 4

What is the difference between creating a Derived Template from $gDDESuiteLinkClient versus creating a direct Instance?

Click to reveal answer

A Derived Template lets you customize the template first (modify settings, add attributes) and then create multiple instances from your customized version. A direct Instance is a straightforward copy of the system template — quick and simple for one-off use. Derived Templates give you more flexibility for reuse.

Question 5

What does a ConnectionStatus of "Connected" and a Quality code of "C0:Good" tell you?

Click to reveal answer

ConnectionStatus = Connected means the DI object has established a live connection to the OI Server. Quality = C0:Good means the data values being received are fresh, reliable, and current. Together, they confirm that the communication path is healthy and the data can be trusted.
