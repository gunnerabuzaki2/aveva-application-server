Lesson 22: Lab 12 — Configuring and Retrieving History

Source: AVEVA Application Server 2023 Training Manual, Lab 12: Configuring and Retrieving History (pages 6-11 to 6-22)

Hands-on lab covering platform-level history settings, enabling history on template attributes, deploying to production, and verifying data collection using Historian Client Web.

22.1 Overview — What This Lab Covers

Reference: Lab 12 (pages 6-11 to 6-12)

This lab walks through the complete history workflow — from configuring the platform's Store and Forward settings, to enabling history on individual attributes in your templates, to deploying the changes to the production platform, and finally verifying that data is flowing into the Historian using Historian Client Web.

Lab Objectives:
  • Review and configure the WinPlatform's Store and Forward (S&F) settings for historical data
  • Enable History on attributes in the SMeter and SValve templates
  • Configure history parameters: Trend High/Low, deadband, interpolation, and storage period
  • Check in template changes and verify propagation to descendant objects
  • Deploy the updated galaxy to the production platform
  • Use Historian Client Web to retrieve and view historical trends

Prerequisites: You should have completed Labs 9–11 with the production platform deployed, objects running under AppEngine1 → ControlSystem → Line1/Mixer100, and Historian Client Web accessible (typically at http://localhost:32573).

22.2 Configuring Platform History and Store-and-Forward

Reference: Lab 12 (pages 6-11 to 6-13)

Before enabling history on individual attributes, verify that the WinPlatform is properly configured for historical data collection. The platform's Store and Forward settings control how historical data is buffered locally before being sent to the Historian.

ProdPlatform General tab: Network address = S00PROD, History store and forward path = C:\S&F, Min RAM = 1024 MB, Stats average period = 10000 ms. Additional tabs visible: Engine, Alarms, Platform History, Scheduler History (page 6-11)

The key settings on this tab:

SettingValuePurpose
NetworkS00PRODThe network address where this platform is accessible
History store and forward pathC:\S&FLocal disk path where historical data is buffered before forwarding to the Historian
Min RAM1024 MBMinimum RAM reserved for platform operations
Stats average period10000 msHow often platform statistics are averaged (10 seconds)
Store and Forward (S&F) is critical: The platform does not write directly to the Historian. Instead, historical data is first written to the local S&F cache at C:\S&F. If the Historian is temporarily unreachable (network outage, maintenance, etc.), data is buffered on disk. When connectivity is restored, the cached data is forwarded automatically. This guarantees zero data loss during transient failures.
Disk Space Warning: The S&F cache consumes disk space. If the Historian is down for an extended period, the cache can grow large. Always ensure the S&F path has sufficient free space. In production, use a dedicated, high-capacity drive — never use the system drive.

The Platform History tab (also visible on this page) provides additional settings for controlling exactly which attributes are historized at the platform level. In most configurations, you enable history at the individual attribute level (covered in the next section) rather than at the platform level.

22.3 Enabling History on Template Attributes

Reference: Lab 12 (pages 6-13 to 6-16)

History is enabled on a per-attribute basis. The most efficient approach is to enable it on the template, then check in the changes so they propagate to all descendant (instance) objects. This lab enables history on two key attributes: the SMeter PV and the SValve command/state attributes.

Step 1: Enable History on the SMeter PV Attribute

Reference: Lab 12 (page 6-13)

Navigate to the SMeter template and open the PV attribute in the Attribute Viewer. Ensure the following properties are set:

SMeter PV attribute configuration: Data type = Float, Object writeable = checked, I/O enabled = checked, History = checked. Enabling the History checkbox tells the system to collect this attribute's values for historical storage (page 6-13)

The critical step is checking the History box. Without this, the Historian will never collect data for this attribute, regardless of any other configuration.

Additional SMeter PV attribute details — verifying the attribute configuration before proceeding to History-specific settings (page 6-13)

Step 2: Configure History Parameters

Reference: Lab 12 (pages 6-14 to 6-15)

After checking the History box, expand the History section to configure the detailed parameters that control how data is collected and displayed:

History configuration: Trend High = 10.0, Trend Low = 0.0, Rollover = 0.0, Interpolation = SystemDefault, Swinging Door compression unchecked (page 6-14)
ParameterValueMeaning
Trend High10.0Upper bound of the Y-axis for Historian Client trends — scale the display to this maximum
Trend Low0.0Lower bound of the Y-axis for Historian Client trends — scale the display to this minimum
Rollover0.0Value at which the counter resets (0.0 = no rollover — used for counter-type tags)
InterpolationSystemDefaultHow the Historian fills in values between data points (SystemDefault uses the server's configured method)
Swinging DoorUncheckedCompression algorithm — when unchecked, all data points are stored without compression
Trend High/Low Best Practice: Set Trend High to slightly above your maximum expected value, and Trend Low to slightly below your minimum. For a level sensor that reads 0–10, Trend Low = 0.0 and Trend High = 10.0 is appropriate. These values only affect the display in Historian Client — they do not filter or truncate data.
On-Change vs. Periodic Storage:
• Force storage period = 0 ms (default) means the Historian stores a new value only when the attribute changes — this is "on-change" storage, the most common and efficient mode.
• Setting a non-zero period (e.g., 5000 ms) forces the Historian to store a value every N milliseconds regardless of whether it changed — useful for critical process variables where you want guaranteed periodic snapshots.
• Value deadband (typically 0.0) controls the minimum change required to trigger on-change storage. A deadband of 0.0 means every single change, no matter how small, is recorded.

Step 3: Review Inherited Attributes in the Mixer Agitator

Reference: Lab 12 (pages 6-16 to 6-17)

Before checking in the template changes, it is useful to understand how attributes are inherited across the template hierarchy. The $Mixer_Agitator object demonstrates this clearly:

$Mixer_Agitator Attributes: CMD and PV are inherited from the [$Motor] template (shown in brackets), while Speed.PV and Speed.SP are local attributes. The bracketed template name indicates the source of each inherited attribute (page 6-16)
Additional view of $Mixer_Agitator attributes — confirming the full attribute list before template check-in (page 6-16)

Similarly, the SValve attributes show the same inheritance pattern:

SValve Attributes: CMD (Valve Command) and LSOpen (Limit Switch Open) — both inherited from the template structure. History will be enabled on these attributes via the parent template (page 6-17)
SValve attribute details — verifying the command and state attributes before enabling history (page 6-17)

Template vs. Instance: Always enable history on the template, not on individual instances. If you enable it on an instance, you would need to repeat the process for every other instance. Template-level changes propagate to all instances during check-in.

22.4 Check-In — Propagating Changes to All Instances

Reference: Lab 12 (pages 6-18 to 6-19)

After enabling history on the template attributes, check in the changes. This triggers the Galaxy to propagate the modifications to all descendant objects that inherit from the modified templates.

Check-in result: "Propagating template changes to 2 descendant objects. SMixer.Level: Successful. Optimizing galaxy database..." — the history configuration has been applied to all instances inheriting from the modified templates (page 6-18)
Check-in progress — the Galaxy database optimization runs after propagation to maintain performance (page 6-18)
Check-in completed successfully — all descendant objects have received the template changes including history configuration (page 6-19)
Propagation in action: When you enable history on a template and check in, the Galaxy identifies every descendant object that inherits from that template and applies the changes. The message "Propagating template changes to 2 descendant objects" confirms the automatic propagation. The "Optimizing galaxy database" step ensures the Galaxy remains performant after the bulk update.

22.5 Deploying to Production

Reference: Lab 12 (pages 6-19 to 6-20)

With the template changes propagated, deploy the updated galaxy to the production platform. This pushes the new history configuration to the running system.

Deployment tree: TrainingGalaxy → ProdPlatform → AppEngine1 → ControlSystem → Line1/Mixer100 containing Agitator_001, Inlet1_001, Level_001, Outlet_001, Pump1_001, Temperature_001. All objects are now running with history enabled (page 6-20)
Deployment verification — confirming all objects under Mixer100 are deployed and running with the updated history configuration (page 6-20)

At this point, the production platform is actively collecting historical data for all attributes that have History enabled. The Store and Forward mechanism on the WinPlatform buffers this data locally before forwarding it to the Historian.

Deployment tip: After deploying, allow a few minutes for data to accumulate before checking Historian Client Web. The first data points will appear once the Historian receives values from the S&F cache.

22.6 Viewing Historical Data with Historian Client Web

Reference: Lab 12 (pages 6-21 to 6-22)

Historian Client Web is the primary tool for viewing and analyzing historical data. Access it through a web browser — typically at http://localhost:32573 or the Historian server's IP address.

Historian Client Web — the main dashboard interface for browsing, trending, and analyzing historical data. Use the Tag Selector to find attributes and create trends (page 6-21)
Historian Client Web — trend view showing historical data for selected tags (page 6-21)

Step 4: Create a Trend for Level_001.PV

Reference: Lab 12 (pages 6-21 to 6-22)

Select the Level_001.PV tag in Historian Client Web and create a trend to verify data collection is working:

Level_001.PV(I) = 63.09 with an upward trend, time range Jan 18, 2023. The (I) notation indicates interpolated values — the Historian calculates these between actual stored data points (page 6-22)
Level_001.PV trend details — the current value and trend direction confirm that historical data is being collected and displayed correctly (page 6-22)

The (I) suffix after the value indicates interpolation — the Historian calculated this value by interpolating between the nearest stored data points. This is normal behavior and confirms the interpolation setting (SystemDefault) is working correctly.

Step 5: Examine Long-Term Trends

Reference: Lab 12 (page 6-22)

Use the time selector in Historian Client Web to zoom out and view long-term trends. This reveals the full history of the tag since collection began:

Long-term trend: Level_001.PV stays flat near 0, then spikes to approximately 100 at 3 PM. The time selector ranges from 30 days (30D) down to 1 hour (1H) — use this to zoom in/out on historical data (page 6-22)
Reading Long-Term Trends: A flat line at 0 followed by a sharp spike typically means:
  • The Historian started collecting data at a specific point in time (when history was enabled)
  • Before that point, no data existed (flat line at 0 or default value)
  • The spike represents the first real values arriving from the process
  • Over time, the trend will show a continuous, meaningful line as more data accumulates
Time Selector Tips:
  • 30D — 30-day view: good for seeing overall patterns and identifying when events occurred
  • 1D — 1-day view: see daily cycles and shifts
  • 1H — 1-hour view: detailed analysis of recent behavior
  • Use the zoom controls or enter a custom time range to focus on specific periods of interest

Quiz — Test Your Understanding

Q1: What is the purpose of the Store and Forward (S&F) cache on the WinPlatform?

Click to reveal answer

The S&F cache buffers historical data on local disk when the Historian is temporarily unreachable. When connectivity is restored, the buffered data is forwarded automatically, guaranteeing no data loss during transient outages.

Q2: What happens if you forget to check the History box on an attribute?

Click to reveal answer

The Historian will not collect any data for that attribute, regardless of other settings. The History checkbox is the master switch — without it enabled, no historical data is stored for that attribute.

Q3: What does the (I) suffix after a value in Historian Client Web mean?

Click to reveal answer

It indicates the value is interpolated — the Historian calculated it by interpolating between the nearest actual stored data points. This is normal behavior, especially when viewing data at time points between stored samples.

Q4: Why should you enable history on the template rather than on individual instances?

Click to reveal answer

Enabling history on the template allows you to configure it once and have the changes propagate automatically to all descendant objects during check-in. Enabling it on instances would require repeating the configuration for each one individually.

Q5: In the ProdPlatform General tab, what is the default Store and Forward path shown in the lab?

Click to reveal answer

C:\S&F — this is the local disk path where the platform buffers historical data. In production, use a dedicated, high-capacity drive with sufficient space for extended Historian outages.

Q6: What does a flat line at 0 followed by a sharp spike in a long-term trend indicate?

Click to reveal answer

It indicates when history collection started — the flat line represents the period before data was being collected (no data stored), and the spike represents the first real values arriving once history was enabled. Over time, the trend becomes a continuous line.

Q7: What does setting Force storage period to 0 ms mean, and when would you use a non-zero value?

Click to reveal answer

Force storage period = 0 ms means on-change storage — data is stored only when the attribute value actually changes. A non-zero value (e.g., 5000 ms) forces periodic storage at that interval regardless of change, which is useful for critical process variables where guaranteed periodic snapshots are needed.

Key Takeaways

  1. The Store and Forward mechanism on the WinPlatform ensures zero data loss by buffering historical data locally before forwarding to the Historian.
  2. History is enabled per-attribute — always check the History box on template attributes for efficient, consistent configuration.
  3. Trend High/Low control the Y-axis display range in Historian Client — they do not filter or truncate data.
  4. Force storage period = 0 means on-change storage; use non-zero values for guaranteed periodic snapshots.
  5. Template propagation during check-in automatically applies history configuration to all descendant objects.
  6. Historian Client Web is the primary tool for viewing trends, with configurable time ranges from 30 days down to 1 hour.
  7. The (I) suffix in trends indicates interpolated values calculated between actual data points.
