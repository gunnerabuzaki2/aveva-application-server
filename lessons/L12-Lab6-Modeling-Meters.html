Lesson 12: Lab 6 — Modeling Meters

Source: AVEVA Application Server 2023 Training Manual, Module 4, Lab 6 (pages 4-17 to 4-30)

Hands-on lab — create derived templates for meter devices, configure attributes, create instances, and deploy them.

12.1 Overview

Reference: Module 4, Lab 6 (pages 4-17 to 4-18)

Lab Objective: In this lab you will create derived templates for meter devices (meters, level transmitters, temperature sensors), configure their PV attributes, create instances, and deploy them to a running System Platform runtime. This lab brings together all the template and attribute concepts from earlier lessons.

The lab follows this sequence:

  1. Create a $Meter derived template from $UserDefined
  2. Configure the PV attribute (Float, I/O Read)
  3. Create specialized templates ($Level, $Temperature) derived from $Meter
  4. Create instances (L1, T1) from the specialized templates
  5. Configure IO device mapping (Autobind)
  6. Deploy and verify at runtime

12.2 Creating the $Meter Derived Template

Reference: Module 4, Lab 6 (pages 4-17 to 4-19)

Step 1: Navigate to Templates

In the IDE, open the Templates pane. You should see the Application folder with the default templates: $AnalogDevice, $DiscreteDevice, $Sequencer, $SQLData, and $UserDefined. The Training > Global folder also contains derived templates from earlier labs.

Templates pane — right-click $UserDefined to create a new Derived Template (page 4-17)

Step 2: Create $Meter Derived Template

Right-click $UserDefined → New → Derived Template. Name it $Meter. This makes $UserDefined the parent template of $Meter, meaning $Meter inherits all attributes from $UserDefined.

Creating the $Meter derived template — enter the name and confirm (page 4-18)
Template Inheritance: When you create a derived template, the child inherits all attributes, scripts, and alarms from the parent. You can override or add attributes in the child. This is the core of template reusability in Application Server.

12.3 Configuring the PV Attribute

Reference: Module 4, Lab 6 (pages 4-19 to 4-20)

Open the $Meter template in the Object Editor. The PV attribute needs to be configured to read a floating-point value from an IO device.

$Meter Object Editor — PV attribute set to Float type with I/O Read source (page 4-19)
PropertyValueWhy
Attribute TypeFloatMeters produce continuous analog values (flow, pressure, etc.)
I/O SourceI/O ReadThe PV reads its value from a field device via an IO device
I/O NameIODev1_...Maps to the IO device and tag for this attribute
Why Float and not Integer? Meter readings are typically continuous values (e.g., 123.456 gallons/min). Float attributes support decimal precision, which is essential for accurate process measurement.

12.4 Check-in and Template Completion

Reference: Module 4, Lab 6 (pages 4-20 to 4-21)

After configuring the PV attribute, check in the $Meter template. Check-in saves the template to the Galaxy database and makes it available for deriving further templates or creating instances.

Check-in confirmation — the $Meter template is now saved in the Galaxy (page 4-20)
Common Mistake: Forgetting to check in after making changes. Unchecked-in templates exist only in your IDE session. Other users cannot see them, and deployment cannot use them. Always check in your work.

12.5 Creating Specialized Templates ($Level, $Temperature)

Reference: Module 4, Lab 6 (pages 4-21 to 4-23)

Now create two specialized templates derived from $Meter. These represent specific types of metering instruments.

Creating $Level and $Temperature as derived templates from $Meter (page 4-21)
TemplateDerived FromPurpose
$Level$MeterLevel transmitter — measures liquid level in a tank
$Temperature$MeterTemperature sensor — measures process temperature
Template Hierarchy:
$UserDefined → $Meter → $Level / $Temperature

Each level inherits from its parent. $Level and $Temperature both get the PV attribute from $Meter, which got it from $UserDefined. You can add template-specific attributes (e.g., engineering units) in the child templates.

12.6 Creating Instances

Reference: Module 4, Lab 6 (pages 4-23 to 4-25)

Instances are the actual runtime objects that represent real equipment. Create:

  • L1 — an instance of $Level (representing a real level transmitter)
  • T1 — an instance of $Temperature (representing a real temperature sensor)
Creating instances — L1 from $Level and T1 from $Temperature (page 4-23)

Instances are created in the Application folder (or a subfolder). Each instance has a unique name and its own attribute values, but shares the template's structure (attributes, scripts, alarms).

InstanceTemplatePV Value (example)Represents

12.7 IO Device Mapping (Autobind)

Reference: Module 4, Lab 6 (pages 4-25 to 4-27)

The IO Devices view is where you map template attributes to real field device tags. Application Server uses Autobind to automatically match attribute names to IO tags.

IO Devices view — showing configured IO devices and tag mappings (page 4-25)
IO device mapping — Autobind configuration connecting PV attributes to field tags (page 4-26)
Autobind: When you configure an IO Source on an attribute (e.g., IODev1_L1_PV), Autobind automatically resolves the tag path at deployment time. This means you don't have to manually map each attribute — the naming convention handles it.

How Autobind Works

  1. The attribute's IO Name follows a pattern: <IODevice>_<Instance>_<Attribute>
  2. At deployment, the runtime resolves the full tag path using the IO device configuration
  3. The PV attribute then reads its value from the resolved tag in real-time

12.8 Deploying to Runtime

Reference: Module 4, Lab 6 (pages 4-27 to 4-29)

With templates, instances, and IO mapping configured, the final step is deployment. Deployment pushes your galaxy objects to the Application Server runtime.

Deploy dialog — selecting objects and target runtime for deployment (page 4-27)
Deployment in progress / completion — objects are now running on the runtime engine (page 4-28)
Deployment Prerequisite: All templates and instances must be checked in before deployment. If you see errors during deployment, verify that your work is checked in and there are no attribute configuration errors.

Deployment Steps

  1. Select the objects to deploy (instances L1, T1)
  2. Choose the target runtime (Application Server node)
  3. Confirm the deployment
  4. Wait for deployment to complete — the runtime engine instantiates the objects

12.9 Verifying at Runtime

Reference: Module 4, Lab 6 (pages 4-29 to 4-30)

After deployment, use the Object Viewer to verify that your instances are running and reading live values.

Object Viewer — L1 and T1 instances showing live PV values from field devices (page 4-29)
Monitoring attributes — verifying that PV values update in real-time (page 4-30)
What to Verify:
  • Instances appear in the Object Viewer with status "Running"
  • PV values are updating (not stuck at 0 or stale)
  • IO quality is "Good" (not "Bad" or "Uncertain")
  • Attribute values match what you expect from the field devices

Knowledge Check

Q1: What is the parent template of $Meter in this lab?

Click to reveal answer

$UserDefined — $Meter is a derived template with $UserDefined as its parent.

Q2: Why is the PV attribute configured as Float rather than Integer?

Click to reveal answer

Meter readings are continuous analog values (e.g., 123.456) that require decimal precision. Float supports fractional values; Integer does not.

Q3: What is the relationship between $Meter, $Level, and $Temperature?

Click to reveal answer

$Level and $Temperature are both derived from $Meter. They form a three-level hierarchy: $UserDefined → $Meter → $Level/$Temperature. Both $Level and $Temperature inherit the PV attribute from $Meter.

Q4: What must you do before deploying objects to the runtime?

Click to reveal answer

You must check in all templates and instances. Unchecked-in objects exist only in the IDE session and cannot be deployed.

Q5: What does Autobind do in the IO device configuration?

Click to reveal answer

Autobind automatically resolves attribute IO names (following a naming convention) to actual field device tags at deployment time, eliminating the need for manual tag mapping.
