Lesson 7: Lab 3 — Creating Plant and Deployment Models

Source: AVEVA Application Server 2023 Training Manual, Module 3, Lab 3 (pages 3-13 to 3-28)

Hands-on lab — building the Plant Model hierarchy and Deployment Model, then deploying and verifying the application.

7.1 Lab Introduction and Objectives

Reference: Module 3, Lab 3 (page 3-13)

Lab Objectives:
  • Create the Plant Model — an area hierarchy using $Area instances in the Model view that represents the physical plant layout
  • Create the Deployment Model — platform and engine objects using $WinPlatform and $AppEngine in the Deployment view that represent the computing infrastructure
  • Deploy the application objects to the target platform and verify successful deployment
Plant Model vs. Deployment Model: The Plant Model (built in the Model view) represents where things are in your physical plant — areas, production lines, and equipment grouped into an area hierarchy. The Deployment Model (built in the Deployment view) represents what computer and engine will run those objects. Every object created from a template must eventually be assigned to a deployment object (hosted under an engine on a platform) to run at runtime.
Lab 3 overview — objectives and the starting point for building both models (Module 3, page 3-13)

This lab assumes you have already completed Lessons 4–6: you have a Galaxy, imported templates, and are ready to build the actual plant and deployment structure.

7.2 Creating the Plant Model

Reference: Module 3, Lab 3 (pages 3-14 to 3-17)

The Plant Model is a hierarchy of $Area objects (derived from the SgArea template) that mirrors your physical plant layout. It acts as the organizational backbone — all equipment objects live inside areas. You create instances in the Model view using the Galaxy Browser.

Step 1: Switch to the Model View

In the IDE, click the Model tab at the bottom of the Galaxy Browser. The Model view shows the hierarchical tree of objects. At the top you will see the Galaxy root node (named after your Galaxy — for example, TrainingGalaxy).

Step 2: Create Area Instances from the SgArea Template

To create your first area, you instantiate the SgArea template:

  1. In the Templates view, right-click the SgArea template.
  2. Select New → Instance.
  3. The new $Area instance appears in the Model view.
  4. Name the instance appropriately (for example, Production for a top-level production area).
Right-clicking the SgArea template → New → Instance to create area objects in the Model view (Module 3, page 3-15)

Step 3: Build the Area Hierarchy

Repeat the instantiation process to create all the areas that represent your plant layout. Nest sub-areas inside parent areas by creating them under the appropriate parent. The resulting Plant Model hierarchy for this lab includes:

Area NameLevelPurpose
PlantRootTop-level area encompassing the entire plant
ProductionLevel 1Production operations area
Line1Level 2Production Line 1 — selected in the Model view
Line2Level 2Production Line 2
PackagingLevel 1Packaging area
ReceivingLevel 1Material receiving area
ShippingLevel 1Shipping and dispatch area
ControlSystemLevel 1Control system infrastructure area
Completed Plant Model hierarchy in the Model view — Plant > Production > Line1 is selected, showing ControlSystem, Line2, Packaging, Receiving, and Shipping as sibling areas (Module 3, page 3-17)
Tip: The Plant Model naming should follow a consistent convention that matches your P&ID or plant documentation. This makes it intuitive for operators and engineers to navigate the object hierarchy at runtime.
Watch out: Creating $Area instances in the Model view builds the organizational hierarchy, but areas are purely structural — they do not run on their own. Equipment objects placed inside areas must still be assigned to a deployment object (an engine on a platform) to execute at runtime.

7.3 Creating the Deployment Model

Reference: Module 3, Lab 3 (pages 3-18 to 3-23)

The Deployment Model defines the computing infrastructure using two key object types: a $WinPlatform (representing a physical or virtual computer) and one or more $AppEngine objects (representing application engines that execute object logic on that platform). You build this in the Deployment view.

Step 1: Examine the Deployment View

Switch to the Deployment view in the Galaxy Browser. Initially, you will see:

  • TrainingGalaxy — the root Galaxy node
  • Unassigned Host — a container for objects that have not yet been assigned to a platform/engine
  • GRPlatform — the Galaxy Repository platform (already created as part of the Galaxy)
  • ProdPlatform — a $WinPlatform representing the target production computer (if already created, or you create it now)

All your Plant Model $Area instances initially appear under Unassigned Host because they have not yet been assigned to run on any engine:

ObjectLocation in Deployment View
ControlSystem, Line1, Line2, Packaging, Plant, Production, Receiving, ShippingUnder Unassigned Host
GRPlatformUnder Unassigned Host (Galaxy Repository platform)
ProdPlatformStandalone $WinPlatform — the target deployment computer
Deployment view showing TrainingGalaxy with Unassigned Host (all area objects waiting to be assigned) and ProdPlatform (the target deployment computer) (Module 3, page 3-21)

Step 2: Create the $AppEngine Object

To provide an engine that will run the application objects:

  1. In the Deployment view, right-click the ProdPlatform object.
  2. Select Insert Object → $AppEngine.
  3. Name it AppEngine1.
  4. Click Create.
  5. The AppEngine1 now appears nested under ProdPlatform in the Deployment view.
Hosting chain: The deployment hierarchy follows: $WinPlatform → $AppEngine → equipment/area objects. The WinPlatform represents the physical computer, the AppEngine is the process that executes object logic, and the objects hosted under the engine are what actually run at runtime.

Step 3: Assign Objects to the Engine

With AppEngine1 created under ProdPlatform, you now assign your area and equipment objects to the engine so they know where to run:

  1. In the Deployment view, drag area objects from Unassigned Host onto AppEngine1.
  2. Alternatively, right-click an object under Unassigned Host and use the Assign to option to move it under the engine.
  3. Repeat for all objects that need to be deployed.
AppEngine1 created under ProdPlatform in the Deployment view — objects are being assigned from Unassigned Host to the engine (Module 3, page 3-23)
Important: The $WinPlatform Node Name must exactly match the network hostname of the target computer. If it does not match, the Galaxy Repository cannot connect to the target platform and deployment will fail.

7.5 Deploying Objects to Runtime

Reference: Module 3, Lab 3 (pages 3-26 to 3-28)

Deployment is the process of pushing object configurations from the Galaxy Repository (GR) to the target platform's runtime. After deployment, objects exist both in the Galaxy database AND as running instances on the target computer.

Step 1: Initiate Deployment

  1. In the Deployment view, right-click the ProdPlatform object.
  2. Select Deploy... from the context menu.
  3. The Deploying Objects dialog appears.
Right-clicking ProdPlatform → Deploy... to initiate deployment of the platform and all hosted objects (Module 3, page 3-27)

Step 2: Review and Confirm Deployment

The Deploy dialog lists all objects that will be deployed:

ColumnMeaning
Object NameThe full path of each object being deployed
Object TypeThe template type (e.g., $WinPlatform, $AppEngine, $Area)
StatusCurrent deployment state — Deploying, Deployed, OnScan, or Error

Step 3: Monitor Deployment Status

During deployment, the dialog shows real-time progress. Objects transition through these states:

IndicatorMeaning
DeployingObject is being pushed to the target platform
OnScanObject is deployed and actively scanning (running normally)
ErrorDeployment failed — check the Output pane for details
Deploy complete — 10 objects deployed (ProdPlatform, AppEngine1, and 8 area objects), all showing OnScan status. SUCCESS message confirms successful deployment (Module 3, page 3-28)
Deployment summary for this lab: 10 objects were deployed in total — ProdPlatform, AppEngine1, and 8 area objects (Plant, Production, Line1, Line2, Packaging, Receiving, Shipping, ControlSystem). All objects reached the OnScan state, meaning they are deployed and running successfully.
Common error: If deployment fails with a node name mismatch, verify that the $WinPlatform Node Name exactly matches the target computer's network hostname. Case sensitivity matters!

Q1: What is the primary purpose of the Plant Model in Application Server?

Q2: What two key object types make up the Deployment Model?

$WinPlatform (representing a physical or virtual computer) and $AppEngine (representing an application engine that runs object logic on that platform). Together they form the hosting chain: $WinPlatform → $AppEngine → hosted objects.

Right-click the $WinPlatform object (e.g., ProdPlatform) and select Deploy... from the context menu. The Deploying Objects dialog opens, listing all objects to be deployed.

Key Takeaways

  1. The Plant Model uses nested $Area instances (from the SgArea template) to mirror the physical plant layout — created in the Model view.
  2. The Deployment Model uses $WinPlatform (computer) and $AppEngine (engine) objects — created in the Deployment view.
  3. Newly created objects initially appear under Unassigned Host and must be explicitly assigned to an engine.
  4. Deployment pushes configurations from the Galaxy Repository to the target platform by right-clicking the platform and selecting Deploy.
  5. After deployment, all objects should reach the OnScan state — confirming they are running successfully.
  6. The $WinPlatform Node Name must match the target computer's hostname exactly.
