Lesson 17: Connecting Application Objects to Field Data + Lab 10-11

Source: AVEVA Application Server 2023 Training Manual, Module 5, Section 3 + Labs 10-11 (pages 5-13 to 5-34)

The critical link — binding your Galaxy objects to real-world I/O through OI Servers, deploying to the runtime engine, and verifying data flow.

17.1 Connecting Attributes to I/O — The Big Picture

Reference: Module 5, Section 3 (pages 5-13 to 5-20)

Up to this point, your automation objects have been working with internal (simulated) values. In a real plant, those values need to come from — and be written back to — actual field devices: PLCs, RTUs, and sensors. This is where attribute I/O binding comes in.

I/O Binding is the process of linking an object's attribute (e.g., Level1.PV) to a real data point on a PLC or other field device, routed through an OI Server. Once bound, the attribute no longer holds a hardcoded simulation value — it reflects the live state of the physical process.

The data flow works like this:

StepComponentWhat Happens
1 Field Device (PLC/RTU) The physical sensor or actuator produces a value (e.g., a pressure reading of 45.2 PSI)
2 OI Server (Communication Driver) Polls the field device over its native protocol (Modbus, EtherNet/IP, OPC, etc.) and holds the value in a topic-level tag
3 Application Server (Galaxy) The automation object's attribute is bound to the OI Server tag. Application Server reads/writes the value in real time.
4 HMI Client (InTouch/OMI) The operator screen displays the live value and can send commands (setpoints, alarms acknowledged, etc.)

How Binding Works in the IDE

In the Galaxy IDE, you configure I/O binding by editing the attribute's I/O Source property. The I/O source string follows a specific syntax that tells Application Server which OI Server, which topic, and which tag to connect to.

I/O Source Syntax: TopicName.TagName

For example, if your OI Server has a topic called Topic1 and the PLC tag is Agitator_001, the I/O source would be:
Topic1.Agitator_001

You simply type this into the I/O source field for whichever attribute needs the live data.

Object Types and I/O Binding

Different object types use I/O binding in different ways:

Object TypeCommonly Bound AttributesTypical Use
Discrete PV (Process Value), SP (Setpoint) On/off states: pump running, valve open/closed, motor on/off
Analog PV, SP, SPH (Setpoint High), SPL (Setpoint Low) Continuous values: temperature, pressure, flow, level
Motor PV (Status), SP (Command), AHSP (Auto High Setpoint) Motors with start/stop and status feedback
Pump PV, SP, RunCmd Pumps with run command and status
Tip — Simulation vs. I/O: Before you bind an attribute to a real I/O source, it is running in simulation mode (using an internal expression or hardcoded value). This is extremely useful during development — you can build and test your entire application without any hardware. When you are ready to go live, you simply change the I/O source from the simulation expression to the real PLC tag address.

Reference: Module 5, Section 3 (pages 5-20 to 5-25)

When you connected to the OI Server in Lesson 16 (Lab 9), you established a communication link between Application Server and the field devices. Now, let's look at how that link is structured within the Galaxy.

In the IO Devices view, you can see the full communication hierarchy:

LevelName in Our GalaxyWhat It Represents

17.3 Lab 10: Deploying and Undeploying Objects

Reference: Module 5, Lab 10 (pages 5-25 to 5-30)

Lab 10 — Deploying Your Galaxy Objects

Objective: Understand the deploy/undeploy lifecycle and practice deploying and undeploying automation objects.

What Does "Deploy" Mean?

In Application Server, deployment is the process of taking your objects from the development environment (the Galaxy IDE) and pushing them to the runtime engine on the target deployment node(s). Until an object is deployed, it exists only in the Galaxy database — it is not executing, not communicating with field devices, and not producing alarms or historian data.

Deploy vs. Save:
  • Save = writes your changes to the Galaxy database (the development copy)
  • Deploy = pushes the saved objects to the runtime nodes where they actually execute
You can save many times without deploying. But until you deploy, nothing changes in the running system.

Deploying Objects

To deploy objects in the Galaxy IDE:

  1. In the Model View or Object Browser, select the object(s) you want to deploy
  2. Right-click and select Deploy
  3. The deployment wizard will show you which nodes will receive the objects
  4. Confirm the deployment
  5. Application Server compiles, validates, and pushes the objects to the target nodes
Tip — Partial Deployment: You don't have to deploy everything at once. You can deploy individual objects, entire areas, or the whole Galaxy. This is useful for making incremental changes without disrupting the entire running system.

Undeploying Objects

Undeployment is the reverse — removing objects from the runtime engine and returning them to development-only status. You might undeploy objects when:

  • An object is no longer needed in the running system
  • You need to make major changes and want a clean slate
  • Commissioning is complete and you're cleaning up test objects
Right-clicking Mixer100 in the Model-Tagname view shows the Undeploy option highlighted in the context menu (Module 5, page 5-28)

To undeploy an object:

  1. In the Model View, locate the object you want to undeploy
  2. Right-click the object (e.g., Mixer100)
  3. Select Undeploy from the context menu
  4. Confirm the undeployment
  5. The object is removed from all runtime nodes and returns to development-only status
Undeploy completion dialog: "Undeploy complete. 13 objects undeployed successfully." — the objects are now removed from the runtime engine (Module 5, page 5-29)
The Undeploy Objects dialog after completion: "Undeploy complete. 11 automation objects + 2 area objects offscan from Production and AppEngine1. SUCCESS" — showing that both the automation objects and their containing area objects are undeployed (Module 5, page 5-30)
⚠ Pitfall — Undeploying Area Objects: Notice in the screenshot that undeploying automation objects also undeploys the area objects that contain them (2 area objects in this case). Be careful — undeploying a parent area may remove more objects than you intended. Always review the list before confirming.

Lab 10 — Practice Steps

Step 1: In the Galaxy IDE, navigate to Model View and locate your Mixer100 object.

Step 2: Right-click Mixer100 and select Deploy. Watch the deployment wizard and confirm the target nodes.

Step 3: Verify the deployment by opening the Object Viewer (F12) and confirming the object's attributes are now live (not in simulation).

Step 4: Now right-click Mixer100 again and select Undeploy. Confirm the undeployment.

Step 5: Review the undeploy completion dialog. Note how many automation objects and area objects were removed.

Step 6: Re-deploy Mixer100 so it is available for Lab 11.

17.4 Lab 11: Modifying I/O Values and Verifying Data Flow

Reference: Module 5, Lab 11 (pages 5-30 to 5-34)

Lab 11 — Verifying Field Data Connectivity

Objective: Modify I/O values through the Galaxy IDE and verify that data flows correctly between the Galaxy objects and the OI Server (simulated field devices).

Modifying Numeric Values

Once your objects are deployed and connected to an OI Server, you can modify their values directly from the Galaxy IDE. This is useful for testing — you can write setpoint values to see if your logic responds correctly.

The Modify Numeric Value dialog: setting Agitator_001.Speed.SP to 55. The Cancel/Apply/OK buttons allow you to commit or discard the change (Module 5, page 5-32)

To modify a numeric attribute value:

  1. In the Object Browser or Object Viewer, locate the attribute you want to change
  2. Double-click the value (or right-click and select Modify Value)
  3. The Modify Numeric Value dialog opens
  4. Enter the new value (e.g., 55 for Agitator_001.Speed.SP)
  5. Click Apply to write the value without closing the dialog, or OK to write and close
  6. The value is sent through the OI Server to the field device (or simulator)
Apply vs. OK:
  • Apply — writes the value immediately but keeps the dialog open so you can make further changes
  • OK — writes the value and closes the dialog
  • Cancel — closes the dialog without writing any changes

Verifying Data Flow

After modifying a value, verify the data flow end-to-end:

  1. Write a value — Use the Modify Numeric Value dialog to write a new setpoint (e.g., Speed.SP = 55)
  2. Check the Object Viewer — The attribute should reflect the new value. If the attribute is bound to I/O, the value will also be visible in the OI Server's tag database
  3. Check the Simulator/PLC — If using the Simulator driver, you can verify the value appears in the simulator's tag list. If using a real PLC, use the PLC programming software to confirm
  4. Check the HMI — If you have InTouch or OMI connected, the operator screen should display the updated value
Tip — Bidirectional Communication: I/O binding is bidirectional. If the PLC changes a value (e.g., a sensor reading changes), Application Server will pick up the new value automatically. And if you write a setpoint from Application Server, the PLC receives it. This two-way flow is the foundation of industrial automation control.

Lab 11 — Practice Steps

Step 1: Ensure your objects are deployed and connected to the OI Server (from Lab 10, Step 6).

Step 2: Open the Object Browser and navigate to the IO Devices view. Confirm that your OI Server topics are visible and tags are populated.

Step 3: In the Model View, find an analog object (e.g., a level or temperature object) and open its attributes.

Step 4: Double-click on the SP (Setpoint) attribute to open the Modify Numeric Value dialog.

Step 5: Enter a test value (e.g., 55) and click Apply. The dialog stays open.

Step 6: Verify the value was written — check the Object Viewer and the OI Server tag list.

Step 7: Change the value again (e.g., to 75) and click OK to close the dialog.

Step 8: Open the Object Viewer (F12) and confirm the attribute shows the updated value and is no longer in simulation mode.

Step 9: If using the Simulator, check the simulator window to confirm the tag value matches what you wrote.

Step 10: Document your observations — what was the old value, what did you change it to, and did the change propagate correctly?

17.5 Deploy/Undeploy Lifecycle Summary

Reference: Module 5, Section 3 (pages 5-30 to 5-34)

Understanding the full deploy/undeploy lifecycle is critical for managing a production Galaxy:

StateWhat It MeansData Flow
Development (Undeployed) Object exists only in the Galaxy database. You can edit, save, and modify it freely. No data flow — attributes use simulation expressions or hardcoded values only.
Deployed (Running) Object is pushed to the runtime engine. It is executing logic, communicating with I/O, and generating alarms. Full data flow — attributes read from and write to field devices via OI Servers.
Off-Scan (Undeployed) Object has been undeployed and is no longer running. It returns to development-only status. Data flow stops. Attributes revert to simulation/internal values.
Key Lifecycle Rules:
  • You must save before deploying — deploy pushes the saved version, not unsaved changes
  • Deploying an object that was already deployed updates it (hot deployment) — the running system picks up the changes without stopping
  • Undeploying removes the object from runtime but does not delete it from the Galaxy
  • You can undeploy individual objects, areas, or the entire Galaxy
  • Undeploying an area undeploys all objects within it

17.6 Key Terms

TermDefinition
I/O Binding The process of linking an automation object attribute to a real-world data point through an OI Server
I/O Source The string that identifies which OI Server topic and tag an attribute is connected to (e.g., Topic1.Agitator_001)
Deploy Pushing objects from the Galaxy IDE to the runtime engine for execution
Undeploy Removing objects from the runtime engine, returning them to development-only status
Hot Deployment Updating a running object with new configuration without stopping the runtime engine
Topic A configured communication channel within an OI Server that connects to a specific PLC or device
Off-Scan The state of an object that has been undeployed — it is no longer executing in the runtime
Simulation Mode When an attribute uses an internal expression or hardcoded value instead of live I/O data

17.7 Quiz

Question 1

What is the difference between saving an object and deploying it?

Click to reveal answer

Saving writes your changes to the Galaxy database (the development copy). Deploying pushes the saved objects to the runtime engine where they actually execute, communicate with field devices, and produce alarms and historian data. You can save many times without deploying — changes only take effect in the running system when you deploy.

Question 2

If you undeploy a parent area object that contains three automation objects, what happens to the child objects?

Click to reveal answer

All three child automation objects are also undeployed. Undeploying a parent area undeploys everything within it. This is why the completion dialog showed "11 automation objects + 2 area objects" — both the containers and their contents were removed from the runtime. Always review the undeploy list carefully before confirming.

Question 3

What is the I/O source syntax for connecting an attribute to a tag called Flow1 on Topic2?

Click to reveal answer

Topic2.Flow1

The I/O source string follows the format TopicName.TagName. Application Server uses this string to locate the correct OI Server, topic, and tag for the data binding.

Question 4

When you click Apply in the Modify Numeric Value dialog versus clicking OK, what is the difference?

Click to reveal answer

Apply writes the value immediately to the attribute (and through I/O binding to the field device) but keeps the dialog open so you can make further changes. OK writes the value and closes the dialog. Cancel closes the dialog without writing any changes.

Question 5

Before you deploy an object, where does it exist? After you deploy it, where does it execute?

Click to reveal answer

Before deployment, the object exists only in the Galaxy database (the development environment). It is not executing and has no data flow. After deployment, the object is pushed to the runtime engine on the target deployment node(s), where it executes logic, communicates with field devices through OI Servers, and produces alarms and historian data.
