Section 2 — Variables and Control Statements

Source: AVEVA Application Server 2023 Training Manual, Module 11, Section 2

This section describes the usage of variables and control statements in a script. (pages 11-25 to 11-32)

32.3 QuickScript.NET Variables

Reference: QuickScript .NET Variables (pages 11-25 to 11-26)

Variables in QuickScript.NET must be declared before use. They can appear on both the left and right sides of statements and expressions.

Variable Scope — Two Types:
  • Script body variables: Declared within the script body. They lose their value after the script executes and cannot be accessed by other scripts.
  • Declarations area variables: Declared in the Declarations area of the Script tab. They maintain their values throughout the lifetime of the associated object and persist between script executions.

DIM Statement Syntax

Each variable must be declared with a separate DIM statement followed by a semicolon:

DIM <variable_name> [ ( <upper_bound>
  [, <upper_bound> [, <upper_bound>]] ) ]
  [ AS <data_type> ];
ElementDescription
DIMRequired keyword
<variable_name>Must begin with a letter (A–Z or a–z); remaining characters can be letters, digits (0–9), or underscores (_). Limited to 255 Unicode characters. Must not contain dots.
<upper_bound>A number between 1 and 2,147,483,647 for array dimensions. Up to 3 dimensions supported. Lower bound is always 1.
ASOptional keyword for declaring the data type.
<data_type>One of: Boolean, Integer, ElapsedTime, Float, Double, String, Time, or InternationalizedString. Also supports .NET types and imported script library types.
Default Data Type: If you omit the AS clause, the variable defaults to Integer. For example: 
DIM LocVar1;
' is equivalent to:
DIM LocVar1 AS Integer;

DIM Statement Rules

Important Rules:
  • DIM statements can appear anywhere in the script body, but they must precede the first referencing statement or expression.
  • Do not cascade DIM statements — each variable needs its own DIM statement: 
    ' INVALID:
DIM LocVar1 AS Integer, LocVar2 AS Float;
DIM LocVar3, LocVar4, LocVar5 AS String;

' VALID:
DIM LocVar1 AS Integer;
DIM LocVar2 AS Float;
DIM LocVar3 AS String;
  • Variable names and data type identifiers are not case sensitive.
  • If a variable name conflicts with an attribute name, alias, or object name, the variable name takes precedence.

Array Variables

Arrays are declared with dimensions in the DIM statement. The syntax for referencing an entire array is [ ]:

DIM locarr[10] AS Float;
' Assign attribute array to local array:
locarr[] = tag.attr[];

The lower bound of each array dimension is always 1. After declaration, the upper bound is fixed — ReDim is not supported.

Quality Propagation with Variables

When used on the right side of an equation, declared local variables always cause the left side to have Good quality:

DIM x AS Integer;
DIM y AS Integer;
x = 5;
y = 5;
me.attr = 5;      ' Quality: Good
me.attr = x;      ' Quality: Good
me.attr = x + y;  ' Quality: Good

Null Values

Use null to indicate no object is currently assigned to a variable. This is equivalent to C#'s null or VB's Nothing. Assigning null makes the variable eligible for garbage collection.

Warning: You may not use a variable whose value is null — doing so terminates the script with an error. However, you can test a variable for null: 
IF myvar == null THEN ...

Passing Attributes as Parameters

It is not possible to pass attributes directly as parameters for system objects. Use a local variable as an intermediary, or explicitly convert the attribute to a string using an appropriate function call.

32.4 Control Structures

Reference: QuickScript .NET Control Structures (pages 11-27 to 11-32)

QuickScript.NET provides five primary control structures:

  1. IF … THEN … ELSEIF … ELSE … ENDIF
  2. FOR … TO … STEP … NEXT Loop
  3. FOR EACH … IN … NEXT
  4. TRY … CATCH
  5. WHILE Loop

32.4.1 IF … THEN … ELSEIF … ELSE … ENDIF

Reference: IF-THEN-ELSE-ENDIF (pages 11-27 to 11-28)

Conditionally executes various instructions based on the state of an expression:

IF <Boolean_expression> THEN;
  [statements];
  [ { ELSEIF;
    [statements] } ];
  [ ELSE;
    [statements] ];
ENDIF;

Boolean Expression Evaluation

Depending on the data type, expressions are evaluated as True or False:

Data TypeMapping
BooleanDirectly used (no mapping needed)
IntegerValue == 0 → False; Value <> 0 → True
FloatValue == 0 → False; Value <> 0 → True
DoubleValue == 0 → False; Value <> 0 → True
StringCannot be mapped — results in validation error
TimeCannot be mapped — results in validation error
ElapsedTimeCannot be mapped — results in validation error
InternationalizedStringCannot be mapped — results in validation error
BigStringCannot be mapped — results in validation error

IF/THEN Example

IF value == 0 THEN
  Message = "Value is zero";
ELSEIF value > 0 THEN;
  Message = "Value is positive";
ELSEIF value < 0 THEN;
  Message = "Value is negative";
ELSE;
  {Default. Should never occur in this example};
ENDIF;

Nested IF Statements

You can nest IF statements, but each nesting requires an additional ENDIF:

IF <Boolean_expression> THEN;
  [statements];
  IF <Boolean_expression> THEN;
    [statements];
  ENDIF;
ENDIF;

IF/THEN and Attribute Quality

Quality Handling in IF Statements:
  • When copying an Attribute value to another Attribute of the same type, the quality is also copied.
  • If an Attribute has BAD quality and you need to copy only the value, use a temporary variable: 
    DIM temp AS Integer;
temp = me.Attribute1;
me.Attribute2 = temp;
  • If a comparison involves an Attribute with BAD quality, the statements within the IF block are not executed.
  • Always check quality first using IsBad(): 
    IF (IsBad(me.Attribute1)) THEN
  LogMessage("Attribute1 quality is bad");
ELSE
  IF me.Attribute1 <> me.Attribute2 THEN
    me.Attribute2 = me.Attribute1;
  ENDIF;
ENDIF;

32.4.2 FOR … TO … STEP … NEXT Loop

Reference: FOR-NEXT Loop (page 11-29)

Performs a function (or set of functions) multiple times during a single script execution:

FOR <analog_var> = <start_expression> TO <end_expression>
  [STEP <change_expression>];
  [statements];
  [EXIT FOR;];
  [statements];
NEXT;
ElementDescription
analog_varA variable of type Integer, Float, or Double
start_expressionInitial value for the loop variable
end_expressionTermination condition — loop exits when analog_var exceeds this value (positive step) or falls below it (negative step)
change_expressionIncrement/decrement value (can be positive or negative). Defaults to +1 or −1 if STEP is omitted.

FOR Loop Execution Flow

  1. analog_var is set to start_expression
  2. System tests if analog_var exceeds end_expression (positive step) or falls below it (negative step). If so, exit loop.
  3. Statements in the loop body execute (can exit via EXIT FOR)
  4. analog_var is incremented by the step value
  5. Steps 2–4 repeat
Note: FOR-NEXT loops can be nested. The number of nesting levels depends on memory and resource availability.

32.4.3 FOR EACH … IN … NEXT

Reference: FOR EACH Loop (page 11-30)

Used with collections exposed by OLE Automation servers:

FOR EACH <object_variable> IN <collection_object>
  [statements];
  [EXIT FOR;];
  [statements];
NEXT;

Where object_variable is a dimmed variable and collection_object is a variable holding a collection object. You can exit the loop with EXIT FOR.

32.4.4 TRY … CATCH

Reference: TRY-CATCH (pages 11-30 to 11-31)

Provides error handling — allows scripts to continue running rather than terminating when errors occur:

TRY
  [try statements]  ' guarded section
CATCH
  [catch statements]
ENDTRY;

TRY-CATCH Execution Flow

  1. Runtime error handling starts with TRY. Put code that might fail in the try block.
  2. If no error occurs, the script runs normally — catch block is ignored.
  3. If a runtime error occurs, the rest of the try block is skipped, and execution jumps to the CATCH statement.
The ERROR variable: Statements inside the Catch block may reference the reserved ERROR variable, which is a .NET System.Exception object thrown from the Try block. You can call GetType() to determine the exception type.

TRY-CATCH Example — SQL Database Query

DIM command = new System.Data.SqlClient.SqlCommand;
DIM reader AS System.Data.SqlClient.SqlDataReader;
command.Connection = new System.Data.SqlClient.SqlConnection;
TRY
  command.Connection.ConnectionString = "Integrated Security=SSPI";
  command.CommandText = "select * from sys.databases";
  command.Connection.Open();
  reader = command.ExecuteReader();
  WHILE reader.Read()
    me.name = reader.GetString(0);
    LogMessage(me.name);
  ENDWHILE;
CATCH
  LogMessage(error);
ENDTRY;
IF reader <> null AND NOT reader.IsClosed THEN
  reader.Close();
ENDIF;
IF command.Connection.State == System.Data.ConnectionState.Open THEN
  command.Connection.Close();
ENDIF;

32.4.5 WHILE Loop

Reference: WHILE Loop (pages 11-31 to 11-32)

Performs a function or set of functions repeatedly while a condition is true:

WHILE <Boolean_expression>
  [statements]
  [EXIT WHILE;]
  [statements]
ENDWHILE;

WHILE Loop Execution Flow

  1. Evaluate the Boolean expression. If false, exit the loop and continue after ENDWHILE.
  2. Execute the statements in the loop body (can exit via EXIT WHILE).
  3. Repeat steps 1–2.
Note: WHILE loops can be nested. The number of nesting levels depends on memory and resource availability.

32.5 Quality of Input/Output Extensions

Reference: Quality of Input, InputOutput, and Output Extensions (page 11-32)

Extended Attribute Quality Behavior:
  • OnScan: The value and quality of the Input extended attribute mirrors the quality of the externally referenced attribute for successful reads. Data quality is set to Bad when reads fail due to communication errors or datatype conversion failures.
  • External set while OnScan: If a user changes the value or quality of an extended attribute, a write to the destination occurs during the next execute phase. Quality must be Good or Uncertain for a write to occur.
  • OffScan: Quality is always Bad and user sets are accepted.

32.15 Summary — Variables and Control Statements

Key Takeaways from This Lesson:
  • Variables must be declared with DIM before use; each variable needs its own DIM statement
  • Declarations area variables persist across script executions; script body variables do not
  • 8 data types: Boolean, Integer, ElapsedTime, Float, Double, String, Time, InternationalizedString
  • Arrays support up to 3 dimensions with lower bound always 1
  • IF/THEN/ELSEIF/ELSE/ENDIF for conditional logic — Boolean, Integer, Float, and Double can be used in conditions; String, Time, ElapsedTime cannot
  • FOR/NEXT for counted loops with optional STEP; FOR EACH for collection iteration
  • TRY/CATCH for error handling with the ERROR exception object
  • WHILE/ENDWHILE for condition-based loops
  • Quality propagation: Variables on the right side of assignments produce Good quality on the left side; always check quality with IsBad() before comparisons
  • Lab 22 demonstrated array attributes and IF/THEN logic for valve state tracking
  • Lab 23 demonstrated custom alarm attributes with script-based alarm condition monitoring

📝 Knowledge Check — Lesson 32

  1. What is the difference between declaring a variable in the Declarations area vs. in the script body?
  2. What happens if you omit the AS clause in a DIM statement?
  3. Why can't you use a String expression as the Boolean_expression in an IF/THEN statement?
  4. What is the purpose of the EXIT FOR statement in a FOR/NEXT loop?
  5. In Lab 23, what two conditions must both be true to trigger the Flow.Alarm.Pump1 alarm?
  6. How does the PVStates array in Lab 22 make the valve state script more maintainable?
Show Answers
  1. Variables in the Declarations area persist throughout the object's lifetime and between script executions. Variables in the script body lose their value after the script executes and cannot be accessed by other scripts.
  2. The variable is declared as an Integer by default.
  3. Strings cannot be mapped to Boolean — using a String expression as a Boolean_expression results in a script validation error.
  4. EXIT FOR allows you to exit the loop early from within the loop body, jumping to the statement immediately after NEXT.
  5. The inlet valve must be closed (Inlet1.StateIndex == 1) AND the pump must be running (Pump1.PV is True).
  6. The array holds state names ("Open", "Closed", "Traveling") so the script uses index-based lookup instead of hardcoding strings. State names can be updated without modifying the script logic.
