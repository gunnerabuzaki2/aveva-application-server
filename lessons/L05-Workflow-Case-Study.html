Lesson 5: Module 2 — Application Planning + Case Study

Source: AVEVA Application Server Training Manual — Module 2: Application Planning (Sections 1 & 2, pages 2-1 to 2-8)

This lesson covers the complete Module 2 of the AVEVA Application Server course. We begin with the structured project workflow that guides every successful Galaxy deployment, then apply it to a real-world case study involving a mixer system in a fictitious manufacturing plant. Proper planning is the foundation upon which a robust, maintainable Galaxy is built.

Part A: Section 1 — Project Workflow

Reference: Module 2, Section 1 (pages 2-1 to 2-4)

Why Planning Matters

Before you create a single object in the Galaxy, you need a plan. A production Galaxy can contain hundreds or even thousands of objects — each with attributes, relationships, scripts, and deployments. If you start building without a clear roadmap, you will inevitably need to backtrack: renaming objects, restructuring templates, reassigning deployments, and correcting security. Every change costs time and risks introducing inconsistencies.

Key Concept: In an AVEVA Application Server Galaxy, the cost of making changes increases dramatically after the initial design phase. Investing time upfront in planning templates, naming conventions, and hierarchy structure saves enormous effort later. The derivation hierarchy — the chain from base templates to derived objects — is especially costly to rework once objects are in production.

The course teaches a structured 6-step workflow that addresses all major design decisions before implementation begins. This workflow ensures you think through every layer of the application systematically.

Figure 2-1: The 6-step Application Server project workflow — each step builds upon the previous one (Module 2, page 2-3)

The 6-Step Workflow

The flowchart illustrates a sequential planning process. Each step feeds the next, and together they cover every aspect of the Galaxy application design.

Step 1: Identify Field Devices and Functional Requirements

This is the starting point of any project. You must gather all the documentation that defines what the system needs to control and monitor:

  • P&IDs (Piping and Instrumentation Diagrams) — show the physical equipment, piping, valves, instruments, and their interconnections
  • Equipment lists — enumerate every field device (pumps, valves, transmitters, sensors, motors, etc.)
  • Functional requirements — describe what each device does, what signals it has, and how devices interact
  • I/O mapping documents — define which I/O channels connect to which field devices

The output of this step is a complete inventory of every object that must exist in the Galaxy, along with their attributes and behaviors. In the case study for this course, for example, the equipment list identifies 8 devices (3 valves, 2 pumps, 1 agitator, 2 transmitters) that together form a mixer system.

Step 2: Define Naming Conventions

Before creating any objects, establish a consistent naming standard that will be used across the entire Galaxy. This includes:

  • Tag naming format — the structure for object names (e.g., MixerX00.[Equipment].[Signal])
  • Area hierarchy naming — the convention for the $Area plant model structure
  • Template naming — how base and derived templates are labeled in the Model hierarchy
  • Attribute naming — any custom attribute or property names that follow a project standard
Pitfall: Naming conventions must be decided and documented BEFORE creating any objects. If you rename objects later, every reference — including scripts, graphic displays, historian tags, and alarm configurations — must be updated. In large projects, a single naming inconsistency can cascade into hours of rework.

Step 3: Plan Templates (Derivation Hierarchy)

This is the most critical planning step. Templates define the object types and their inheritance chain — the derivation hierarchy. You must decide:

  • Base templates — what fundamental object types exist (e.g., Valve, Pump, Transmitter)
  • Derived templates — how templates inherit and specialize (e.g., a 2-way valve derived from a base Valve, a control valve derived from that)
  • Attribute composition — which attributes each template level provides
  • Script attachment — where behavior logic lives in the hierarchy
Critical Warning: The derivation hierarchy is the hardest thing to change later. Once you derive objects from templates and those objects are in use, reorganizing the inheritance tree requires moving every affected object, re-pointing every derivation, and verifying that attributes and scripts still resolve correctly. Plan this carefully and completely before proceeding.

Step 4: Define the Plant Model ($Area Hierarchy)

Define the area structure that mirrors your physical plant layout. The $Area object serves as the root of a hierarchical tree that organizes objects by their physical location or functional area. For example:

  • $Area → Plant1 → Production → Mixer100
  • $Area → Plant1 → Packaging → Packer1

This hierarchy is used for object navigation, alarm grouping, security scoping, and display organization. It is flexible and can be updated as the project evolves.

Step 5: Define the Security Model

Configure the permission structure that controls who can view, modify, and operate objects. This includes:

  • User groups — roles such as Operator, Engineer, Administrator
  • Object-level permissions — read/write/operate access for specific objects or areas
  • Attribute-level security — which attributes can be modified by which roles

The security model can also be refined after initial deployment as operational needs become clearer.

Step 6: Define the Deployment Model ($WinPlatform Placement)

Finally, plan how objects are deployed across platforms. The $WinPlatform objects represent the runtime servers where Galaxy objects execute. You must decide:

  • Which objects run on which platform
  • How load is distributed across multiple platforms
  • Fault tolerance and failover requirements
  • Network topology between platforms and field devices

Like the plant model and security model, the deployment model is flexible and can be adjusted post-deployment.

Critical Asymmetry: Steps 1-3 vs. Steps 4-6

Key Takeaway: The 6 steps are NOT equally difficult to revise later:
  • Steps 1-3 (Identify Requirements, Define Naming Conventions, Plan Templates): Extremely difficult and costly to change after implementation. These are the hard decisions — get them right the first time.
  • Steps 4-6 (Define Plant Model, Define Security Model, Define Deployment Model): Much more flexible. These can be updated, restructured, and refined as the project progresses and requirements evolve.

Part B: Section 2 — Case Study

Reference: Module 2, Section 2 (pages 2-5 to 2-8)

To make the planning workflow concrete, the course introduces a case study based on a fictitious manufacturing plant. We will use this plant throughout the course to build, configure, and test our Galaxy application.

Factory Layout

The fictitious factory is organized into four main functional areas arranged in a linear process flow. The Production area contains two mixer lines (Line1 and Line2), and the Packaging area handles finished product packaging. These two areas are the focus of our case study.

Figure 2-2: Factory layout — Receiving → Production (Line1 + Line2) → Packaging → Shipping. The red dotted line highlights the Production and Packaging areas as the focus of our case study (Module 2, page 2-5)
  1. Receiving — Raw materials arrive and are staged for production
  2. Production — Materials are processed and mixed. Contains two production lines (Line1 and Line2), each with mixer systems
  3. Packaging — Finished products are packaged for shipment
  4. Shipping — Packaged products are loaded and dispatched

For our course, we focus on the Production and Packaging areas. Specifically, we model the mixer systems in Production that combine raw materials into finished products.

Tip: The dot-separated naming convention (Area.Equipment.Signal) mirrors the object hierarchy in the Galaxy. The dots represent parent-child relationships — Mixer100 is the parent object, Inlet1 is a child of Mixer100, and CMD is an attribute of Inlet1. This makes navigation intuitive and scripts readable.

Simulated Process

The mixer performs a continuous 4-step batch process that repeats indefinitely. This simulated process provides realistic behavior for testing our Galaxy application.

Figure 2-5: Simulated mixer process — the 4-step batch cycle that repeats continuously (Module 2, page 2-8)

Step 1: Add First Material

  • Inlet1 valve opens (CMD = ON, LSClosed = OFF, LSOpen = ON)
  • Pump1 starts (CMD = ON, LSClosed = OFF)
  • Level rises from 0 to 60% of capacity (approximately 150 Liters)
  • When level reaches 60%, Inlet1 closes and Pump1 stops (LSClosed = ON)

Step 2: Add Second Material

  • Inlet2 valve opens (CMD = ON, LSClosed = OFF, LSOpen = ON)
  • Pump2 starts (CMD = ON, LSClosed = OFF)
  • Level rises from 60% to 100% of capacity (250 Liters)
  • When level reaches 100%, Inlet2 closes and Pump2 stops (LSClosed = ON)

Step 3: Mix

  • Agitator starts (CMD = ON, PV = running)
  • Agitator speed set to the configured setpoint (Speed.SP)
  • Agitator runs for a configured number of seconds
  • When the mixing time elapses, agitator stops (CMD = OFF, PV = stopped)

Step 4: Drain

  • Outlet valve opens (CMD = ON, LSClosed = OFF, LSOpen = ON)
  • Level drops from 100% to 0 (empty)
  • When level reaches 0, Outlet valve closes (LSClosed = ON)
  • The cycle then restarts at Step 1
Lab Preview: In subsequent lessons, you will create this exact process using scripts attached to objects in the Galaxy. The step sequencing, level simulation, and device control logic will all be implemented as part of your Application Server application.

Analog Value Behavior

The simulated process includes realistic analog value behavior with built-in variability:

  • Level — proportional to temperature. As temperature rises, the effective level reading reflects thermal expansion. The level value is the primary indicator of process progress (0 → 60% → 100% → 0).
  • Temperature — varies based on a random coefficient, simulating real-world temperature fluctuations. This ensures each cycle produces slightly different values, making the simulated data realistic rather than perfectly repetitive.
  • Agitator Speed — binary behavior: either 0 (stopped) or the configured setpoint value (Speed.SP). There is no ramp-up or ramp-down — it is on or off.

The random coefficient used in the temperature simulation adds variability so that each cycle produces unique data points. This is important for testing graphics, trends, and alarms that need to respond to changing values.

Tip: The analog simulation approach — using a random coefficient to add variability — is a common technique in training environments. It allows you to see how graphics, trends, and alarms respond to changing values without needing actual field hardware.

Part C: Module 2 Summary + Quiz

Key Takeaways from Module 2

Summary:
  1. Planning is not optional. A Galaxy with hundreds or thousands of objects demands a structured approach. The 6-step workflow provides that structure.
  2. Steps 1-3 are high-stakes. Identifying requirements, defining naming conventions, and planning the template hierarchy are difficult to change after implementation. Invest heavily in getting these right.
  3. Steps 4-6 are flexible. Plant model, security, and deployment can be refined iteratively as the project evolves.
  4. Consistent naming saves time everywhere. The MixerX00.[Equipment].[Signal] format used in the case study is a model of clarity and consistency.
  5. The case study provides a complete, realistic test scenario. The 4-step mixer process exercises valves, pumps, transmitters, limit switches, scripts, and analog simulation — covering the full range of Application Server capabilities.
  6. Template design is the foundation of reuse. A well-designed derivation hierarchy lets you create new mixers (Mixer200, Mixer300) by simply deriving new instances rather than building from scratch.
  7. Level and Temperature are analog values with defined ranges. Level is 0–250 Liters; Temperature is 0–250 °C. Both are proportional and include a random coefficient for realistic variability.

Question 2: Hardest to Change

Which steps in the 6-step workflow are the most difficult and costly to change after implementation, and why?

Click to reveal answer

Steps 1-3 (Identify Requirements, Define Naming Conventions, Plan Templates) are the most difficult to change. The derivation hierarchy in Step 3 is especially hard to restructure because every derived object depends on its parent template. Changing the hierarchy requires moving, re-pointing, and re-verifying every affected object, including their attributes, scripts, and deployments.

Question 3: Mixer Components

How many devices make up the mixer system in the case study? List them.

Click to reveal answer

The mixer system has 8 devices: Inlet1 (valve), Inlet2 (valve), Outlet (valve), Pump1, Pump2, Agitator (motor), Level (transmitter), and Temperature (transmitter).

Question 4: Signal Differences

What signals does a valve have versus a pump? How does the Agitator differ from both?

Click to reveal answer

Valves (Inlet1, Inlet2, Outlet) have three signals: CMD (command), LSClosed (limit switch closed), and LSOpen (limit switch open). Pumps (Pump1, Pump2) have two signals: CMD and LSClosed — no LSOpen because pumps are binary devices (running or stopped). The Agitator is more complex: CMD (command), PV (process value / running status), Speed.PV (actual speed), and Speed.SP (speed setpoint).

Question 5: Analog Ranges

What are the ranges and units for the Level and Temperature transmitters in the mixer system?

Click to reveal answer

Level: 0–250 Liters. Temperature: 0–250 °C (Celsius). Both are analog signals with the Level value being proportional to Temperature (thermal expansion simulation).

Question 6: Batch Process

Describe the four steps of the mixer's simulated batch process and the level after each step.

Click to reveal answer

Step 1 — Add First Material: Inlet1 opens and Pump1 runs, level rises from 0 to 60% capacity (~150 Liters). Step 2 — Add Second Material: Inlet2 opens and Pump2 runs, level rises from 60% to 100% capacity (250 Liters). Step 3 — Mix: Agitator runs at setpoint speed for a configured duration, level stays at 100%. Step 4 — Drain: Outlet valve opens, level drops from 100% to 0 (empty), then the cycle restarts at Step 1.
