Lesson 9: Lab 4 — Using Object Viewer

Source: AVEVA Application Server 2023 Training Manual, Module 3, Lab 4 (pages 3-43 to 3-55)

After deploying your Galaxy and starting runtime, Object Viewer is your window into the live system — the tool that lets you browse every object, inspect every attribute, and even modify values on the fly.

9.1 Lab Introduction and Objectives

Reference: Module 3, Lab 4 (page 3-43)

In this lab you will use the Object Viewer — the runtime inspection and monitoring utility built into Application Server — to interact with a deployed Galaxy. Object Viewer connects directly to the Galaxy Server runtime and provides a live view of all deployed objects and their attributes.

Lab Objectives:
  1. Open Object Viewer from the IDE and connect it to the deployed Galaxy runtime
  2. Navigate the object tree to locate specific objects and inspect their attributes
  3. Use the Watch Window to monitor real-time attribute values across multiple objects
  4. Modify attribute values at runtime to test logic and override defaults
Key Concept: Object Viewer connects to the Runtime (the Galaxy Server), not the IDE. Changes made in Object Viewer affect live attribute values in the running system. This is different from the IDE, where you modify the configuration of objects (their definitions, templates, and static properties).
Module 3, Lab 4 introduction — objectives for opening Object Viewer, navigating the tree, using the Watch Window, and modifying values at runtime (page 3-43)

9.2 Opening Object Viewer

Reference: Module 3, Lab 4 (pages 3-44 to 3-46)

Object Viewer can be launched directly from the IDE's Deployment view. Before opening it, ensure that:

  1. Your Galaxy has been deployed to the target platform
  2. The Galaxy Server service is running on the target node
  3. You have appropriate security credentials to connect

Launching Object Viewer from the Deployment View

The most common way to open Object Viewer is from the IDE's Deployment view:

  1. In the IDE, switch to the Deployment view
  2. Locate a deployed object (such as a Platform or an AppEngine) in the tree
  3. Right-click on the object to open the context menu
  4. Select View in Object Viewer from the context menu
Right-clicking on a deployed object in the Deployment view — the context menu includes "View in Object Viewer" along with Deploy, Undeploy, Upload Changes, and Properties (page 3-45)
Tip: The context menu also offers Deploy, Undeploy, Upload Changes, and Properties — the same deployment management functions available from the Deployment Manager. "View in Object Viewer" launches Object Viewer pre-connected to the selected object's Galaxy.

Object Viewer can also be launched independently from the Windows Start Menu at AVEVA → Application Server → Object Viewer. When launched this way, you must manually select the Galaxy to connect to from the connection dialog.

Connecting to the Galaxy Runtime

When Object Viewer launches, it establishes a connection to the Galaxy Server runtime. If you launched it via "View in Object Viewer", the connection is made automatically to that Galaxy. If launched standalone, a Connect to Galaxy dialog appears:

  1. Select the deployed Galaxy from the list
  2. Enter your username and password if prompted
  3. Click Connect
Object Viewer connecting to the Galaxy runtime — establishing the session that provides live attribute data (page 3-46)
Key Concept: Object Viewer does not modify the Galaxy configuration — it connects to the running Galaxy Server to inspect and interact with live deployed objects. The Galaxy must be deployed and the Galaxy Server must be running for Object Viewer to function.

9.3 The Object Viewer Window

Reference: Module 3, Lab 4 (page 3-48)

Once connected, the Object Viewer window is divided into two main areas:

Area Purpose
Tree Pane (left) Displays the Galaxy hierarchy — platforms, deployed objects, and system folders organized in a tree structure
Details Pane (right) Shows the attributes, values, timestamps, quality, and type information for the currently selected object
The Object Viewer main window — the left pane shows the TrainingGalaxy hierarchy with GRPlatform[S00ENG] and ProdPlatform[S00PROD] platforms; AppEngine1 is selected and its attributes (PlatformInfo, PageFaultsAvg, NetNMXHeartbeatPeriod, etc.) are displayed in the right pane with Value, Timestamp, Quality (C0:Good), Security, and Type columns (page 3-48)

Understanding the Galaxy Hierarchy

In the tree pane, the Galaxy is organized under a top-level node (e.g., TrainingGalaxy). Below it are the Platform objects representing the deployment nodes:

Node Description
GRPlatform[S00ENG] A platform deployed to the S00ENG node (e.g., the development or engineering station)
ProdPlatform[S00PROD] A platform deployed to the S00PROD node (e.g., the production station)
AppEngine1 An Application Engine object deployed under a platform — contains the application logic and attributes

Clicking on a platform node in the tree reveals the objects deployed under it. Clicking on a specific object (like AppEngine1) displays its attribute list in the right-hand details pane.

Reading the Attribute Columns

When an object is selected, the details pane shows a table of attributes with the following columns:

Column What It Shows
Attribute Name The identifier of the attribute (e.g., PlatformInfo, PageFaultsAvg, NetNMXHeartbeatPeriod)
Value The current real-time value of the attribute
Timestamp When the value was last updated
Quality A quality code indicating data reliability — C0:Good means the value is valid and trustworthy
Security The security access level required to modify this attribute
Type The data type of the attribute (e.g., String, Integer, Boolean)
Key Concept: The Quality code is critical for diagnostics. C0:Good means the attribute value is valid. Other codes indicate problems: for example, a quality code starting with B or C with a non-zero value may indicate a communication failure, out-of-range value, or sensor fault. Always check quality when troubleshooting unexpected values.

9.4 Navigating the Object Tree

Reference: Module 3, Lab 4 (page 3-49)

The left pane of Object Viewer presents the Galaxy in a hierarchical tree structure. Understanding this tree is essential for finding and inspecting any object in your deployment.

Expanding the Tree

  1. Expand nodes by clicking the + icon (or triangle) next to each item in the tree
  2. Navigate from the top-level Galaxy node → Platform → deployed objects
  3. Click on any object to see its attributes in the right-hand details pane
Navigating the object tree — expand nodes to drill down through platforms and their deployed objects to find specific items (page 3-49)

Finding Objects Quickly

For large Galaxies with many deployed objects, manually expanding the tree can be time-consuming. Object Viewer provides search capabilities:

  • Use Edit → Find (or press Ctrl+F) to search for an object by name
  • Right-click on any node to access context-menu options including Find, Properties, and Copy Path
Tip: The object path in Object Viewer follows the same format used in scripts and references — for example, TrainingGalaxy.GRPlatform.AppEngine1.PV. You can copy an object's path from the context menu for use in scripts or other tools.

9.5 Using OCMC Platform Manager

Reference: Module 3, Lab 4 (page 3-51)

While Object Viewer provides detailed attribute-level inspection, the Operations Control Management Console (OCMC) — specifically the Platform Manager — provides a higher-level view of platform status and management.

The OCMC Platform Manager — shows TrainingGalaxy[DefaultUser] with platform deployment status and management options (page 3-51)

The Platform Manager in OCMC allows you to:

  • View the deployment status of all platforms in the Galaxy
  • See which platform is currently connected (e.g., TrainingGalaxy[DefaultUser])
  • Monitor platform health and connectivity
  • Perform platform-level management tasks
Key Concept: Object Viewer and OCMC serve complementary roles. Object Viewer is for inspecting individual object attributes and values at a detailed level. OCMC Platform Manager is for monitoring and managing the overall platform infrastructure — which nodes are running, their status, and deployment state.

9.6 The Watch Window

Reference: Module 3, Lab 4 (pages 3-53 to 3-54)

The Watch Window is one of Object Viewer's most powerful features. It allows you to collect attributes from different objects across the Galaxy and display them together in a single, real-time updating view — like a custom monitoring dashboard.

Setting Up the Watch Window

To add attributes to the Watch Window:

  1. In the Object Viewer tree, navigate to an object whose attribute you want to monitor
  2. In the details pane, right-click on the attribute name
  3. Select Add Watch Window from the context menu
  4. The attribute appears in the Watch Window, showing its full path and current value
  5. Repeat for additional attributes from the same or different objects
Setting up the Watch Window — selecting attributes to monitor (page 3-53)

The Watch Window in Action

Once populated, the Watch Window displays an AttributeReference list — each entry shows the full attribute path and its live value. The entries update automatically as the runtime values change.

The Watch Window with AttributeReference entries — showing ProdPlatform.PlatformInfo, GRPlatform.PlatformInfo, and GRPlatform.GR.TimeOfLastDeploy with live values. The right-click menu provides options including "Add Watch Window" for adding more attributes (page 3-54)

In this example, the Watch Window monitors attributes from multiple platforms simultaneously:

Attribute Path What It Shows
ProdPlatform.PlatformInfo Platform information string for the production platform
GRPlatform.PlatformInfo Platform information string for the engineering platform
GRPlatform.GR.TimeOfLastDeploy Timestamp of the last deployment to the GR platform — useful for verifying deployment recency

Managing the Watch Window

Action How
Add an attribute Right-click attribute in details pane → Add Watch Window
Remove an attribute Right-click attribute in Watch Window → Remove
Clear all entries Right-click in Watch Window → Clear All
Copy attribute path Right-click → Copy — useful for pasting into scripts or other tools
Key Concept: The Watch Window is read-only by default — it monitors values but does not change them. It is ideal for watching how multiple attributes across different objects change over time, diagnosing communication issues, and verifying that script logic produces expected results.

9.7 Modifying Attribute Values at Runtime

Reference: Module 3, Lab 4 (page 3-55)

Object Viewer does not just let you observe — it also lets you write new values to certain attributes at runtime. This is extremely useful for testing logic, overriding defaults, and simulating field conditions without changing the actual hardware.

Writing a New Value

  1. In the Object Viewer tree, navigate to the object whose attribute you want to modify
  2. In the details pane, double-click on the attribute's Value cell (or right-click → Modify Value)
  3. A dialog appears prompting you to enter the new value
  4. Type the desired value and click OK
  5. The new value is immediately written to the runtime, and you will see the attribute update in the details pane
Modifying attribute values at runtime — writing a new value directly to an output attribute in the running Galaxy (page 3-55)

Important Restrictions

Pitfall — Not All Attributes Are Writable: You can only modify attributes that have a writable access level. Attributes marked as read-only (such as computed values, alarm states, or system-generated timestamps) will not accept manual writes. Attempting to write to a read-only attribute will either be silently ignored or produce an error message.

Typically writable attributes include:

  • Output (OP) attributes — the values the object writes to field devices
  • Setpoint (SP) attributes — target values for control loops
  • Mode attributes — operating mode selections (Auto, Manual, etc.)
  • Custom attributes defined with Read/Write access in the template

Typically read-only attributes include:

  • PV (Process Value) — in most cases, PV comes from the field or simulation and cannot be written directly
  • Alarm status attributes — generated by the alarm engine
  • System timestamps and counters
Tip: Writing a value to an output attribute in Object Viewer is a temporary override. If the object's script or logic recalculates that attribute on the next cycle, your manually written value may be overwritten. This is by design — runtime logic takes precedence over manual overrides unless the object is placed in Manual mode.

9.8 Summary

In this lab you learned the essential skills for using Object Viewer:

Skill What You Did
Open and connect Launched Object Viewer from the Deployment view via right-click → "View in Object Viewer", connecting to the Galaxy runtime
Browse the hierarchy Navigated the Galaxy tree to find platforms (GRPlatform, ProdPlatform) and their deployed objects (AppEngine1)
Inspect attributes Read attribute values, quality codes, timestamps, security levels, and data types in the details pane
Watch Window Added attributes from multiple platforms to a single monitoring view (AttributeReference list)
Platform Manager Used OCMC Platform Manager to view platform deployment status and connectivity
Modify values Wrote new values to writable attributes at runtime to test and override behavior
Key Concept: Object Viewer is a runtime tool, not an IDE tool. It connects to the Galaxy Server's live data and lets you interact with the running system. For configuration changes (adding attributes, changing templates, modifying scripts), you must use the IDE — Object Viewer is for inspection, monitoring, and value overrides only.

Quiz

Question 1: Opening Object Viewer

What is the most common way to launch Object Viewer from the IDE, and what does the right-click context menu on a deployed object include?

Click to reveal answer

The most common way is to right-click on a deployed object in the IDE's Deployment view and select "View in Object Viewer" from the context menu. The context menu also includes options for Deploy, Undeploy, Upload Changes, and Properties.

Question 2: Attribute Columns

When you select an object in the Object Viewer tree, what columns are displayed in the details pane, and what does a Quality code of C0:Good mean?

Click to reveal answer

The details pane displays columns for Attribute Name, Value, Timestamp, Quality, Security, and Type. A Quality code of C0:Good means the attribute value is valid, reliable, and trustworthy — there are no communication failures or data quality issues.

Question 3: Galaxy Hierarchy

In the TrainingGalaxy shown in the lab, what are the two platforms, and what type of object is AppEngine1?

Click to reveal answer

The two platforms are GRPlatform[S00ENG] (deployed to the engineering station) and ProdPlatform[S00PROD] (deployed to the production station). AppEngine1 is an Application Engine object deployed under a platform — it contains application logic and exposes attributes like PlatformInfo, PageFaultsAvg, and NetNMXHeartbeatPeriod.

Question 4: Watch Window

How do you add an attribute to the Watch Window, and what is the advantage of the Watch Window compared to the details pane?

Click to reveal answer

To add an attribute to the Watch Window, right-click on the attribute in the details pane and select "Add Watch Window". The advantage is that the Watch Window lets you monitor attributes from multiple different objects and platforms simultaneously in a single view (the AttributeReference list), whereas the details pane only shows attributes for one selected object at a time.

Question 5: Object Viewer vs. IDE

Is Object Viewer used for modifying Galaxy configuration (templates, scripts, object definitions)? Explain the distinction between Object Viewer and the IDE.

Click to reveal answer

No — Object Viewer does not modify Galaxy configuration. It is a runtime tool that connects to the Galaxy Server to inspect and interact with live deployed objects. For configuration changes (adding attributes, changing templates, modifying scripts), you must use the IDE. Object Viewer is for inspection, monitoring, and temporary value overrides only.

Question 6: Temporary Overrides

If you manually write a new value to an output attribute using Object Viewer, will that value persist indefinitely? What might overwrite it?

Click to reveal answer

No, the value will not necessarily persist. Writing a value via Object Viewer is a temporary override. If the object's script or control logic recalculates that output attribute on the next execution cycle, the manually written value will be overwritten. To maintain a manual override, the object should typically be placed in Manual mode, which prevents automatic recalculation of the output.
