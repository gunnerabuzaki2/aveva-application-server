Lesson 13: Change Control and Propagation + Lab 7

Source: AVEVA Application Server 2023 Training Manual, Module 4, Section 3 + Lab 7

How template changes propagate through the object hierarchy — the locking mechanism that protects derived objects and the check-in workflow that ensures changes are safely committed to the Galaxy.

13.1 Change Control Overview

In a multi-developer environment, many engineers may work on the same Galaxy simultaneously. Change control is the system that manages who can modify what, and ensures that template changes propagate correctly through the object hierarchy without causing unintended side effects.

Change Control in Application Server governs how template changes flow down to derived objects. It uses a locking mechanism to protect objects from unwanted changes, and a check-out/check-in workflow to manage multi-user access. When you modify a template, you control exactly which attributes propagate and which remain locked.

The change control system addresses a fundamental challenge: templates are shared. When you change a template, every object derived from it could be affected. Without change control, a single template edit could break dozens or hundreds of deployed objects.

13.2 The Object Hierarchy and Locking

Reference: Module 4, Section 3 (pages 4-31 to 4-33)

Application Server uses a hierarchical template structure where derived objects inherit attributes from their parent templates. The lock state of each attribute determines whether changes from the parent template will propagate down.

The Change Control hierarchy — $DiscreteDevice → $Valve (Unlocked) → $Inlet/$Outlet (Locked in Parent) → CV101–CV104 (Locked in Parent). The legend at the bottom shows all three lock states: Unlocked, Locked, and Locked in Parent (Module 4, page 4-31)

The hierarchy shown in the diagram illustrates a typical equipment model:

LevelObjectLock StateMeaning
Base Template $DiscreteDevice Root The base — changes here can propagate to all descendants
Area Template $Valve Unlocked (green) Receives changes from parent; can also propagate changes downward
Equipment Group $Inlet, $Outlet Locked in Parent (grey with person icon) Protected from parent changes — attributes locked so upstream edits don't overwrite local customizations
Object Instances CV101, CV102, CV103, CV104 Locked in Parent (grey with person icon) Each instance is locked — protecting their specific I/O and configuration from template-level changes
Locking prevents propagation. When an attribute is locked, changes made to that attribute in a parent template will NOT flow down to the locked object. This is essential for protecting local customizations — such as unique I/O references, scaling values, or alarm setpoints that differ from the template defaults.

13.3 Lock States — Unlocked vs. Locked vs. Locked in Parent

Reference: Module 4, Section 3 (pages 4-32 to 4-33)

Every attribute on every object exists in one of three lock states. Understanding these states is critical for managing change propagation. The following diagrams show how the same hierarchy looks with different lock configurations, illustrating how lock states change when a parent template is modified:

The same hierarchy with updated lock states — showing how locks change when the parent template ($Valve) is modified. Some attributes that were previously unlocked now show modified states (Module 4, page 4-32)
Propagation of lock changes — after check-in, the template modifications flow down to unlocked descendants while locked objects retain their local values. The visual indicators confirm which objects received updates (Module 4, page 4-33)
Lock StateIconBehavior
Unlocked Green open lock The attribute is synchronized with the parent template. Any change to the parent's value will automatically propagate to this object. The object's value matches the template value.
Locked Grey/person icon The attribute is protected from parent changes. The attribute has been locally modified (or explicitly locked) so that parent template updates will NOT overwrite it. The value is "frozen" at the object level.
Locked in Parent Grey/person icon The attribute was locked by the parent template, so this object inherits the locked state. The attribute cannot be changed from this level — it can only be modified by unlocking at the template level where the lock was applied.
Common mistake: Locking an attribute is not the same as locking the entire object. You can lock individual attributes while leaving others unlocked. This gives you fine-grained control — for example, you might lock the I/O reference (which is unique to each device) but leave the alarm limits unlocked (so they propagate from the template).

How to Lock and Unlock Attributes

  • To lock an attribute: Right-click the attribute in the Object Editor and select Lock. The attribute's lock icon changes to the locked state, and it will no longer receive updates from the parent template.
  • To unlock an attribute: Right-click the locked attribute and select Unlock. The attribute immediately reverts to the parent template's value and becomes synchronized again.
  • Bulk locking: You can lock or unlock multiple attributes at once by selecting them and applying the lock/unlock action to the selection.
Best practice: When creating derived objects from templates, lock attributes that will be unique to each instance (I/O references, tag names, scaling) and leave shared attributes (alarm limits, engineering units, descriptions) unlocked so they propagate template changes automatically.

13.4 Attribute Propagation Rules

Propagation follows predictable rules that determine whether a template change reaches its derived objects:

Template Attribute StateDerived Attribute StateWhat Happens
Modified in template Unlocked in derived ✅ Change propagates — derived attribute updates to match template
Modified in template Locked in derived ❌ Change blocked — derived attribute retains its local value
Modified in template Locked in Parent (derived) ❌ Change blocked — lock was inherited from a higher-level template
Unchanged in template Any state in derived No change — nothing to propagate
Propagation is one-way: Changes flow from parent template → derived object, never the reverse. If you modify an attribute on a derived object, it does NOT push the change up to the parent template. This ensures template changes are always intentional and controlled.

Visual Feedback in the IDE

The IDE provides clear visual cues about lock states:

  • Green open lock icon: Unlocked — synchronized with parent, changes will propagate
  • Grey lock/person icon: Locked — protected from parent changes
  • Bold attribute name: The attribute has been locally modified (different from parent value)
  • Normal attribute name: The attribute matches the parent template value

13.5 Check-Out / Check-In Workflow

Reference: Module 4, Section 3 (pages 4-34 to 4-36)

In a multi-developer environment, the Galaxy Repository manages concurrent access through a check-out/check-in system. This prevents two developers from editing the same object simultaneously.

Check-out/Check-in is Application Server's version control mechanism. When you check out an object, you gain exclusive editing rights. When you check it in, your changes are saved to the Galaxy Repository and others can check it out again.

The Workflow

StepActionResult
1 Check Out an object You gain exclusive editing rights. The object is locked for other developers. You can modify attributes, scripts, and configuration.
2 Make changes in the Object Editor Your modifications are stored locally in the IDE. They are not yet visible to other developers or to runtime.
3 Check In the object Changes are saved to the Galaxy Repository. The object is unlocked for other developers. The IDE shows the updated object.
4 Deploy (if needed) Checked-in changes are pushed to runtime. Check-in and deployment are separate steps — you can check in without deploying.
Don't forget to check in! If you check out an object and walk away, no one else can edit it. The Galaxy Repository tracks checked-out objects. If a developer leaves without checking in, an administrator may need to force-check the object back in.

The first step is to lock specific attributes in the $Valve template so that downstream objects are protected from changes to those attributes. This simulates a real-world scenario where certain valve properties are customized per-instance and should not be overwritten by template updates.

Step 1a: Open the $Valve template in the Object Editor. Navigate to the attributes you want to protect.

Step 1b: Select the attributes that should be unique to each valve instance — typically the I/O reference, tag name, and device-specific scaling values.

Step 1c: Right-click the selected attributes and choose Lock. The lock icons change to indicate the locked state.

Step 1d: Verify the lock states. Locked attributes should display the grey lock/person icon. Unlocked attributes retain the green open lock icon.

Initial lock states in the hierarchy — $Valve is unlocked and receives changes from $DiscreteDevice, while $Inlet, $Outlet, and CV101–CV104 are all locked in parent, protecting their local customizations (Module 4, page 4-31)
Think about what you're locking: In this lab, you're locking attributes like the I/O reference and tag path — these are unique to each physical valve. If you left them unlocked, every template change would overwrite the individual valve's connection to its PLC tag, breaking runtime communication.

13.8 Lab 7 Step 2 — Modify the Template and Observe Propagation

Now the critical test: modify an attribute in the $Valve template and observe which derived objects receive the change and which are protected by the locks.

Step 2a: Open the $Valve template and modify an unlocked attribute — for example, change an alarm limit or description text.

Step 2b: Observe how the hierarchy diagram changes after modification. The lock states update to reflect which attributes were affected:

After modifying the $Valve template — the hierarchy shows updated lock states. Unlocked attributes in $Valve have changed, and the propagation begins flowing down to unlocked descendants (Module 4, page 4-32)

Step 2c: Open one of the derived valve objects (e.g., CV101). Verify that:

  • ✅ The unlocked attributes now show the updated value from the template
  • ❌ The locked attributes still show their original local values — the template change was blocked
Propagation results — after check-in, template modifications flow down to unlocked descendants while locked objects retain their local values. The visual indicators confirm which objects received updates and which remained protected (Module 4, page 4-33)
This is the core lesson: The locking mechanism gives you precise control over which template changes reach your deployed objects. Without locking, every template edit would cascade to every derived object — potentially breaking unique configurations. With locking, you can confidently update shared settings (alarm limits, descriptions) while protecting instance-specific data (I/O references, tag paths, unique setpoints).

13.9 Lab 7 Step 3 — Check-In Workflow with Propagation

When you check in a modified template, the system propagates changes to all derived objects. The check-in dialog provides important information about what will happen.

Step 3a: After making your template modifications, select Check In. The dialog warns you about propagation:

The Check-In dialog warns that changes will propagate — "This check-in operation will propagate changes to 5 derived objects. This may temporarily block other users..." Click Check-in to proceed, or Cancel to abort (Module 4, page 4-36)
Read the propagation warning carefully! The dialog tells you exactly how many derived objects will be affected. If the number seems wrong, you may have locked or unlocked the wrong attributes. You can click Cancel to go back and review your lock states before committing.

Step 3b: Click Check-in to proceed. The system processes the changes and shows a completion dialog:

Check-in completion — "Checking in objects — Check In complete. Propagating template changes to 1 descendent objects. Optimizing the galaxy database..." Click Close to finish (Module 4, page 4-37)

The completion dialog confirms:

  • ✅ Check In complete — your changes are saved to the Galaxy Repository
  • ✅ Propagation count — shows how many descendent objects received the template changes (only unlocked ones)
  • ✅ Database optimization — the Galaxy database is being optimized after the changes
Watch the propagation count: If the completion dialog shows more objects propagating than expected, you may have left some attributes unlocked that should have been locked. This is your last chance to review — if needed, you can modify lock states and check in again.

13.10 Lab 7 Step 4 — Multi-Object Check-In

In a complex Galaxy, you may need to check in multiple related objects at once. The system handles this gracefully, propagating changes through the entire hierarchy in the correct order.

Step 4a: When checking in multiple objects, the system shows the propagation steps for each affected object:

Checking in multiple objects — the dialog shows propagation steps as the system processes changes through the hierarchy. Each step updates the affected derived objects in order from parent to child (Module 4, page 4-38)

Step 4b: After check-in completes, open one of the final derived objects (e.g., CV101) and verify the attribute states:

  • ✅ Attributes that were unlocked at all levels received the template updates
  • ❌ Attributes that were locked at any level in the hierarchy retained their local values
  • Attributes marked as Locked in Parent could not be changed from the derived object level
Propagation order matters: Changes always flow from the top of the hierarchy downward. If $DiscreteDevice is modified, the change reaches $Valve first, then $Inlet/$Outlet, then CV101–CV104. At each level, the lock state of each attribute determines whether the change passes through or is blocked.

Quiz — Test Your Understanding

Q1: What does it mean when an attribute is "Locked" in Application Server?

Click to reveal answer

A locked attribute is protected from parent template changes. Modifications made to the attribute in the parent template will NOT propagate to the derived object — the attribute retains its local value.

Q2: What is the difference between "Locked" and "Locked in Parent"?

Click to reveal answer

"Locked" means the attribute was explicitly locked on this object. "Locked in Parent" means the lock was inherited from a higher-level template — the attribute was locked at a parent template level, and this object inherits that locked state.

Q3: You modify an alarm limit on a base template. A derived object has that alarm limit attribute locked. What happens to the derived object's alarm limit?

Click to reveal answer

Nothing — the derived object's alarm limit retains its current value. The lock prevents the template change from propagating. The derived object keeps its locally-set alarm limit.

Q4: In the check-out/check-in workflow, what happens if you check out an object but never check it back in?

Click to reveal answer

The object remains locked and no other developer can check it out or edit it. An administrator may need to force-check the object back in to release the lock.

Q5: Why would you lock the I/O reference attribute on a derived valve object but leave the alarm limit attribute unlocked?

Click to reveal answer

The I/O reference is unique to each physical valve — it connects the object to a specific PLC tag. If unlocked, a template change would overwrite it, breaking communication. The alarm limit is a shared setting that should propagate from the template so all valves use the same alarm thresholds.

Q6: What information does the check-in dialog provide before you commit changes?

Click to reveal answer

The check-in dialog warns you how many derived objects will be affected by the propagation. It says something like "This check-in operation will propagate changes to 5 derived objects. This may temporarily block other users..." This lets you review the scope of changes before committing — you can click Cancel to abort if the number seems wrong.

Key Takeaways

  1. Change control manages how template modifications propagate through the object hierarchy — protecting deployed objects from unintended changes.
  2. Three lock states exist: Unlocked (synchronized with parent), Locked (protected from parent changes), and Locked in Parent (lock inherited from a higher-level template).
  3. Lock individual attributes for fine-grained control — lock instance-specific data (I/O references, tag paths) and leave shared settings (alarm limits, descriptions) unlocked.
  4. Propagation is one-way: parent template → derived object, never the reverse.
  5. The check-out/check-in workflow prevents concurrent editing conflicts in multi-developer environments.
  6. The check-in dialog shows how many derived objects will be affected — always review the propagation count before committing.
  7. Always check in objects after editing — unchecked-out objects block other developers.
