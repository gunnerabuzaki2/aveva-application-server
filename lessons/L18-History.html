Lesson 18: History + Lab 12 — Module 5-6 Summary

Source: AVEVA Application Server 2023 Training Manual, Module 6, Section 1 + Lab 12: Historizing Data (pages 6-3 to 6-22)

Module 6 covers how data flows from Application Server into the Historian, how to enable history on attributes, and how to view historical trends.

Module 6 — Historizing Data

18.1 History Architecture — How Data Flows to the Historian

Reference: Module 6, Section 1 (pages 6-3 to 6-6)

Historizing data is one of the most important functions in any SCADA/MES system. Application Server provides the bridge between live process data and a long-term Historian. Understanding the architecture is the first step.

High-level history architecture: Area contains the AppEngine (CLIENT) which communicates with the Historian server. The WinPlatform sits at the bottom providing the platform-level services.
Key Concept — Store and Forward: Application Server does not write directly to the Historian. Instead, it writes data to a Store and Forward (S&F) cache on the WinPlatform. If the Historian is temporarily unreachable, data is buffered locally and forwarded when connectivity is restored. This guarantees no data loss.
Complete data flow: PLC → DAServer/Proxy/DBObject → AppEngine (Application Object in Area) → WinPlatform (Store and Forward cache) → Historian (Storage Engine → Run-time Database)

Let's trace the full path:

StepComponentRole
1PLC / DeviceSource of raw process data
2DAServer / Proxy / DBObjectData access — pulls values from the PLC and exposes them as attributes
3AppEngine (Application Object)Runs inside an Area; receives attribute updates and applies logic
4WinPlatform (Store and Forward)Buffers historical data on disk; forwards to Historian when available
5Historian ServerStorage Engine writes to the run-time database (permanent archival)

18.2 Configuring History on an Attribute

Reference: Module 6, Section 1 (pages 6-7 to 6-11)

To historize a tag, you simply enable the History feature on its attribute. The training uses a mixer malfunction alarm as an example.

Tag configuration: Name = Alarm.Condition, Description = "Mixer Malfunction", Data type = Integer, User writeable. The History feature is enabled with Value deadband = 0.0, Trend High = 10.0, Trend Low = 0.0, and Force storage period = 0 ms (meaning data is stored on-change only).
On-Change vs. Periodic Storage:
• Force storage period = 0 ms means the Historian stores a new value only when the attribute changes (on-change).
• If you set a non-zero period (e.g., 5000 ms), the Historian stores a value every N milliseconds regardless of change.
• Value deadband (0.0 = none) controls the minimum change required to trigger storage. A deadband of 0.0 means every single change is recorded.

The same pattern applies to any analog attribute. Here is an example of configuring history on a PV attribute:

SMeter PV attribute: Float data type, Object writeable, I/O enabled, History checked. This enables historian collection for this attribute.
History configuration for the PV: Trend High = 10.0, Trend Low = 0.0, Rollover = 0.0, Interpolation = SystemDefault, Swinging Door compression unchecked.
Tip — Trend High/Low: These values define the Y-axis range for Historian Client trends. Set Trend High to slightly above your maximum expected value and Trend Low to slightly below the minimum. The system uses these to scale the display — they do not filter or truncate data.
Common Pitfall: Forgetting to check the History box on an attribute. If the History checkbox is not enabled, the Historian will not collect data for that attribute no matter what other settings you configure. Always verify History is checked before deploying.

18.3 Platform History and Store-and-Forward Settings

Reference: Module 6, Section 1 (pages 6-8 to 6-10)

The WinPlatform itself has history-related settings that control the Store and Forward behavior.

ProdPlatform General tab: Network address = S00PROD, History store and forward path = C:\S&F, Min RAM = 1024 MB, Stats average period = 10000 ms. Additional tabs: Engine, Alarms, Platform History, Scheduler History.
Store and Forward Path: The path C:\S&F is where the platform caches historical data on local disk. Make sure this drive has sufficient space — if the S&F cache fills up, data can be lost. In production, use a dedicated, high-capacity drive.

18.4 Lab 12 — Historizing Data (Hands-On)

Reference: Lab 12 (pages 6-12 to 6-22)

Lab 12 Objective: Enable history on attributes, deploy to production, and verify historical trends using Historian Client Web.

Step 1: Review Template Inheritance

Before enabling history on the production instance, verify how attributes are inherited from templates.

$Mixer_Agitator Attributes: CMD and PV come from the $Motor template. Speed.PV and Speed.SP are local attributes. The [$Motor] notation shows template origin.
SValve Attributes tab: CMD (Valve Command) and LSOpen (Limit Switch Open) — both inherited from the template.

Step 2: Check In Template Changes

After modifying template attributes (enabling history), check in the changes to propagate them to all descendant objects.

Check-in result: "Propagating template changes to 2 descendant objects. SMixer.Level: Successful. Optimizing galaxy database..." — changes have been applied to all instances.

Step 3: Deploy to Production

Deploy the galaxy to the production platform and verify the objects are running.

Deployment tree: TrainingGalaxy → ProdPlatform → AppEngine1 → ControlSystem → Line1/Mixer100 with objects: Agitator_001, Inlet1_001, Level_001, Outlet_001, Pump1_001, Temperature_001.

Step 4: View Historical Trends in Historian Client Web

Open Historian Client Web (typically at http://localhost:32573) and create a trend to verify data is being collected.

Historian Client Web — the main dashboard interface. This is your tool for browsing, trending, and analyzing historical data.
Trend result: Level_001.PV = 64.47, showing an upward trend over a 24-hour period (May 12–13, 2020). Data is being collected successfully.
Level_001.PV(I) = 63.09 with an upward trend, time range Jan 18, 2023. The (I) indicates interpolated values.
Long-term trend: Level_001.PV stays flat near 0, then spikes to approximately 100 at 3 PM. Time selector ranges from 30 days to 1 hour — use this to zoom in/out on historical data.
Historian Client Web — second view of the dashboard confirming successful data collection.
Tip — Reading Trends: A flat line at 0 followed by a sharp spike usually means the Historian started collecting data at a specific point in time (when history was enabled). The spike represents the first real values arriving. Over time, the trend will show a continuous line.

Quiz

Question 1

What does setting the "Force storage period" to 0 ms mean?

Click to reveal answer

Data is stored on-change only — a new value is written to the Historian only when the attribute's value actually changes. No periodic storage occurs.

Question 2

What is the purpose of the Store and Forward (S&F) cache on the WinPlatform?

Click to reveal answer

The S&F cache buffers historical data on local disk when the Historian is unreachable. When connectivity is restored, the buffered data is forwarded to the Historian, ensuring no data loss during temporary outages.

Question 3

If you enable history on a template attribute and check in the changes, what happens to the descendant objects?

Click to reveal answer

The changes propagate automatically to all descendant objects. For example, "Propagating template changes to 2 descendant objects. SMixer.Level: Successful" — all instances inherit the history configuration from the template.

Question 4

What is the significance of Trend High and Trend Low in the history configuration?

Click to reveal answer

Trend High and Trend Low define the Y-axis display range for Historian Client trends. They do not filter data — they only control how the data is visually scaled. Set them to bracket your expected min/max values.

Question 5

What does the notation [$Motor] next to an attribute indicate?

Click to reveal answer

It indicates that the attribute is inherited from the $Motor template. The bracketed template name shows the origin of inherited attributes in the object's attribute list.
